S13

部署 · 生产架构

单机双角色：nginx 兼任静态分发与 API 网关，systemd 保障进程级可用性。

平台以单台华为云 VPS 承载全部线上服务，架构核心在于 nginx 的双重职责——对前端构建产物直接分发，对 /api 请求反向代理至后端；后端进程由 systemd 托管，实现开机自启与故障自愈的进程级可用性。发布流程收敛为单一脚本，并以健康探活作为上线闸门，达成可重复、低风险的部署。

华为云 VPS · huawei2winbeau.top 主 + feiyue.selab.top 镜像 systemd · feiyue-backend :8001nginx 静态服 + 反向代理
平台部署在一台华为云 VPS 上，nginx 同时承担两个职责：把前端 dist 作静态资源直接分发，同时把 /api 反向代理到后端；后端是 FastAPI，用 uvicorn 跑在 8001 端口，由 systemd 托管，开机自启、崩溃自愈。上线只需一条 deploy.sh：拉取主分支、同步依赖、迁移数据库、重启服务、健康探活、再构建前端。winbeau.top 为主域名，feiyue.selab.top 为镜像。
S14

部署 · CI 与双库维护

双库分工 + 提交前静态检查，维持多人协作下的代码有序性。

面向多人协作，代码库通过双库分工与提交前静态检查维持有序：规范库承载文档、规范与源码评审，部署源承载生产实际运行代码，二者职责隔离；每次提交先经本地检查门禁校验提交规范，通过后方进入发布流水线，从而保证提交历史的一致性与可追溯性。

我们采用双库策略：团队规范库存放文档与规范，部署源承载生产代码，职责相互隔离。所有提交遵循 conventional commits，由 Husky 配合 lint-staged 在提交前自动检查、拦截不合规改动，保证历史干净可追溯。对外的「发展历程」时间轴由 GitHub Pages 自动部署。前端以 pnpm + Vite 打包，后端以 uv 锁定依赖版本，确保换机后的一致性。
S15

运维 · 数据持久化与恢复

异地留存与命名空间隔离，实现单命令的整站数据恢复。

数据持久性是运维的首要约束。系统将数据库、用户上传附件与密钥统一同步至私有 HuggingFace 数据集实现异地留存，并按命名空间隔离「实时状态」与「参考数据」，避免相互干扰；面向换机与故障场景，单条命令即可完成整站数据的恢复复现，显著降低单点丢失风险。

make sync-push # 定时(cron)将 DB+附件+密钥推送至私有数据集
make data-pull # 换机 / 灾备：一键拉回整站数据并复现
make schools-pull / conf-pull # 拉取高校导师 / CCF 会议参考库
数据不能丢是运维的底线。我把数据库、上传附件和密钥统一同步到一个私有的 HuggingFace 数据集异地留存，并按命名空间分区：state 存放实时状态，schools 与 conf 存放高校导师库和会议库。日常备份由 make sync-push 定时 cron 完成；一旦换机或灾备，一条 make data-pull 即可拉回整站数据复现。内容真源为 62 篇 seed，加上站内用户直接发布的笔记。

安全与性能上设置了四道防线。鉴权采用 HS256 的 JWT，有效期 7 天，口令经 bcrypt 哈希。文件安全上，上传阶段拦截 svg、html 等可被当作脚本执行的扩展名，静态服务再统一注入 nosniff 头，双重抑制存储型 XSS，并配以 50MB 上限与 magic-byte 类型核验。性能上首屏 JS 压缩后约 144KB，全路由懒加载，AI 欢迎语缓存 3 小时以节省外部调用。管理后台支持登录审计、数据热更新与会议爬取手动触发。
