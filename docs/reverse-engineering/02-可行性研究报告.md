## 1 引言

### 1.1 编制目的

本报告旨在对已上线运行的 xju-feiyue 平台在**技术、经济、操作、社会/法律**四个维度进行系统性可行性论证。与常规立项前可行性研究不同，本项目属于逆向工程场景：系统已实际运行约 26 天（2026-05-09 至规模统计口径日 2026-06-04）、承载 108 名注册用户与 85 篇笔记，本报告的价值在于——

1. 为技术选型的合理性提供事后论证，识别哪些决策经受住了真实负载与运维的检验；
2. 明确当前架构的容量边界与技术债务，为下一阶段演进（用户规模增长、生产数据库选型、限流治理等）划定决策依据；
3. 与其余各类逆向工程文书（需求规格说明书、体系结构设计说明书等）共享同一套术语与事实基线，形成完整的文档闭环。

### 1.2 项目背景

xju-feiyue 是面向新疆大学在校生与毕业生的社区式知识沉淀平台，核心是 7 分类（research/course/recommend/competition/kaggle/tools/life）长文 Markdown 笔记（草稿→发布、点赞、锚点引用评论），并围绕其扩展出五个独立纵切功能：共享课程资料库（`frontend/src/features/materials/`）、CCF 会议截稿追踪器（`frontend/src/features/conferences/` + `backend/app/services/conferences_crawler.py`）、导师/院校招生情报库（`frontend/src/features/schools/`）、教务成绩单一键导入的学分自查（`frontend/src/features/credits/` + `extension/content.js`），以及 DeepSeek 驱动的个性化问候与 AI 笔记润色（`backend/app/services/ai_compose.py`、`backend/app/services/greeting.py`）。后端为 FastAPI 单体（`backend/app/`），前挂三套物理隔离的存储域：一个可写主库（`labnotes.db`）与两个只读、可热重载的外部导出 SQLite（`data/schools/schools.sqlite`、`data/conferences/conferences.sqlite`）。

项目由 1 名人类负责人（winbeau，产品设计/部署运维/内容导入）与多个 Claude Opus 结对实例协作完成，全程 182 次提交，开发周期约 26 天（`README.md` "开发历程亮点""规模数据"两节；该组运行时统计数字与《架构逆向总览》第 8 节所列"陈旧点"分属不同性质——后者指向路由数量/schema 形状等**架构性断言**已过时，前者是 2026-06-04 时点对生产库的**一次性统计快照**，二者不冲突，本报告予以采信）。这一研发组织形态（个位数人力、AI 结对、短周期迭代）是理解本项目"低成本—轻量架构"选型逻辑的关键背景，将在第 4、5 节economic/operational 可行性论证中反复引用。

## 2 现有系统与技术现状概述

### 2.1 系统定位与功能范围

xju-feiyue 是一个中等复杂度的全栈 Web 应用：React 18 单页应用 + FastAPI 单体后端 + 三域 SQLite 持久化 + 一个外部 AI 服务依赖（DeepSeek）+ 一个浏览器扩展执行者。整体部署拓扑（引自《架构逆向总览》第 1 节，本报告据可行性论证需要精简复述）：

```
浏览器 SPA (React 18, feiyue.selab.top / winbeau.top)
        │ Bearer JWT / camelCase JSON
        ▼
nginx 反代 ──▶ FastAPI 单体 (backend/app, 11 router / 60 路由装饰器)
        │                 │主库(读写)      │schools(只读)   │conferences(只读)
        ▼                 ▼               ▼                ▼
浏览器扩展/用户脚本    labnotes.db     schools.sqlite    conferences.sqlite
(jwxt 教务页抓取)      (SQLite+Alembic)  (mode=ro,外部导出) (mode=ro,自写爬虫)

外部依赖：DeepSeek（AI 润色/摘要/会议信息抽取）· open-meteo（问候用天气）
          ccfddl.com（CFP 主源）· DuckDuckGo/ddgs（会议信息兜底搜索）
```

## 3 技术可行性

本节逐层论证六个关键技术选型点的成熟度与合理性：前端 React18+Vite+TS、后端 FastAPI+SQLAlchemy async、三存储域只读热重载、DeepSeek AI、pdf.js 客户端解析、浏览器扩展跨源中转。

### 3.1 前端：React 18 + Vite + TypeScript

**成熟度**：React 18.3、Vite 5.4、TypeScript 5.6 均为各自生态在项目起始时点（2026-05）的稳定长青版本，无预览特性依赖。`docs/design-decisions.md` R1 显式记录了一次主动的版本锁定决策——"`pnpm create vite` 默认会拉 React 19 / Vite 8 / TS 6……锁版本可避免无意识跨大版本升级带来的兼容问题"，说明团队具备版本治理意识，而非被动追新。

**选型合理性**：

- TypeScript strict 三件套（`strict` + `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes`，`docs/design-decisions.md` R1）在编译期堵住了数组越界访问、可选属性误赋值两类常见运行时错误，为一个仅 1 名人类负责人审阅代码的团队提供了必要的静态防护网，弥补人工 code review 带宽不足。
- 状态管理选型收敛克制：Zustand 仅 2 个 store（authStore、draftStore，《架构逆向总览》3.4/3.5 节），其余服务端状态全部交给 TanStack Query v5（`staleTime 30s`、写后 invalidate + 乐观更新，`frontend/src/main.tsx`/`App.tsx`）。这种"客户端状态最小化、服务端状态统一交给数据获取库"的模式是当前前端工程的主流最佳实践，技术债务风险低。
- 编辑器层遇到真实生态缺陷时team展现了自主排障能力：`@codemirror/view` 本地打了 patch（`patches/@codemirror__view.patch` + `ime-patch.test.ts` 读 dist 断言已应用）以修复破坏中文输入法（IME）组合输入的正则问题——这是中文场景下 CodeMirror 6 的已知痛点，team 未回避而是自行修补并写回归测试固化，证明技术风险可控。
- 性能预算已量化并留痕：首屏 JS gzip ≈144 KB（`README.md`"性能预算"），全路由 `React.lazy` + `Suspense`（`frontend/src/router.tsx`）；编辑页因 CodeMirror + diff-match-patch + KaTeX 依赖较重达到 gzip 464 KB，已被文档化为已知偏差而非隐藏债务。

**结论**：前端技术栈成熟、选型克制、对已知生态缺陷有主动应对记录，技术可行性充分。

### 3.2 后端：FastAPI + SQLAlchemy 2.x async ORM

**成熟度**：FastAPI + Uvicorn 是 Python 异步 Web 框架的主流工业级选型，SQLAlchemy 2.x 的 asyncio 支持与 Alembic 迁移工具链均已进入稳定期。

**选型合理性与规范性证据**：

- 迁移链严格线性 `0001→0009`（`backend/alembic/versions/`），从 `0001` 初建 users/notes/drafts/likes/comments，到 `0006` 引入自引用 `material_files` 树、`0008` 引入 `users.role` 并回填 `admin_sid`、`0009` 引入 `material_notice` 单例表，体现了规范的 schema 演进纪律，而非"改代码不留痕"的野路子做法。
- 全局 SQLite 连接事件监听器强制发 `PRAGMA foreign_keys=ON`（`backend/app/db/base.py`），弥补了 SQLite 默认不强制外键约束的已知短板，使 `ondelete='CASCADE'`（如删除 `Note` 级联删除 `Like`/`Comment`）在 SQLite 上真正生效——这是一处对底层存储引擎特性有清醒认知的工程细节。
- 一对多关系普遍设为 `lazy='raise'`（禁止 async 隐式懒加载）以规避 `MissingGreenlet` 错误，代价是资料树等端点必须从单条 flat `SELECT` + Python 端手工装配响应对象（`backend/app/services/materials.py` `build_file_tree`）——这是 async ORM 场景下的已知陷阱及其正确应对方式，技术选型本身审慎。
- `users.sid`（11 位学号字符串）作为自然主键贯穿几乎全部业务表的外键（notes/drafts/likes/comments/material_resources 均以 `sid` 为归属键），省去了引入独立 UUID/自增主键再做业务键唯一约束的双轨维护成本，符合"学号即身份"这一高校场景的领域语义。

**当前边界与风险（须在扩展规划中正视）**：

- 生产数据库当前为 SQLite（单文件、单写者语义），`asyncpg` 依赖虽已声明但未启用（第 2.2 节已述）。SQLite 在当前 108 用户/低并发写入场景下技术上完全够用，但若用户规模量级提升或引入更高频写操作（如高并发点赞/评论），需要提前规划向 PostgreSQL 的迁移路径——好在 SQLAlchemy 2.x + Alembic 的技术选型已为此预留了迁移余地（ORM 层无 SQLite 专有语法泄漏到业务代码），迁移成本可控。
- `services/notes.py` 的 `list_notes()` 全量载入后在 Python 内排序/过滤/分页，源码注释自陈"仅适合数千量级"——这是技术选型对当前规模的诚实边界声明，而非缺陷；一旦笔记量增长到万级需重构为数据库端下推查询（`services/schools_query.py` 对 schools 域已采用 SQLite FTS5 下推查询的模式可作为参考实现）。

**结论**：后端技术选型成熟稳健，迁移与外键约束等基础设施规范，SQLite 单机方案与当前规模相匹配，已知边界有明确的技术升级路径，技术可行性充分。

### 3.3 三存储域只读热重载架构

这是本系统最具工程巧思的技术选型之一：主库（可写 ORM+Alembic）与 `schools`/`conferences` 两个只读外部导出 SQLite 物理隔离（《架构逆向总览》4.2 节）。

**技术实现与成熟度**：

- `backend/app/db/schools_engine.py` 与 `conferences_engine.py`（孪生实现）采用单例 holder 模式：只读引擎以 `mode=ro&immutable=1&uri=true` 连接字符串参数 + 应用层再发 `PRAGMA query_only=ON` 形成三重防御，杜绝应用误写只读域；
- 每次 `get_engine()` 轮询文件 `mtime`，变化则加 `asyncio.Lock` 换新引擎并 dispose 旧引擎，配合 sibling `manifest.json` 校验，实现了**无需重启进程**的数据热更新；
- 写者一侧（`services/conferences_crawler.py crawl_sync()`）遵循"临时文件 → `sqlite3` 直写 → `os.replace()` 原子换 → 重写 manifest"的经典模式，读者永不会看到半成品文件；
- 数据缺失时抛 `*DataMissing` 异常，路由层译为 HTTP 503，不影响应用整体启动（`schools.py`/`conferences.py` 路由降级逻辑）。

**选型合理性**：该模式本质是"文件系统级双缓冲 + mtime 轮询"，无需引入 Redis/消息队列/独立配置中心等重量级基础设施即实现了外部数据源（导师库来自 supervisor-claw 离线导出、会议库来自自写爬虫）与主业务库的物理解耦、故障域隔离（一个只读域数据缺失不影响另一个域与主库）。对于一个单 VPS、单进程部署的中小型系统而言，这是复杂度与收益匹配度很高的技术选择，且已被证明可用——`schools.sqlite`（~20MB，因体积过大 gitignore）与 `conferences.sqlite` 均已在生产环境稳定运行热重载周期（会议爬虫默认 72 小时一轮，`settings.py` `conf_crawl_interval_hours`）。

**局限**：该模式假设单一写者（爬虫为唯一 writer），`schools`/`conferences` 两库的表结构不在 `models.py`/Alembic 管辖范围内，仅能从 `services/schools_query.py`、`conferences_crawler.py` 的 raw SQL 反推（《架构逆向总览》4.2/9 节已列为开放问题），权威 schema 契约在仓库外部的 supervisor-claw 项目——这构成一处跨仓库契约依赖风险，但不影响当前单机部署下的技术可行性。

**结论**：三域拆分与只读热重载机制设计成熟、经生产验证，是与系统规模相匹配的轻量级技术方案，技术可行性充分。

### 3.4 DeepSeek AI 集成

**成熟度与兼容性**：后端通过标准 `openai` SDK（`>=1.50`）指向 DeepSeek 的 OpenAI 兼容 `base_url`（`backend/app/settings.py` `deepseek_base_url`），复用了业界最广泛验证的 LLM 调用协议与 SDK 生态，避免了自研 HTTP 客户端的技术债务。

**工程可靠性设计**：

- `services/ai_compose.py` 的 `summarize_or_fallback()` 承诺"永不抛错"，发布/编辑摘要 DeepSeek 失败时降级为首段启发式摘要；
- `services/greeting.py` 任何异常静默降级为 7 时段本地模板，问候语调用还叠加 3 小时进程内缓存以降低延迟与调用量；
- `/ai/compose/stream`（SSE）异常进入 `{error}` 事件而不破坏 HTTP 状态码，前端 `useSummaryCompose.ts` 另有"5s 无首字节回退一次性重放"的第三层兜底；
- `deepseek_dry_run` 全局 kill-switch（`settings.py`）使测试环境可零联网运行，是良好的可测试性工程设计。

**选型合理性**：DeepSeek 相较于自建/微调模型，免去了团队自行运维 GPU 推理服务的资本与人力投入，与"单 VPS、个位数人力"的项目体量高度匹配；三层降级设计使 AI 依赖不构成系统级单点故障（问候语/摘要在 DeepSeek 完全不可用时仍可用本地规则输出兜底结果）。

**结论**：技术选型成熟、故障降级设计完善，技术可行性充分；唯一待关注点是 `/ai/compose` 端点当前无鉴权（第 6 节详述），属安全维度问题而非技术成熟度问题。

### 3.5 pdf.js 客户端解析

学分自查功能（`/credits`）的核心技术路径是 `pdfjs-dist`（`frontend/package.json` `"pdfjs-dist": "^5.7.284"`）在浏览器端解析教务成绩单 PDF：`features/credits/lib/pdf.ts` 定位文本 → 按 y 坐标聚行 → 列模型解析 → 正则匹配"通识选修·X模块" → 按模块学分与四史/美育/劳动子规则输出通过/不通过报告（`parseTranscript.ts`+`rules.ts`）。

**技术成熟度**：pdfjs-dist 是 Mozilla 官方维护的 PDF.js 项目的 npm 发行版，是浏览器端 PDF 解析的事实标准，版本 5.7 已相当成熟。

**选型合理性**：全部解析逻辑在客户端完成，**PDF 全程不落服务器**——后端 `routes/transcript.py` 仅承担字节级中转（详见 3.6 节），不参与任何解析计算。这一选型同时达成了两个目标：其一，零后端算力与存储开销（与第 4 节"低成本架构"论证呼应）；其二，隐私保护（教务成绩单包含身份证号、家庭住址等敏感信息不经过服务器持久化，与第 6 节社会/法律可行性论证呼应）。这是一个"技术选型服务于隐私与成本双重约束"的典型范例。

**结论**：技术成熟、选型合理，且与其他可行性维度形成正向协同，技术可行性充分。

### 3.6 浏览器扩展跨源中转

教务成绩单自动导入依赖 `extension/content.js`（MV3 content script，`world: MAIN`）与后端 `routes/transcript.py` 的 `transcript-stash` 端点配合完成跨源数据搬运。

**技术必要性**：feiyue 与教务系统 `jwxt-443.webvpn.xju.edu.cn` 跨源，浏览器同源策略禁止前端直接跨域取用户在教务系统的会话态数据；CAS 登录重定向后 COOP（Cross-Origin-Opener-Policy）会切断 `window.opener`，使 `postMessage` 方案不可靠（`transcript.py` 模块 docstring 明确记录该决策动机）。在此约束下，"浏览器扩展利用用户自身已登录的教务会话抓取 PDF，再以无鉴权中转端点回传"是**务实且技术可行**的解法——扩展从不接触教务密码，也不读取 `webvpn_username`（常为 `HttpOnly`，无法被 content script 读取），而是从下载返回文件名 `<sid>_<ts>.pdf` 中解析学号（《架构逆向总览》5(c)节）。

**风险控制的技术手段**：

- 中转端点挂在既有 `/notes/*` 前缀下（`transcript.py` `router = APIRouter(prefix="/notes", ...)`），无需变更生产 nginx location 配置；
- 内存态 `_STASH: dict[str, tuple[bytes, float]]`，5 分钟（`_TTL_SECONDS = 300.0`）过期即被 `_gc()` 清理；
- 上传校验 `data[:5] != b"%PDF-"` 拒绝非 PDF 内容，且 10MB（`_MAX_BYTES`）体积上限；
- 取件（`GET /notes/transcript-stash`）需 `Depends(get_current_user)` 鉴权，且仅 `_STASH.pop(user.sid, ...)` 返回**本人**学号对应的暂存件，取后即删。

**技术成熟度评估**：该方案未使用任何超出标准 Web 平台能力的技术（MV3 content script + `fetch(no-cors)` + FastAPI 内存字典），实现简单、依赖面窄，技术风险低。其代价是 POST 端点本身设计上即为无鉴权（原因见上），这一点已被架构基线明确标注为"故意"设计而非疏漏，本报告将其残余风险归入第 6 节安全可行性讨论。

**结论**：面对真实存在的浏览器安全策略限制（COOP 断 opener），选择的跨源中转方案技术可行、实现代价低，是恰当的工程折中。

### 3.7 综合技术可行性结论

六层技术选型（前端框架、后端框架、存储分域、AI 集成、客户端 PDF 解析、扩展中转）彼此之间耦合适度、职责清晰，且均已在生产环境实际验证可用（108 用户、85 笔记、230+ 会议记录、55 个资料文件的真实运行数据）。团队展现出的技术能力（本地打 CodeMirror patch、镜像实现前后端 diff/拖拽/问候校验双份逻辑、`PRAGMA foreign_keys=ON` 等底层细节留意）表明其具备驾驭当前技术栈复杂度的工程能力。技术可行性评估结论：**充分可行**。

需要在后续演进中持续关注的技术边界（非否决项，是规划输入）：单进程内状态（问候语 3h 缓存、会议爬虫单写者、transcript-stash 内存字典）尚未验证在多 worker/水平扩展场景下的行为；生产数据库 SQLite→PostgreSQL 的迁移路径需要在用户规模显著增长前明确规划。

## 5 操作可行性

### 5.1 部署流程：`deploy.sh`

`deploy.sh`（仓库根）实现了一个具备幂等性与可诊断性的手动部署流程：

```
git fetch origin main
  └─ LOCAL == REMOTE ? 跳过 pull : git pull --ff-only   # 即使已最新也不早退（脚本注释：
                                                          # 曾有人/agent 先手动 git pull，
                                                          # 再跑 deploy.sh 导致预期副作用缺失）
可选 --pull-data
  └─ make schools-pull-force / conf-pull-force           # 绝不拉 state/，避免回滚生产 DB
cd backend
  └─ uv sync --quiet
  └─ uv run alembic upgrade head
sudo systemctl restart feiyue-backend.service
sleep 3
curl -sf --noproxy 127.0.0.1 http://127.0.0.1:8001/health  # --noproxy 绕过 GFW 代理环境变量
  ├─ 成功 → echo "backend healthy"
  └─ 失败 → 打印代理环境变量 + curl -v trace + journalctl（按 InvocationID 过滤，
             只看本次重启后的日志，排除历史 SIGTERM 143 噪音）→ exit 1
（health check 通过后才继续，体现"后端先行、前端随后"的部署顺序）
cd frontend && pnpm install && pnpm build
```

该流程有三点值得在操作可行性论证中特别指出：

1. **幂等设计**：即使 `git fetch` 后发现 `LOCAL == REMOTE`，脚本仍会继续执行 `uv sync`/`alembic upgrade`/`systemctl restart` 而非早退，脚本注释明确说明这是为了应对"运维人员已手动 `git pull`"的实际场景，避免因为跳过而遗漏迁移或重启；
2. **失败可诊断**：健康检查失败时自动打印代理环境变量、`curl -v` 原始追踪、以及**按 systemd InvocationID 过滤**的日志（避免历史重启周期的噪音日志误导排障），这大幅降低了非核心开发者（如临时代打运维的协作者）独立排查故障的门槛；
3. **代理感知**：`--noproxy 127.0.0.1` 显式规避 shell 环境中为翻墙工具配置的 `http_proxy` 误将本机健康检查请求路由出网导致假失败——这是华为云 VPS 出海访问 DeepSeek/Hugging Face 等服务需要代理，但本机自检不应走代理这一现实运维场景的正确处理。

### 5.2 systemd 托管

生产后端进程由 systemd 单元 `feiyue-backend.service` 托管（`deploy.sh` `sudo systemctl restart feiyue-backend.service`；`CONTRIBUTING.md` "生产：`systemd` 守 `uv run uvicorn`，监 `127.0.0.1:8001`"与之一致）。**需要指出的文档偏差**：`README.md`"部署"节写作 `aurash-backend.service`，与 `deploy.sh`/`CONTRIBUTING.md` 实际操作的 `feiyue-backend.service` 不一致（《架构逆向总览》第 8.10 节已列为陈旧点），本报告以 `deploy.sh` 真实执行的单元名为准。这一处细节偏差本身不影响系统可操作性（因为真正被执行的脚本是准确的），但提示 README 类描述性文档已滞后于运维实践，存在新接手人员按文档误操作单元名的风险，已在第 7 节纳入改进建议。

### 5.3 定时任务：`cron_install.sh`

`scripts/sync/cron_install.sh` 安装 `*/30` 周期的 `make sync-push-quiet` cron 条目，用于将主库/uploads/密钥同步至 HF 私有 Dataset。脚本对无人值守场景做了细致的鲁棒性处理：

- 显式烘焙 `PATH`（`$HOME/.local/bin:/usr/local/bin:/usr/bin:/bin`，因 cron 默认精简环境无法解析 `uv`/`make`）与当前会话的代理环境变量（`http_proxy`/`https_proxy`/`no_proxy` 等，因 huawei2 出海访问 huggingface.co 依赖本地代理，cron 环境缺省不带此变量将导致同步任务挂起至超时）；
- 幂等安装：`crontab -l | grep -v 'make sync-push-quiet' || true` 先移除旧条目再追加新条目，重复运行不会产生重复任务；脚本注释解释了为何要在 `set -euo pipefail` 下用 `|| true` 兜底（空 crontab 时 `crontab -l` 与 `grep -v` 均以非零码退出，若不兜底会在追加新条目前就终止脚本）；
- `HF_HUB_DISABLE_PROGRESS_BARS=1` 抑制 `huggingface_hub` 的 tqdm 进度条写入无人值守日志。

### 5.4 单人可维护性

结合第 4.2 节所述"1 名人类负责人 + AI 结对"的团队规模与"182 次提交/26 天"的开发节奏，`deploy.sh`/`cron_install.sh` 中大量"解释为什么而非仅是什么"的注释（如上文引用的幂等设计动机、代理陷阱说明）表明脚本本身承担了部分知识传承职责，弥补了团队人数不足带来的隐性知识风险。这种"自解释脚本 + 幂等操作 + 失败自诊断"的运维设计模式，使系统在个位数人力条件下仍具备可持续运维的操作可行性。

### 5.5 用户操作门槛

- **常规使用**：学号+密码登录（`routes/auth.py`）、Markdown 编辑发布，门槛与常见内容平台一致；
- **教务成绩单导入**：设计了三层递进式方案应对不同用户的技术熟悉度——浏览器扩展一键导入（`extension/content.js`）、托管用户脚本（`ImportWizard.tsx` 探测脚本管理器）、手动上传本地 PDF 兜底（`features/credits/` 全客户端解析），`AutoImportButton.tsx` 与 `ImportWizard.tsx` 提供带动画的安装引导向导，降低了非技术背景用户配置浏览器扩展的操作门槛；
- **管理后台**：`require_admin` 对非管理员返回 404（而非 401/403）使 `/admin` 整个界面对普通用户不可发现（`backend/app/deps.py`），前端 `AdminPage.tsx` 亦无导航入口、仅 URL 可达，此设计虽提升了安全性（见第 6 节），但也意味着新管理员上手需要额外的口授路径知识（URL 本身不出现在任何导航中），构成操作可行性中一个需要通过团队内部文档（而非产品 UI）弥补的小缺口。

### 5.6 操作风险提示

- **Bus factor = 1**：当前部署与运维知识高度集中于单一人类负责人；
- **生产 nginx 配置不在仓库版本管理范围内**：《架构逆向总览》第 8.11 节指出，生产 nginx 配置已经过手工修改（含 `/conferences` 等前缀），但仓库内文档（`CONTRIBUTING.md` 等）均已过时，"须 SSH huawei2 直读"才能确认真实路由表——这意味着 nginx 层的关键运维知识存在于仓库之外，一旦负责人不可用，接手者无法仅凭仓库内容复原生产网关配置，是当前操作可行性中最主要的残余风险；
- 该风险不影响"当前系统能否继续稳定运行"（不可行性判定），而是关乎"团队交接/灾难恢复"场景下的操作韧性，已在第 7 节纳入改进建议。

### 5.7 操作可行性结论

部署脚本设计成熟、具备幂等性与失败自诊断能力，定时任务对无人值守场景的边界条件（PATH、代理、日志噪音）处理细致，用户侧操作门槛通过多层兜底方案得到有效控制。操作可行性评估结论：**充分可行**，但生产 nginx 配置游离于仓库版本管理之外、systemd 单元名等运维事实与描述性文档（`README.md`）存在滞后，构成中低等级的交接与灾备风险，建议纳入后续治理。

## 6 社会/法律/安全可行性

### 6.1 教务数据隐私保护

学分自查功能涉及新疆大学教务系统成绩单这一敏感个人数据（通常含身份证号、家庭地址等信息），系统的隐私保护设计体现在两个层面：

1. **PDF 全程不落服务器持久化**：`features/credits/lib/pdf.ts` + `parseTranscript.ts` + `rules.ts` 在浏览器端完成全部解析计算，后端从不接触解析后的结构化学分数据（第 3.5 节已述）；
2. **成绩单原件仅经内存中转，5 分钟 TTL**：`backend/app/routes/transcript.py` 的 `_STASH: dict[str, tuple[bytes, float]]` 是纯内存字典（非数据库、非磁盘文件），`_TTL_SECONDS = 300.0`，且 `_gc()` 在每次读写前主动清理过期条目；`GET /notes/transcript-stash` 读取时 `_STASH.pop(user.sid, None)` **取即删**，不存在成绩单原文在服务端的持久驻留窗口。

这一设计在数据保护原则上符合"数据最小化"与"存储期限最小化"的通行合规导向，法律与社会风险总体可控。

### 6.2 上传内容防存储型 XSS

`backend/app/services/uploads_common.py` 与 `backend/app/main.py` 的 `HardenedStaticFiles` 共同构成两层防御：

- **应用层扩展名黑名单**：`DENY_EXTS` 在上传时拒绝 `.svg`/`.html`/`.htm`/`.xml`（`uploads_common.py`），阻止浏览器将上传内容误当作可执行页面/脚本渲染；
- **响应头强化**：`HardenedStaticFiles.get_response()` 对 `/uploads` 挂载点的**每一个**响应无条件 `setdefault("X-Content-Type-Options", "nosniff")`（`backend/app/main.py`），阻断 MIME 类型嗅探；
- **魔术字节交叉校验**：`sniff_magic()` 对 PDF/OOXML（经 `zipfile` 校验）/OLE2/PNG/GIF/JPEG/WEBP 等格式做文件头与扩展名交叉核验，防止"改名绕过"的可执行文件伪装成允许格式上传；
- **服务端定 MIME**：不信任客户端声明的 `Content-Type`。

`uploads_common.py` 模块 docstring 中记录了一处**尚未在仓库内可验证**的运维依赖：更强的文档类 `Content-Disposition: attachment` 防御被有意放到生产 nginx 层实现（避免破坏头像/PDF 内联预览需求），docstring 附带了明确的 nginx `location` 配置片段供运维落地，但该配置本身是否已在 huawei2 上生效**无法从仓库内容确认**。这是一处需要通过运维 runbook 核实、当前存在"文档声明与生产实况不可验证"缺口的项目，本报告在第 7 节提出改进建议。

### 6.3 无鉴权端点与限流缺口

系统中存在两处刻意设计为无鉴权的端点，均有明确的功能动机，但也构成需要正视的安全可行性讨论点：

1. **`POST /ai/compose`**：源码 `backend/app/routes/ai.py` 第 16-18 行 `@router.post("/compose", ...) async def compose(body: AIComposeIn) -> AIComposeOut` **未挂载任何 `Depends(get_current_user)` 鉴权依赖**，任何未登录访问者均可调用该端点触发对 DeepSeek 的一次真实计费请求。经检索确认，后端未引入任何应用层限流中间件（`grep` 全仓不见 `slowapi`/`Limiter`/`rate_limit` 等依赖或实现，`ai_compose.py` 中出现的 `RateLimitError` 仅是 `openai` SDK 对 DeepSeek 上游 429 响应的异常捕获，属于"感知上游限流"而非"本应用主动限流"）。在当前 108 用户的真实规模下，该端点被恶意批量调用的概率与已观测影响均较低，但这是一处理论上可被脚本化滥用、直接消耗按量付费 AI 预算的敞口；
2. **`POST /notes/transcript-stash`**：中转端点同样无鉴权（第 3.6、6.1 节已述其功能必要性），设计上以 10MB 体积上限、`%PDF-` 魔术字节前缀校验、5 分钟 TTL 限定风险半径；理论上任何人可以向该端点提交伪造的 `sid` 与任意 PDF 文件，但由于取件端（`GET`）要求 JWT 鉴权且仅返回**当前登录用户自身** `sid` 对应的暂存数据，攻击者伪造他人 `sid` 存入的数据不会被泄露给攻击者本人，也不构成对目标用户已有暂存数据的覆盖之外的额外风险（仅可能造成短暂的"占位覆盖"骚扰，5 分钟内自动过期）。

这两处均是"设计上的已知取舍"而非疏漏，但从社会/法律可行性角度看，`/ai/compose` 缺乏限流是当前系统唯一存在直接经济损失敞口的安全设计点，建议列为优先治理项（详见第 7 节）。

### 6.4 管理员身份治理

`backend/app/settings.py` 的 `admin_sid` 字段承担"超级管理员身份锚点"角色：该学号在 `services/auth.py effective_role()` 中被硬编码恒解析为 superadmin，**无视数据库 `users.role` 列的实际取值**（0008 迁移回填 `role` 仅为仪表盘显示口径一致，非权限判定依据）。这是一种"配置常量兜底身份"而非"纯数据库 RBAC"的设计，其优点是数据库迁移/角色列出错不会锁死最高权限运维入口；其代价是权限治理逻辑**部分脱离于数据库这一常规审计客体**之外，若团队人员发生更替，`admin_sid` 常量本身的变更流程（修改 `.env.local` 并重启服务）需要被纳入正式的交接 checklist，而非仅关注数据库角色表。`require_admin` 对非管理员返回 404（而非 401/403）使 `/admin` 整个功能面对未授权访问者不可发现，属于纵深防御中"降低攻击面可探测性"的合理补充手段，但不能替代常规的最小权限与操作审计（`login_events` 表已提供登录审计能力）。

### 6.5 教务系统抓取行为的合规边界

浏览器扩展/用户脚本在用户自身浏览器会话（用户本人已登录 `jwxt-443.webvpn.xju.edu.cn`）下代为发起两个 GET/POST 请求获取成绩单 PDF，扩展本身不持有、不读取用户的教务密码或 `HttpOnly` 会话 cookie（《架构逆向总览》3.6/5(c)节）。行为主体始终是用户本人的浏览器会话，技术实现上未绕过教务系统的鉴权机制。是否符合新疆大学教务系统使用条款、校方数据管理规定，属于校方政策层面的合规判断，超出本报告基于代码事实的论证范围，建议校方/项目负责人在产品对外公开推广前另行确认。

### 6.6 开源许可与合规

项目以 **MIT License** 发布（`LICENSE`，`Copyright (c) 2026 winbeau`），依赖清单中的第三方库均为宽松开源许可（FastAPI/MIT、React/MIT、SQLAlchemy/MIT 等主流许可），无 GPL 类传染性许可引入的合规风险，代码复用与团队协作（`XjuSelab/xju-feiyue` 团队规范库 + `winbeau/Aurash` 部署源双库维护模式）在法律层面畅通。

### 6.7 综合社会/法律/安全可行性结论

隐私保护设计（成绩单不落盘、短 TTL 内存中转）与存储型 XSS 双层防御（应用层黑名单 + `nosniff` 响应头 + 魔术字节校验）已达到与当前系统体量相匹配的合理防护水准；管理员 404 不可发现与 `admin_sid` 恒 superadmin 机制是务实但需配套人工交接纪律的设计。**主要待改进项**集中在：`/ai/compose` 无鉴权且无应用层限流构成直接经济敞口；`Content-Disposition: attachment` 的 nginx 层落实无法从仓库内验证。二者均不构成对系统当前可行性的否定，但应作为优先级较高的治理项纳入下一迭代。综合评估结论：**基本可行，附带明确的、已定位到具体文件与代码行的改进清单**。

## 7 结论与建议

### 7.1 综合结论

xju-feiyue 平台已在生产环境稳定运行约 26 天，服务 108 名注册用户、沉淀 85 篇笔记、230+ 场 CCF 会议记录、55 个共享资料文件，以个位数人力投入与"单 VPS + SQLite + 进程内缓存 + 第三方按量付费 AI"的轻量架构，验证了以下四个维度的可行性：

| 维度 | 评估结论 | 核心依据 |
|---|---|---|
| 技术可行性 | 充分可行 | 六层技术选型均成熟稳健且与规模匹配；已知边界（单进程状态、SQLite 单机）有明确升级路径 |
| 经济可行性 | 充分可行 | 单 VPS+SQLite+按量 AI+免费 HF 备份+全开源工具链，TCO 与规模高度匹配 |
| 操作可行性 | 充分可行 | `deploy.sh`/`cron_install.sh` 幂等且自诊断；主要风险是生产 nginx 配置游离仓库外、bus factor=1 |
| 社会/法律/安全可行性 | 基本可行 | 隐私/XSS 防护设计到位；`/ai/compose` 无限流与 nginx 落实待验证是主要待改进项 |

### 7.2 改进建议

1. **明确生产数据库路线并同步文档**：核实 huawei2 `.env.local` 的实际 `DATABASE_URL`，消除 `backend/pyproject.toml` description（"FastAPI + Postgres + DeepSeek"）与 `settings.py` 默认值（`sqlite+aiosqlite`）及 `backend/.env.example` 注释之间的表述矛盾；若确认继续采用 SQLite，应更新 `pyproject.toml`/`CONTRIBUTING.md` 使技术现状描述与运行时行为一致。
2. **为 `/ai/compose`（含 `/ai/compose/stream`）补充限流治理**：可选路径包括应用层引入轻量限流库（如 `slowapi`）按 IP/会话限速，或在 nginx 层配置 `limit_req`，以消除当前无鉴权端点直接消耗按量付费 DeepSeek 预算的敞口；此为本报告识别出的唯一存在直接经济损失可能性的待改进项，建议优先级最高。
3. **核实并固化生产 nginx 配置**：将 huawei2 实际生效的 `/etc/nginx/sites-available` 配置纳入仓库版本管理（或至少建立定期比对机制），消除《架构逆向总览》第 8.11 节指出的"仓库内无一份对 prod nginx 权威"的交接风险；同时确认 `uploads_common.py` docstring 所述 `Content-Disposition: attachment` 防御是否已按文档片段落地。
4. **同步运维事实性文档**：将 `README.md`"部署"节的 `aurash-backend.service` 更正为实际使用的 `feiyue-backend.service`，降低新接手运维人员误操作单元名的风险。
5. **规划水平扩展路径**：随用户规模增长，评估问候语进程内缓存、CCF 爬虫单写者、`transcript-stash` 内存字典等状态在多 worker/多实例部署下的行为，必要时引入外部共享状态存储（如 Redis）或改造为无状态设计；同期评估 SQLite→PostgreSQL 的迁移时机，`asyncpg` 依赖已预置，ORM 层迁移成本可控。
6. **降低 Bus factor**：将 `deploy.sh`/`cron_install.sh` 已体现的"自解释脚本"传统延伸至 nginx 配置与管理员身份治理（`admin_sid` 变更流程）等仓库外知识，形成正式的运维交接文档，缓解当前单一人类负责人的操作韧性风险。

以上建议均不构成对现有系统可行性的否定性判断，而是在"系统已验证可行"的基础上，为下一阶段规模增长与团队交接提供具体、可执行的治理路线图。
