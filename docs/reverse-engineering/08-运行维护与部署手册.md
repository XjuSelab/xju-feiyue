## 1 部署架构总览

xju-feiyue 生产环境部署于**华为云 VPS（内部代号 `huawei2`）**单机，采用"nginx 反代 + systemd 托管 uvicorn + 三存储域 + HF Dataset 异地备份"的经典单体部署形态，不涉及容器编排或多机集群。

- **域名**：`winbeau.top`（主域名，nginx 直接服务 `frontend/dist` 静态资源）与 `feiyue.selab.top`（镜像，同一 VPS 的多域名 nginx 配置，`README.md` §"部署"）。
- **反向代理**：nginx 同时承担两个角色——① 将 `frontend/dist` 作为静态文件直出；② 将 API 路径反代到本机 `127.0.0.1:8001`（`CONTRIBUTING.md`："后端：`systemd` 守 `uv run uvicorn`，监 `127.0.0.1:8001`"，与 `deploy.sh` 健康检查探测的 `http://127.0.0.1:8001/health` 一致）。nginx location 表的**权威真实内容仓库内不可验证**，详见第 6、8 节。
- **进程托管**：systemd 单元 **`feiyue-backend.service`**（`deploy.sh` 第 72 行 `sudo systemctl restart feiyue-backend.service` 实际操作的对象；`README.md` 写的 `aurash-backend.service` 是历史遗留名，已陈旧，以 `deploy.sh` 为准）。
- **证书**：Let's Encrypt / certbot（`CONTRIBUTING.md`）。
- **三存储域**（详见架构基线 §4.2）：
  1. 主库 `backend/labnotes.db`（SQLite，读写，Alembic 管理 schema，事务性 UGC：用户/笔记/草稿/点赞/评论/登录事件/资料）；
  2. `backend/data/schools/schools.sqlite`（只读，`mode=ro&immutable=1`，外部 supervisor-claw 离线导出）；
  3. `backend/data/conferences/conferences.sqlite`（只读引擎读 / 内部爬虫 `sqlite3` 直写，ccfddl/DeepSeek 抽取产物）。
- **异地备份**：私有 Hugging Face Dataset（`README.md` 示例 `winbeau/xju-feiyue-data`，实际 `repo_id` 由各机器 `~/.config/labnotes-sync/config.toml` 指定），三命名空间 `state/`（主库快照 + uploads + 加密 secrets）、`schools/`、`conferences/`（两个只读参考库镜像）互不覆盖。
- **双代码仓库**：`README.md` 记载"代码双库维护"——`XjuSelab/xju-feiyue`（团队规范库）与 `winbeau/Aurash`（部署源）；`deploy.sh` 用 `git fetch origin main` / `git pull --ff-only origin main` 拉取，具体 `origin` 在 VPS 上指向哪一个仓库**仓库内不可验证**，运维需 SSH 核实 `git remote -v`。

## 3 部署流程

生产部署是**手动触发的脚本化流程**（无 CI/CD 自动发布到生产，`.github/workflows/deploy-pages.yml` 是独立路径，仅把 `site/index.html` changelog 发到 GitHub Pages，与本节部署无关，见架构基线 §3.8）。核心脚本是根目录 `deploy.sh`，通过 `Makefile` 暴露的 `scripts/sync` 系列目标配合数据刷新。

### 3.1 `deploy.sh` 执行步骤

```bash
./deploy.sh [--dry-run] [--pull-data]
```

1. **`git fetch origin main`**，比较 `git rev-parse HEAD` 与 `git rev-parse origin/main`：
   - 若相同：打印 `== HEAD already at $LOCAL`，**不早退**——脚本注释明确说明原因："it's bitten us when someone (or an agent) git-pull'd manually first, then ran ./deploy.sh expecting the side effects"，因此即使代码已最新，后续 `uv sync`/`alembic upgrade`/`systemctl restart` 仍会照常执行（均为幂等操作，代价可接受）。
   - 若不同：`git pull --ff-only origin main`（**非 fast-forward 会直接失败退出**，不做 merge/rebase，保证生产分支线性）。
2. **`--dry-run`**：仅完成步骤 1 的 git 状态检查，`exit 0`，跳过 sync/迁移/重启/构建。
3. **`--pull-data`（默认关闭）**：
   - 依次尝试 `make schools-pull-force` 与 `make conf-pull-force`（对应 `scripts/sync/schools.py pull --force --quiet` 与 `conferences.py pull --force --quiet`）；
   - 两者失败均**非致命**（`if make ...; then ... else echo "!! ... 失败 ..."`），失败后继续用工作区已有的旧数据文件；
   - **关键纪律（`deploy.sh` 注释原文）**：`--pull-data` 只刷新可再生的 `schools/`/`conferences/` 引用数据，**绝不触碰 `state/`**（主库/uploads/secrets）——那是 `sync-push` cron 的职责，且拉取 `state/` 会用陈旧快照回滚生产数据库。
   - 不传该 flag 时打印 `== HF data pull skipped (default; pass --pull-data to refresh schools/conferences)`，直接使用工作区现有拷贝。
4. **依赖同步与迁移**（`cd backend`）：
   ```bash
   /home/winbeau/.local/bin/uv sync --quiet
   /home/winbeau/.local/bin/uv run alembic upgrade head
   ```
   注意 `uv` 路径写死为 VPS 用户 `winbeau` 的本地安装路径，非通用 `uv`（可移植性注意点，见第 8 节）。
5. **重启并等待**：
   ```bash
   sudo systemctl restart feiyue-backend.service
   sleep 3
   ```
6. **健康检查**：
   ```bash
   curl -sf --noproxy 127.0.0.1 http://127.0.0.1:8001/health
   ```
   - `--noproxy 127.0.0.1` 是显式规避该 VPS shell 环境常见的 `http_proxy=http://127.0.0.1:10808`（GFW 代理），否则本机回环请求会被错误代理转发导致假失败——脚本注释专门解释了这一坑，详见第 7 节。
   - 失败时打印四段诊断信息：当前代理环境变量、`curl -v` 详细 trace、以及用 `systemctl show feiyue-backend.service -p InvocationID --value` 拿到本次启动的 `InvocationID` 后 `journalctl _SYSTEMD_INVOCATION_ID=$inv` 过滤日志（只看本次启动之后的输出，避免历史 `SIGTERM 143` 重启噪音）；拿不到 InvocationID 则退化为 `journalctl -u feiyue-backend.service -n 40`。
   - 健康检查失败则 `exit 1`，**不会继续构建前端**（后端优先原则：只有后端确认健康后才动前端）。
7. **前端构建**（健康检查通过后）：
   ```bash
   cd "$ROOT/frontend"
   pnpm install
   pnpm build
   ```
   nginx 直接服务 `frontend/dist` 静态产物，因此 `pnpm build` 是唯一让前端改动上线的步骤；脚本注释说明"deploy.sh was historically backend-only"，前端构建是后补的一环。

### 3.2 Makefile 目标一览

`Makefile` 是仓库根目录的统一入口，`scripts/sync/*` 相关目标一律经 `uv run --project scripts/sync python scripts/sync/<script>.py <args>` 调用，与 `backend/.venv` 物理隔离：

| 目标 | 等价命令 | 作用 |
|---|---|---|
| `sync-bootstrap` | `scripts/sync/bootstrap.py` | 首次配置：校验依赖、写 age passphrase、写 `~/.config/labnotes-sync/config.toml`、`create_repo` |
| `sync-push` / `sync-push-quiet` | `scripts/sync/push.py [--quiet]` | 快照主库 + tar uploads + 加密 secrets → `state/`（cron 用 quiet 版本） |
| `sync-pull` | `scripts/sync/pull.py` | 恢复 `state/`（DB + uploads + secrets），旧文件先 `.bak.<UTC戳>` |
| `sync-status` | `scripts/sync/push.py --status-only` | 打印上次 `state/` manifest 摘要 |
| `sync-cron-install` | `bash scripts/sync/cron_install.sh` | 装 `*/30` cron |
| `schools-push` / `schools-pull` / `schools-pull-force` / `schools-status` | `scripts/sync/schools.py {push,pull,pull --force --quiet,status}` | 驱动 `schools/` 命名空间 |
| `conf-push` / `conf-pull` / `conf-pull-force` / `conf-status` | `scripts/sync/conferences.py {push,pull,pull --force --quiet,status}` | 驱动 `conferences/` 命名空间 |
| `data-push` | `$(MAKE) sync-push && schools-push && conf-push` | 三命名空间一次性全推（新机器/全量对齐场景） |
| `data-pull` | `$(MAKE) sync-pull && schools-pull && conf-pull` | 新机器一键还原（配合 `sync-bootstrap`） |

### 3.3 新机器 / 灾难恢复的最短路径

```bash
git clone <repo> && cd xju-feiyue
make sync-bootstrap                                     # HF 登录 + age passphrase + repo_id
make data-pull                                           # DB + uploads + secrets + schools + conferences
cd backend && uv sync && uv run alembic upgrade head      # alembic 幂等
```
（`scripts/sync/README.md` "Fresh-machine restore" 一节；`alembic upgrade head` 即使已在 head 也安全重复执行。）

## 4 配置项清单

配置源为 `backend/app/settings.py`（`pydantic-settings` `Settings` 类），环境变量优先级由 `model_config = SettingsConfigDict(env_file=(".env", ".env.local"), case_sensitive=False, extra="ignore")` 决定：`.env.local` 覆盖 `.env`，环境变量大小写不敏感，未识别键被忽略而非报错。`backend/.env.example` 是给开发者复制到 `.env.local` 的模板，但**未覆盖 `settings.py` 全部字段**——下表逐项列出并标注该键是否已出现在 `.env.example` 中。

| 环境变量键 | `Settings` 字段 | 默认值 | `.env.example` 是否列出 | 说明与安全注意 |
|---|---|---|---|---|
| `DATABASE_URL` | `database_url` | `sqlite+aiosqlite:///./labnotes.db` | 是 | 生产切换见第 6 节；`.env.example` 注释给出 Postgres 格式但默认仍走 SQLite |
| `JWT_SECRET` | `jwt_secret` | `"dev-only-change-in-prod"` | 是 | **生产必须覆盖**，`.env.example` 建议 `openssl rand -hex 32`；默认值字面量即写着"开发专用"，遗漏覆盖等于让 JWT 签名密钥公开可猜 |
| `JWT_EXPIRE_MINUTES` | `jwt_expire_minutes` | `10080`（7 天） | 是 | 与架构基线 §5(a) 描述的 HS256 7 天令牌一致 |
| `DEEPSEEK_API_KEY` | `deepseek_api_key` | `""` | 是 | 为空时：`/ai/compose` 等走 `summarize_or_fallback` 降级路径；`main.py::_conf_crawl_loop` 检测为空会打 `warning` 并整轮跳过（见第 7 节） |
| `DEEPSEEK_BASE_URL` | `deepseek_base_url` | `https://api.deepseek.com` | 是 | OpenAI 兼容 SDK 指向的 endpoint |
| `DEEPSEEK_MODEL` | `deepseek_model` | `deepseek-v4-flash` | 是 | 模型/额度调整见第 6 节完善性维护 |
| `DEEPSEEK_DRY_RUN` | `deepseek_dry_run` | `False` | 是（`DEEPSEEK_DRY_RUN=0`） | 全局 kill-switch：置真后 `ai_compose.py` 走 echo/假 chunk，不发起真实上游请求，用于测试/离线环境 |
| `CORS_ORIGINS` | `cors_origins` | `http://localhost:5173,http://127.0.0.1:5173` | 是 | 逗号分隔字符串（刻意不用 JSON list，避免 pydantic-settings v2 对 `list[str]` 环境变量做 JSON 解析而在纯 CSV 输入上崩溃）；生产必须改为实际前端域名（如 `https://winbeau.top,https://feiyue.selab.top`），否则跨源请求（如镜像域名与 API 不同源时）被浏览器拒绝 |
| `PUBLIC_BASE_URL` | `public_base_url` | `http://localhost:8000` | **否（.env.example 未列）** | 拼接头像/上传文件的对外可访问 URL 前缀（如 `${public_base_url}/uploads/avatars/xxx.png`），DB 中存的是**绝对 URL**；一旦部署域名变更而未同步更新此值，历史存量记录里的 URL 不会自动重写（`scripts/sync/README.md`："The DB stores absolute `https://winbeau.top/uploads/...` URLs"）——这是域名迁移场景下的隐蔽故障点 |
| `ADMIN_SID` | `admin_sid` | `"20241401231"` | **否（.env.example 未列）** | **安全关键项**：该 sid 在 `services/auth.py::effective_role()` 恒被解析为 superadmin，无视 `users.role` 列；`.env.example` 未列出该键意味着若运维忘记在 `.env.local` 显式设置，生产实际生效的是源码里硬编码的默认学号——建议部署清单中强制核对 |
| `SCHOOLS_DATA_DIR` | `schools_data_dir` | `data/schools` | **否** | 相对 `backend/` 解析（`main.py::_resolve_data_dir`，即 `Path(app/main.py).parent.parent`）；文件缺失不阻塞启动，`/schools/*` 降级 503 |
| `CONFERENCES_DATA_DIR` | `conferences_data_dir` | `data/conferences` | **否** | 同上，供内部爬虫写入 |
| `AUTHOR_SYNC_ENABLED` | `author_sync_enabled` | `True` | **否** | 关闭后 `main.py` lifespan 不拉起 `_author_sync_loop`；测试环境关闭以保持 lifespan 确定性 |
| `CONF_CRAWL_ENABLED` | `conf_crawl_enabled` | `True` | **否** | 关闭后不拉起 `_conf_crawl_loop`（会议爬虫） |
| `CONF_CRAWL_INTERVAL_HOURS` | `conf_crawl_interval_hours` | `72` | **否** | 会议爬虫唤醒周期；单条会议的检查频率另由 `crawl_state` 派生（`unannounced`→1 天、`announced`→5 天、`closed`→停） |
| `DEEPSEEK_TIMEOUT_S` | `deepseek_timeout_s` | `30.0` | **否** | 上游调用超时秒数；`.env.example` 与 `settings.py` 之间的这一遗漏是**完善性维护**候选项（见第 6 节） |

**前端配置**（`frontend/.env.local.example`）：仅一项 `VITE_API_BASE`（默认注释掉）。设置后 Vite 构建时跳过开发期 mock 分发表，所有 API 调用改走真实 `fetch` 到该 base URL；生产构建走 nginx 同源反代场景下通常**不设置**（`baseURL` 默认 `''`，同源直穿）。

## 5 数据同步与备份

同步工具位于 `scripts/sync/`（README 自称沿用旧名 `labnotes-sync`），核心是**注册表驱动**的三命名空间镜像，全部指向同一个私有 HF Dataset（`repo_id` 存于 `~/.config/labnotes-sync/config.toml`，由 `SyncConfig.load()` 读取）。

### 5.1 三命名空间与内容（`scripts/sync/config.py`）

| 命名空间 | 本地路径 | Dataset 内路径 | 形式 | 谁写 |
|---|---|---|---|---|
| `state/` | `backend/labnotes.db` | `state/labnotes.db` | `VACUUM INTO` 快照（明文） | 仅 huawei2（唯一 writer，README 明文约束） |
| `state/` | `backend/uploads/` | `state/uploads.tar` | 确定性 tar（明文，`--sort=name --mtime=@0 --owner=0 --group=0 --numeric-owner`） | 同上 |
| `state/` | `backend/.env.local` + `frontend/.env.local` | `state/secrets.age` | tar + `age -p --armor` 加密 | 同上 |
| `schools/` | `backend/data/schools/{schools.sqlite,manifest.json}` | `schools/*` | 明文（非密，可再生） | 外部 supervisor-claw 离线导出后手动/半自动 push |
| `conferences/` | `backend/data/conferences/{conferences.sqlite,manifest.json}` | `conferences/*` | 明文 | 内部会议爬虫产出后 push |

`config.py::ARTIFACTS` 是 `state/` 一侧的注册表（`Artifact(name, kind, dataset_path, source, sensitive)`），`push.py`/`pull.py` 遍历它按 `kind`（`db_snapshot`/`dir_tar`/`encrypted_tar`）分派处理；新增一类运行时状态只需追加一条 `Artifact` 而不改 `push.py`/`pull.py` 主体逻辑（README "Adding future state"）。

### 5.2 隔离机制：`delete_patterns` 前缀隔离

三条命名空间共用**同一个** HF Dataset，靠 `huggingface_hub.upload_folder(..., allow_patterns=[...], delete_patterns=[...])` 的模式匹配互不清除：

- `push.py`：`delete_patterns=["state/*"]`
- `schools.py`：`delete_patterns=["schools/*"]`
- `conferences.py`：`delete_patterns=["conferences/*"]`

每次推送只清空并重建自己前缀下的文件，**绝不触碰另外两个命名空间**，这是三条独立 push/pull 命令能安全共享一个 Dataset 的关键设计（`scripts/sync/schools.py` 模块 docstring 明确点出这一点）。

### 5.3 加密与快照细节

- **主库快照**（`_common.py::snapshot_db`）：优先 `sqlite3 <src> "VACUUM INTO '<dst>'"`，失败（如 `sqlite3` 不在 PATH）则退化为 `shutil.copy2`（非事务一致，仅作兜底，函数返回值会标注 `used_fallback=True` 供上层记录）。
- **uploads 归档**（`_common.py::archive_dir`）：确定性 tar（固定排序/mtime/owner），使未变化内容产生相同 sha256，从而在 manifest 比对时可跳过重复上传。
- **secrets 加密**（`_common.py::encrypt_secrets/decrypt_secrets`）：`age -p --armor` 对称加密，passphrase 从 `~/.config/labnotes-sync/age.passphrase`（0600）读取；实现上用 `pty`+`pipe` 分离——passphrase 走伪终端（供 `age` 的 `/dev/tty` 交互式提示），二进制 payload 走普通管道（避免走 pty 的 line discipline 导致回显/死锁）。**passphrase 遗失即无法解密任何历史 `secrets.age`**（README 明文警告"irrecoverable"）。
- **Manifest**：`build_manifest()` 记录 `schema_version`（当前 `1`）、`pushed_at`（UTC ISO-8601 带 `Z`）、`pushed_by`（machine_id）、`git_ref`（短 SHA + dirty 标记）、逐文件 `sha256`/大小；`pull.py` 先校验 `schema_version` 与逐文件 `sha256` 再落盘，任一环节失败即中止不覆盖本地文件。

### 5.4 拉取的非破坏性策略

`pull.py`（`state/`）与 `schools.py pull`（`schools/`）在写入本地前，若目标已存在，一律先 `rename` 为 `<name>.bak.<UTC戳>` 再写入新内容（`--force` 只跳过交互确认提示，**不跳过**这一备份步骤）。

### 5.5 定时备份

`scripts/sync/cron_install.sh` 安装：
```
*/30 * * * * cd $REPO_ROOT && PATH=... HF_HUB_DISABLE_PROGRESS_BARS=1 [代理变量...] make sync-push-quiet >> ~/.cache/labnotes-sync.log 2>&1
```
- **仅推 `state/`**（schools/conferences 是 claw/手动驱动，不进 cron）。
- 显式烘焙 `PATH=$HOME/.local/bin:/usr/local/bin:/usr/bin:/bin`（cron 环境精简，`uv`/`git`/`age`/`sqlite3`/`make` 需要能被解析到）。
- 安装时**捕获当前 shell 已导出的代理变量**（`http_proxy`/`https_proxy`/`no_proxy`/大写变体/`all_proxy`）并原样写入 crontab 条目——huawei2 之类 GFW 侧主机访问 `huggingface.co` 必须经本地代理，裸 cron 环境没有这些变量会导致 push 挂起直到超时；**代理地址变更后必须重新运行安装脚本**。
- 幂等：先 `crontab -l | grep -v 'make sync-push-quiet'` 剔除旧条目再追加新条目。
- `HF_HUB_DISABLE_PROGRESS_BARS=1` 防止 `huggingface_hub` 自带的 tqdm 进度条脏写无人值守的 cron 日志。

## 6 维护类型与配置点

按软件维护四分类（纠错性/适应性/完善性/预防性），结合本仓库真实配置点举例：

### 6.1 纠错性维护（corrective）

- **`admin_sid` 默认值管理隐患**：`settings.py` 中 `admin_sid: str = "20241401231"` 是**源码内硬编码的字面学号**，且未出现在 `backend/.env.example` 中（第 4 节已标注）。任何克隆本仓库、未在 `.env.local` 显式覆盖 `ADMIN_SID` 的部署，其超级管理员身份都会绑定到同一个默认 sid。这是一个应通过纠错性维护补丁处理的缺陷模式：可考虑将默认值改为空字符串并在 `Settings` 加载后校验非空（生产环境缺失即拒绝启动），而非依赖运维记忆覆盖。
- **只读库 schema 漂移无编译期防护**：`schools.sqlite`/`conferences.sqlite` 的表结构完全不在 `backend/app/db/models.py`/Alembic 管辖范围内（架构基线 §4.2），服务层 `schools_query.py`/`conferences_crawler.py` 用裸 SQL 字符串操作。若外部 supervisor-claw 一侧升级导出 schema（增删字段/改类型）而服务层 SQL 未同步，会在运行期触发 SQL 异常或——更隐蔽地——静默取到错位的列值。此类问题只能在故障发生后靠日志定位再修复裸 SQL，属典型纠错性维护场景。

### 6.2 适应性维护（adaptive）

- **SQLite → PostgreSQL 切换**：
  1. 改 `DATABASE_URL=postgresql+asyncpg://user:pass@host:5432/labnotes`（`.env.example` 已给出格式）；`asyncpg` 依赖已在 `backend/pyproject.toml` 声明（`asyncpg>=0.30`），无需额外安装。
  2. `backend/alembic/env.py`（async env）强制用 `settings.database_url` 覆盖 `alembic.ini`，因此迁移链 `0001..0009` 理论上可直接对 Postgres 重放；`db/models.py::StringList` 类型本身就是双态设计（PG 走原生 `ARRAY(Text)`，SQLite 走 `JSON`），说明该切换在建模阶段已被纳入考量，但**尚无证据表明已在 Postgres 上实测跑过这条迁移链**，切换前必须在非生产环境完整跑一遍 `alembic upgrade head` 验证。
  3. **两个只读引擎不随主库切换**：`schools_engine.py`/`conferences_engine.py` 是独立的 `mode=ro&immutable=1` SQLite 引擎，与 `settings.database_url` 无关，主库转 Postgres 后这两个只读参考源仍是本地 SQLite 文件。
  4. **`scripts/sync` 的主库快照逻辑尚未适配 Postgres**：`_common.py::snapshot_db` 硬编码调用 `sqlite3 ... VACUUM INTO`，这是纯 SQLite 语义。若主库切换到 Postgres，`state/` 命名空间的 DB 备份分支需要改写（例如换成 `pg_dump`），当前代码中不存在这个分支——这是一处明确的适应性维护缺口，必须在真正切换 Postgres 之前补齐，否则切换后定时备份会静默对着错误的文件路径操作或直接报错。
- **nginx location 需要跟着新 feature 纵切同步扩容**：`CONTRIBUTING.md` 记载的旧 location 正则（`/(auth|notes|drafts|interactions|ai|health|uploads|admin)/*`）不包含 `/materials`、`/schools`、`/conferences`、`/credits`（`docs/plan-materials-integration.md` 已自行标注这段陈旧）。当前代码层面这些前缀均已在 `main.py` 注册路由（`schools.router`/`conferences.router`/`materials.router`），但**是否已在 prod nginx 落地映射，仓库内无法验证**——这是每次新增纵切都需要显式核对的适应性维护点：代码部署完成不等于该 feature 对外可用，还需确认 nginx 反代表已同步扩展，否则前端请求会在 nginx 层直接 404（而非到达 FastAPI 层）。

### 6.3 完善性维护（perfective）

- **DeepSeek 模型/额度调整**：`settings.py::deepseek_model` 默认 `deepseek-v4-flash`，通过 `DEEPSEEK_MODEL` 环境变量即可换模型；`deepseek_timeout_s`（默认 30.0 秒）控制上游超时。扩大调用额度或更换更强模型前应注意：`routes/ai.py` 的 `/ai/compose`（及其 `/stream` 变体）**无鉴权**即可触发付费上游调用（架构基线第 9 节列为开放问题"无速率限制"），额度调大前应先补齐速率限制，否则等于放大了一个未授权的成本敞口。
- **`.env.example` 补全**：第 4 节列出的 `PUBLIC_BASE_URL`/`ADMIN_SID`/`SCHOOLS_DATA_DIR`/`CONFERENCES_DATA_DIR`/`AUTHOR_SYNC_ENABLED`/`CONF_CRAWL_ENABLED`/`CONF_CRAWL_INTERVAL_HOURS`/`DEEPSEEK_TIMEOUT_S` 均已在 `settings.py` 声明可覆盖，但未出现在 `backend/.env.example` 模板中。补齐这些键属于不改变运行时行为、只提升运维可见度的典型完善性维护动作，建议列入下一次文档维护窗口。

### 6.4 预防性维护（preventive）

- **只读库 schema 升级需要与外部 claw 导出同步验证**：`schools.sqlite`/`conferences.sqlite` 权威 schema 在外部 supervisor-claw 仓库的 `EXPORT_SCHEMA_v1.md`（架构基线 §4.2），本仓库无法在编译期发现契约破坏。预防性做法：每次 claw 侧发布新版本导出文件前，先在非生产环境跑一轮 `/schools/list`、`/schools/{advisor_id}`、`/admin/schools/reload` 的回归检查，确认服务层裸 SQL 仍与新 schema 兼容,再灰度到 huawei2。
- **Alembic 迁移链可逆性检查**：迁移链严格线性 `0001_init.py → ... → 0009_material_notice.py`（`backend/alembic/versions/`），`0006_materials.py` 的 docstring 已注明"autogenerate 后手改命名 + down_revision 手钉"，说明该链条对手工编辑较敏感。预防性维护应在每次新增迁移前，对一份干净测试库跑 `alembic upgrade head` 再 `alembic downgrade -1` 验证可逆，避免线性链因手改断裂而在生产 `uv run alembic upgrade head` 时失败。
- **age passphrase 无异地备份**：`~/.config/labnotes-sync/age.passphrase` 是解密 `state/secrets.age` 的唯一密钥，README 明确"irrecoverable"，但仓库内**未见该 passphrase 本身的异地备份机制**（本手册撰写过程中核实发现，非架构基线原有条目，见第 8 节补充）。预防性维护建议：将该 passphrase 以独立于本仓库的安全方式（如密码管理器）另行备份，避免 VPS 单机故障导致历史加密 secrets 永久不可恢复。

## 7 监控、日志与故障排查

### 7.1 健康检查端点

`backend/app/main.py`：
```python
@app.get("/health", tags=["meta"])
async def health() -> dict[str, str]:
    return {"status": "ok"}
```
这是一个**无依赖检查的存活探针**——只要 uvicorn 进程能响应 HTTP 请求就返回 `200 {"status":"ok"}`，**不检测**主库连接、schools/conferences 只读引擎状态或 DeepSeek 可达性。`deploy.sh` 用它作部署后的唯一自动化验收标准；日常监控若需要更细粒度的就绪判断（如"schools 数据是否已加载"），需自行调用 `/schools/meta` 等业务端点辅助判断。

### 7.2 后台任务日志

- **会议爬虫**（`logging.getLogger("xju_feiyue.conf_crawler")`）：`_conf_crawl_loop` 每 `conf_crawl_interval_hours`（默认 72 小时）唤醒一次；若 `settings.deepseek_api_key` 为空，记 `warning("conf_crawl: DEEPSEEK_API_KEY not set, skipping")` 并直接 `sleep(interval)` 跳过整轮（不会更频繁重试）；正常完成一轮后记 `info("conf_crawl: %s", ...)`（过滤掉 `results` 键，避免整表 dump 进日志）；任何异常被 `except Exception` 捕获打 `exception` 级别日志后继续下一轮循环，**不会使 FastAPI 进程崩溃**。
- **作者校准**（`logging.getLogger("xju_feiyue.author_sync")`）：`_author_sync_loop` 每 24 小时跑一次 `services/author_sync.py::repair()`，有 mismatch 才记 `info`；异常同样被吞掉只留 `exception` 日志。
- 两个循环均通过 `main.py::lifespan` 的 `asyncio.create_task` 拉起，进程终止（`systemctl restart`）时在 `finally` 块里 `task.cancel()` + `await` 优雅收尾。

### 7.3 只读库 503 处置

`schools_engine.py`（`conferences_engine.py` 结构孪生）：
- 数据文件缺失时抛 `SchoolsDataMissing`（`RuntimeError` 子类），**不影响应用启动**（`lifespan` 里 `schools_holder.boot()` 允许缺失），路由层捕获后翻译为 `503`。
- `get_engine()` 每次调用都会 `stat()` 校验 sqlite 文件 mtime；变化则加 `asyncio.Lock` 换新引擎并 `dispose()` 旧引擎——这意味着 `make schools-pull-force`/`conf-pull-force`（或内部爬虫 `os.replace()` 原子换文件）落地新数据后，**下一次请求即可见新数据，无需重启后端**。
- 逃生口：`POST /admin/schools/reload`、`POST /admin/conferences/reload`（`routes/schools.py:140`、`routes/conferences.py:63`）可手动强制 `force_reload()`；`POST /admin/conferences/crawl`（`routes/conferences.py:72`）可手动触发一轮爬虫。
- 故障排查顺序：503 → 确认 `backend/data/{schools,conferences}/*.sqlite` 是否存在且可读 → 检查最近一次 `make {schools,conf}-pull(-force)` 或爬虫日志是否成功 → 必要时手动调用对应 `/admin/*/reload`。

### 7.4 GFW 代理导致健康检查假失败

`deploy.sh` 用 `curl -sf --noproxy 127.0.0.1 http://127.0.0.1:8001/health` 探测，**显式绕开代理**；脚本注释直接点出典型故障现象：该 shell 环境常导出 `http_proxy=http://127.0.0.1:10808`（面向 GFW 的本地代理工具），若不加 `--noproxy`，curl 会把对 `127.0.0.1` 的探测请求也转发给这个代理，代理拒绝处理回环地址请求，从而产生"health check failed"的**假阳性**（后端实际健康）。排障时应先看 `deploy.sh` 失败输出里"proxy env"那一段（`env | grep -iE '^(http|https|all|no)_proxy='`），排除该因素后再深入 journalctl 日志。同样的坑也适用于 `scripts/sync/cron_install.sh` 安装的定时任务——反向场景（该场景需要代理才能连上 `huggingface.co`），故该脚本反而是**主动烘焙**代理变量而非规避（见 5.5 节）。

### 7.5 成绩单中转（transcript-stash）内存占用

`routes/transcript.py`（架构基线 §3.1/§5(c)）：`POST /notes/transcript-stash` 未鉴权，将 ≤10MB 的 PDF 按学号存入**进程内内存字典** `dict[sid] = (bytes, expire_at)`，5 分钟 TTL，GET 取走即删。运维需关注：
- 该内存态**不落盘、不进数据库**，仅存在于当前 uvicorn worker 进程内；
- 理论峰值占用 ≈ 并发发起导入且尚未被前端取走的用户数 × 10MB 上限；正常使用下 5 分钟 TTL 会及时回收，但若前端轮询逻辑异常或用户中途放弃，条目会占满 TTL 窗口才释放；
- **若后端未来横向扩展为多 worker/多实例**，该内存态不共享（架构基线第 9 节已列为"横向扩展未处理"风险之一）——同一用户的 `POST`（扩展写入）与后续 `GET`（前端取走）若被负载均衡分到不同 worker，会出现"stash 丢失"的表现；当前单 uvicorn 进程部署下无此问题，但需在未来扩容前显式处理。
