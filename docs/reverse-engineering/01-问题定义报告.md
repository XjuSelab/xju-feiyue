## 1 引言

### 1.1 编写目的

本报告的目的是从已上线运行的 xju-feiyue（内部代号 Aurash，站名"费阅"）平台代码中，逆向还原其应当解决的业务问题、系统边界与利益相关者结构，形成一份符合 GB/T 8567-2006《计算机软件文档编制规范》体例的《问题定义报告》，作为后续《可行性研究报告》《需求规格说明书》《概要设计说明书》等文书的立项依据与追溯基准。由于本项目并非按照传统瀑布模型自上而下开发（未见立项文档、无原始需求说明书留存），其"问题—方案"对应关系只能通过 3.5 节所列 6 个纵切 feature 模块（`frontend/src/features/*`）与 3.1 节 11 个后端 router（`backend/app/routes/*`）的实际实现逐条反推。本报告的读者包括：项目验收与文档审计人员、后续承接开发/维护的工程师、以及需要理解本平台立项动机的实验室/学院管理方。

### 1.2 项目背景

xju-feiyue 是面向新疆大学在校学生与校友的社区式知识沉淀平台，由实验室（selab）自建自维护，部署于华为云 VPS（huawei2），域名 `feiyue.selab.top`。"飞跃"取自学生俗语中"保研/深造/能力进阶"之意，站内以长文 Markdown 笔记为核心内容形态，覆盖 research（科研）/course（课程）/recommend（推荐）/competition（竞赛）/kaggle/tools（工具）/life（生活）7 个分类（`backend/app/db/models.py::CATEGORY_VALUES`）。系统起初以 Notion 页面与手写经验帖为内容来源，通过 `scripts/notion_pull.py`/`scripts/xju_pull.py` 等一次性内容管线导入种子数据（详见架构基线 §3.7、§5(f)），随后逐步扩展出五个独立业务纵切：共享课程资料库（`features/materials/`）、CCF 学术会议截稿追踪器（`features/conferences/`）、导师/院校招生情报库（`features/schools/`）、教务成绩单一键导入的学分自查（`features/credits/` + `extension/content.js`）、以及 DeepSeek 驱动的个性化问候与 AI 笔记润色（`features/home/`、`features/editor/ai/`）。截至架构基线成文时，生产环境已有 108 个用户账号、85 篇笔记，后端暴露 60 个路由装饰器（远超陈旧文档所称的 19 路由），已是一个功能完整、持续运行的多 feature 全栈系统，而非单纯的"笔记 Demo"。

## 2 问题定义

### 2.1 项目名称与性质

- **项目名称**：新疆大学飞跃手册（xju-feiyue，费阅）
- **项目性质**：面向在校生与校友的公益性/实验室自建知识分享与工具平台，非商业产品，无付费墙；技术形态为前后端分离的全栈 Web 应用（React 18 SPA + FastAPI 单体），并配有浏览器扩展/用户脚本作为轻量客户端代理。
- **建设单位**：新疆大学 selab 实验室（推断自域名 `feiyue.selab.top` 及部署脚本 `huawei2` 命名）。
- **项目形态归类**：属于"内容社区 + 垂直工具集"复合型平台——核心是社区化长文笔记（类知乎/博客），外围叠加四个具体到"新大学生日常刚需"的垂直工具（资料库、会议追踪、导师情报、学分自查），并以 AI 能力（DeepSeek）贯穿润色、摘要、问候、信息抽取多处，属于典型的"熟人社区 + 强场景工具"组合型产品，而非通用型 SaaS。

### 2.2 当前存在的问题

以下问题条目（编号 PR-01 ~ PR-08）均从对应 feature 模块的代码实现逆向推得——即代码解决方案的存在本身即是问题存在的证据。

**PR-01：新生"飞跃"经验散落于多渠道、缺乏统一沉淀入口**

新大学生在保研、竞赛、工具使用、课程选择等方面的经验，原先分散在 Notion 个人页面、GitHub 上的课程 wiki（`SunSeaLucky/xju-course-wiki`）、以及学长学姐手写的零散笔记（如 `baoyan-*.md`）中，彼此不互通、检索困难、随时间流失。证据：项目建设了一条完整的一次性内容 ingest 管线——`scripts/notion_pull.py`（拉取 43 个 Notion 页面）→`scripts/notion_to_md.py`→`scripts/wrap_frontmatter.py`，以及并行的 `scripts/xju_pull.py`（浅克隆课程 wiki 仓库，17 门课程）→`scripts/xju_to_md.py`→`scripts/xju_wrap.py`，再经 `backend/scripts/seed.py` 统一写入主库 `notes` 表。最终 `content/notes/` 下 62 个 md 文件（43 Notion + 17 wiki + 2 手写）与生产 85 篇笔记的差额（~23 篇）来自站内直接发布的 DB-only 内容——说明系统建成后经验沉淀已从"离线迁移"转为"站内原生沉淀"，印证了"统一入口"正是待解决的核心问题。

**PR-02：教务学分自查繁琐、通识/四史/美育/劳动等特殊学分规则易错**

教务系统（jwxt）只提供成绩流水账，不做毕业学分条件的自动核验；学生须手工对照培养方案逐项统计通识选修模块学分、四史/美育/劳动教育等专项子规则，容易疏漏。证据：`features/credits/lib/parseTranscript.ts` + `rules.ts` 实现了"pdf.js 定位文本 → 按 y 坐标聚行 → 列模型解析 → `'通识选修·X模块'` 正则匹配 → 按模块学分与四史/美育/劳动子规则输出通过/不通过报告"的完整客户端解析流水线，且**全程本地计算、PDF 不上传服务器**，说明系统设计者认定这是一个高频、高出错率、且对隐私敏感（成绩单含隐私信息）的痛点。

**PR-03：教务系统（jwxt）与本平台跨域，成绩单导入存在会话隔离障碍**

学生虽能在浏览器登录 jwxt，但 jwxt 与 feiyue 跨源，WebVPN 登录重定向后 COOP（Cross-Origin-Opener-Policy）会切断 `window.opener`，postMessage 通信不可靠，导致"一键导入"技术上受阻。证据：为此专门设计了 `extension/content.js`（MV3 content script，仅注入 `jwxt-443.webvpn.xju.edu.cn`）在 jwxt 页面内抓取 PDF，`no-cors` fire-and-forget 方式 POST 到未鉴权的 `backend/app/routes/transcript.py::stash_transcript()`（内存 `dict[sid]=(bytes, now+300s)`，10MB 上限 + `%PDF-` 魔术字节校验），再由 `/credits` 页轮询取走即删。这一整套绕行方案的存在，本身证明"浏览器同源策略导致的会话隔离"是必须解决的技术性问题，而非可有可无的便利特性。

**PR-04：保研/考研导师与院校招生信息不透明、检索分散**

导师研究方向、招生动态、院校层次等信息原先分散在各校研究生院官网、导师个人主页、论坛口碑等渠道，缺乏统一检索比对入口。证据：`features/schools/SchoolsPage.tsx` + `AdvisorDrawer`（4 个信息 tab）+ `backend/app/services/schools_query.py`（全字段 filter/sort/分页/FTS5 全文检索 + 拼音模糊匹配下推 SQLite）+ 只读引擎 `schools_engine.py` 消费外部工具 supervisor-claw 离线导出的 `schools.sqlite`（4000+ 导师行，覆盖 C9+ 院校）。该子系统专门为"检索与比对"而生，且刻意与主库物理隔离（只读 + 独立引擎 + 热重载），说明"信息分散、比对困难"是被明确识别并针对性解决的问题。

**PR-05：学术会议投稿截止日期难以人工追踪，易错过 deadline**

科研学生原本需要逐个访问会议官网或 ccfddl 网页确认投稿截止日及当前阶段（是否已公布/是否截止），信息滞后、人工维护成本高。证据：`backend/app/services/conferences_crawler.py` 实现了一套自动化爬虫（每 72 小时全量/增量运行一次，`main.py` lifespan 中的后台循环 `_conf_crawl_loop`），以 `ccfddl` 的 `allconf.yml`/`allacc.yml` 为主源（置信度 0.95，无需 LLM），辅以 DDG 搜索 + 主页抓取 + DeepSeek JSON 结构化抽取兜底，按 deadline 自动派生 `crawl_state`（unannounced/announced/closed）与下次检查时间，维护约 230 条会议记录。这套"自动化替代人工巡检"的机制表明会议截止日追踪的时效性与人工维护成本是明确痛点。

**PR-06：课程资料/共享文件散落于个人网盘与群聊，缺乏结构化统一管理**

课件、作业、答案等课程资料原本散落在个人网盘链接、群文件、邮件附件中，命名混乱、缺少在线预览、检索困难。证据：`features/materials/` 实现了递归文件树（`material_files` 自引用表 + `parent_id` CASCADE）、拖拽排序（`@dnd-kit` + `lib/tree.ts` 投影算法，与后端 `services/materials.py::reorder_file()` 语义镜像）、在线 PDF/Word/Excel 预览（`PreviewPane.tsx` 按扩展懒加载 viewer）、上传冲突自动改名（`unique_upload_name()`）。且该库被设计为**任意登录用户可读、仅所有者可写的共享知识库**（`GET /materials/resources` optional-auth 返回所有人非删资源），说明"资料共享缺乏统一场所"是被针对性解决的问题。

**PR-07：笔记写作门槛高、经验分享意愿受限于文字表达能力**

分享经验的学生笔记写作水平参差不齐、提炼摘要耗时，可能降低 UGC 供给意愿。证据：`backend/app/services/ai_compose.py` 提供 7 种 DeepSeek 驱动的写作辅助模式（polish/shorten/expand/tone/translate/custom/summarize），`features/editor/ai/useAICompose.ts`（一次性润色）与 `useSummaryCompose.ts`（`/ai/compose/stream` 真 SSE 流式摘要，先吐 `: ready` 破 nginx 缓冲）；发布时若摘要为空，`summarize_or_fallback()` 自动兜底生成摘要（DeepSeek 失败降级为首段启发式），确保写作门槛不会成为发布阻塞点。

**PR-08：新用户参与度低、平台缺乏"人格化"引导，管理侧缺乏信任分级与审计能力**

传统信息门户对所有访问者呈现统一内容，缺少个性化触达；同时随着 UGC 与用户量增长，运营方需要用户管理、登录审计与角色分级能力。证据（前半）：`backend/app/services/greeting.py` 为每用户生成 3 条个性化问候（结合 `familiar_name()` 称呼推断、7 时段本地模板、open-meteo 天气定性描述，DeepSeek 生成 + `is_valid_greeting()` 校验去重，任何异常静默降级不影响首屏）。证据（后半）：`backend/app/routes/admin.py` 实现 3 级角色仪表盘（用户列表聚合查询、14 天登录火花线统计、创建用户默认密码 123456、重置密码、角色升降）+ `login_events` 表（0003 迁移，记录 XFF 来源 IP 与截断 UA）；`deps.py::require_admin` 对非管理员返回 404 使整个 `/admin` 面不可发现，`require_superadmin` 对普通 admin 升级为 403（他已知面存在）——这套精细的权限分层与审计设计说明"内容与用户治理"是伴随平台增长而显性化的问题。

## 3 系统边界

### 3.1 系统范围内（做什么）

1. **社区化长文笔记全生命周期**：草稿创建/编辑/AI 润色（`routes/drafts.py` 全 CRUD）→ 发布校验与摘要兜底（`POST /notes/drafts/{id}/publish`）→ 浏览（列表 filter/cursor 分页、hot/latest/liked 视图，`routes/notes.py`）→ 互动（点赞幂等 upsert、评论 keyset 分页与锚点引用，`routes/interactions.py`）→ 编辑/删除（作者自限，PATCH 触发按需 AI 重摘要，DELETE 级联）。**注意：笔记无独立的 POST create 端点，发布是唯一入口**（架构基线 §7）。
2. **AI 能力嵌入写作与首页体验**：一次性润色/摘要生成（`/ai/compose`）、流式摘要（`/ai/compose/stream`，SSE）、个性化问候（`/ai/greetings`，3 小时进程内缓存 + 本地模板降级）。
3. **共享课程资料库**：文件树的增删改查、拖拽排序、多格式在线预览（PDF/Word/Excel/图片/代码）、下载（RFC5987 Content-Disposition + nosniff）。
4. **CCF 会议截稿追踪**：自动爬取（ccfddl 主源 + DDG/DeepSeek 兜底）、状态分类展示（表格/时间线视图）、管理员手动触发重爬。
5. **导师/院校招生情报检索**：只读展示外部导出的导师数据，支持分组、筛选、全文/拼音检索。
6. **教务成绩单一键导入与学分自查**：浏览器扩展/用户脚本代理抓取 PDF → 未鉴权 5 分钟 TTL 中转 → 客户端 pdf.js 解析出学分报告（全程不落服务器持久化存储）。
7. **用户账号与三级角色治理**：注册（管理员创建）/登录/头像/资料修改、管理员仪表盘（用户列表、统计、登录审计、角色升降）。
8. **内容与状态的离线管线与灾备**：Notion/课程 wiki 内容一次性迁移入库、经 HF 私有 Dataset 的定期状态快照与同步（`scripts/sync/`，DB VACUUM 快照 + uploads tar + `.env` age 加密）。

### 3.2 系统范围外（不做什么）

1. **不替代教务系统本身**：不写入、不修改 jwxt 数据，仅借助学生自身会话读取其导出的成绩单 PDF；不提供选课、成绩录入、缴费等教务事务性功能。
2. **不做导师/院校原始数据采集**：`schools.sqlite` 由外部独立工具 supervisor-claw 离线生成，本平台只消费只读镜像（`schools_engine.py`），不实现爬虫或人工录入通道；`services/schools_overlay.py` 中 `note_count=0`/`is_starred=False` 为硬编码桩，UGC 叠加（如收藏导师）尚未落地。
3. **不做会议数据的完全原创化自动抽取**：以 ccfddl 权威数据为主，DDG+DeepSeek 仅为无法从 ccfddl 匹配时的补充兜底，不追求 100% 自动化准确率（架构基线标注 DeepSeek 抽取无置信度背书）。
4. **不持久化存储学生成绩单原文**：`transcript-stash` 仅为内存态 5 分钟 TTL 中转，取走即删，不写入数据库或磁盘，是刻意的隐私边界设计而非功能缺失。
5. **不做逐段 AI diff 接受/拒绝**：`diffEngine.ts` 的 `applyAccept/applyReject` 仅支持整块接受或拒绝，无逐段粒度控制（架构基线 §5(b)）。
6. **不做原创课程 wiki 编纂**：课程 wiki 内容来自外部仓库 `SunSeaLucky/xju-course-wiki` 的浅克隆镜像，本平台不维护其原始内容。
7. **不做移动端原生 App**：交付形态为 React 18 SPA（Web），无 iOS/Android 客户端。
8. **不做强多实例分布式部署假设**：`greeting.py` 问候缓存为进程内缓存、会议爬虫为单写者临时文件+`os.replace()`原子换，均未针对多 worker/多实例场景做协调（架构基线 §9 开放问题）。
9. **不做支付、选课、成绩正式认定等具有法律效力的教务事务**：学分自查仅为学生自助参考工具，不等同教务处官方认定结果。
10. **不做跨校统一账号体系**：账号以新疆大学学号（`sid`，11 位）为自然主键，非新大学号无法登录，导师/院校情报虽覆盖 C9+ 等外部院校，但仅作展示参考，不代表本平台向该校学生开放账号。

## 4 主要服务对象

系统的执行者（Actor）可分为两类：人类角色执行者（由三级角色体系区分权限边界）与外部系统执行者（§3.3 已详述交互性质，此处聚焦其作为"执行者"的动机与诉求）。

### 4.1 人类角色执行者

| 角色代号 | 角色名称 | 典型诉求 | 权限判定代码依据 |
|---|---|---|---|
| `anon` | 匿名访客 | 未登录浏览者，希望免注册即可预览公开笔记、共享资料库、会议列表、导师情报等公开内容，评估是否值得注册 | `deps.get_optional_user` 静默返回 None；前端 `RequireAccess` 按 `mode` 表驱动放行 |
| `user` | 登录学生 | 拥有 `sid` 账号的在校生/校友，核心诉求是发布/浏览/点赞/评论笔记、维护自己的共享资料、导入成绩单自查学分、接收个性化问候、使用 AI 辅助写作 | `deps.get_current_user`（401 兜底）；写操作普遍要求 `owner_sid == user.sid` 或 `author_sid == user.sid` 自限 |
| `admin` | 管理员 | 由 `role` 列区分的运营/维护者，负责用户列表管理、创建账号（默认密码 123456）、重置密码、查看 14 天登录统计火花线、审计登录事件 | `deps.require_admin`：非管理员返回 **404**（而非 401/403），使 `/admin` 面对普通用户完全不可发现 |
| `superadmin` | 超级管理员（`admin_sid`） | 系统创始/最高权限运维者，可执行角色升降等最高权限操作；其身份判定不依赖数据库 `role` 列是否被正确迁移或维护，而是由配置项 `settings.admin_sid` 在 `services/auth.py::effective_role()` 中**恒解析为 superadmin**，即使 DB 列滞后也不会锁死运维通道 | `deps.require_superadmin`：普通 admin 访问返回 **403**（他已知晓该面存在），非 admin 仍返回 404 |

**角色与问题条目的对应关系**：`user` 是 PR-01~PR-07 全部七项问题的直接受益角色；`admin`/`superadmin` 是 PR-08（治理能力缺失）的直接受益角色；`anon` 受益于 PR-01（笔记浏览）、PR-04（导师情报浏览）、PR-05（会议列表浏览）、PR-06（资料库浏览，因 `materials` 读操作对所有非删资源公开）——即系统刻意把"发现/浏览"类诉求向未登录人群开放，只把"贡献/管理"类诉求收紧到已登录及更高权限角色，体现"低门槛获客、高门槛贡献"的社区治理策略。

### 4.2 外部系统执行者

除人类角色外，系统架构中还存在若干"代表某类利益相关方行动"的外部系统执行者，它们虽非人类用户，但在用例分析中应视为独立执行者：

| 执行者 | 代表的利益相关方 | 在用例中的角色 |
|---|---|---|
| DeepSeek | 平台方（技术能力供应商） | 作为"被动服务"型执行者，响应系统发起的润色/摘要/问候/信息抽取请求；系统设计遵循"AI 永不硬失败用户体验"原则（任何 DeepSeek 侧异常均静默降级，见架构基线 §7），说明该执行者被视为不可完全信赖的外部依赖 |
| jwxt 教务系统 | 学校教务处（数据源持有方） | 被动数据源，系统不与其建立任何账号信任关系，仅借道学生自身浏览器会话读取其导出物（成绩单 PDF），是典型的"零信任第三方系统"执行者 |
| 浏览器扩展/用户脚本代理（`extension/content.js` / `feiyue-importer.user.js`） | 学生本人 | 代表学生本人在 jwxt 页面内执行的自动化代理，其行为边界受限于"不读取 HttpOnly cookie""从文件名解析 sid 而非窃取凭证""`window.top !== window.self` 与 `window.__feiyueImporter` 防重入"等约束，体现其作为"学生本人自动化助手"而非"平台后门"的定位 |
| ccfddl.com / DuckDuckGo | 学术共同体（会议信息公共基础设施） | 作为免费公共数据源被动响应抓取请求，系统对其数据新鲜度和准确性做置信度分级（ccfddl 优先，DDG+DeepSeek 兜底降级） |
| supervisor-claw（外部工具） | 实验室内部数据团队/维护者 | 作为"人工在环"的数据供应执行者，离线生成后交付静态文件，系统本身不参与其数据采集过程，只做只读消费与热重载 |
| Hugging Face 私有 Dataset | 平台运维方 | 作为异地灾备/状态同步的存储执行者，被 `cron`（*/30 分钟）周期性调用，不参与实时请求路径 |

## 5 建设动机与预期收益

以下逐条对应 §2.2 问题条目与 §4 服务对象，说明建设动机（MB，Motivation & Benefit）与预期收益。

**MB-01（对应 PR-01，受益角色 user / anon）**
动机：新生"飞跃"经验（保研、竞赛、工具、生活）此前散落在个人 Notion 页、GitHub wiki、口耳相传的手写帖中，代际传承效率低。
预期收益：通过 7 分类统一笔记库 + 一次性内容迁移管线（`scripts/notion_pull.py`/`xju_pull.py`），把历史沉淀的经验一次性"注入"平台，并以草稿→发布的常态化写作流程（`routes/drafts.py`）承接后续新增内容，形成持续增长的机构记忆库（生产已积累 85 篇笔记，仍在增长）。

**MB-02（对应 PR-02/PR-03，受益角色 user）**
动机：教务学分自查繁琐、跨源会话隔离导致"一键导入"技术受阻，若不解决则用户被迫手工下载/上传成绩单甚至手工计算学分，体验割裂。
预期收益：`extension/content.js` + `transcript-stash`（未鉴权、5 分钟 TTL、内存态、取即删）绕开 COOP 限制实现近似"一键"体验；`parseTranscript.ts`/`rules.ts` 客户端解析保证成绩单隐私数据不落服务器，兼顾便利性与隐私合规，预期显著降低学生自查毕业学分条件的时间成本与出错率。

**MB-03（对应 PR-04，受益角色 user）**
动机：保研/考研择校择导师决策信息严重不对称，新大学生相对信息弱势。
预期收益：`schools.sqlite`（4000+ 导师行，覆盖 C9+ 院校）经 `schools_query.py` 提供全文/拼音检索、分组筛选，让学生在同一站内完成"发现—比对"决策链路的前半段，降低信息搜集成本；只读+热重载的架构（`schools_engine.py`）保证数据可随外部工具 supervisor-claw 的新导出定期刷新而不需重启服务。

**MB-04（对应 PR-05，受益角色 user）**
动机：科研学生对会议截稿日的追踪原本依赖零散书签或人工巡检，容易错过窗口。
预期收益：`conferences_crawler.py` 自动化维护约 230 条会议记录的截稿状态（unannounced/announced/closed）与下次检查时间，配合前端 `classify.ts`/`filter.ts`/`sort.ts` 提供表格/时间线双视图，把"人工巡检"转化为"平台代为监控、用户按需查阅"，降低认知负担。

**MB-05（对应 PR-06，受益角色 user）**
动机：课程资料原先散落各处、无统一预览与检索，重复劳动（如反复请学长学姐要资料）普遍。
预期收益：`features/materials/` 提供递归文件树 + 多格式在线预览（PDF/Word/Excel/图片/代码）+ 拖拽排序，且刻意设计为"任意登录用户可读、仅所有者可写"的共享库模式，鼓励资料的自愿贡献与复用，减少信息孤岛。

**MB-06（对应 PR-07，受益角色 user）**
动机：笔记写作与摘要提炼耗费额外精力，可能抑制经验分享的发布意愿。
预期收益：`ai_compose.py` 提供多模式 AI 辅助写作（润色/缩写/扩写/改语气/翻译/自定义/摘要）与真流式摘要生成（SSE，`/ai/compose/stream`），并在发布环节兜底自动生成摘要（`summarize_or_fallback()`），降低"写"与"整理"的门槛，预期提升 UGC 供给意愿与发布频率。

**MB-07（对应 PR-08 前半，受益角色 user / anon）**
动机：传统信息门户千篇一律，新用户首次访问缺乏"被欢迎"的体验，粘性建立慢。
预期收益：`greeting.py` 结合称呼推断、时段模板、天气定性描述与 DeepSeek 生成，在登录/首屏即呈现个性化问候（3 小时缓存 + 任何异常静默降级不影响可用性），提升首次与重复访问的情感体验，预期提升新用户留存与活跃度。

**MB-08（对应 PR-08 后半，受益角色 admin / superadmin）**
动机：随着用户与内容规模增长（108 用户/85 笔记且持续增长），需要具备用户管理、登录审计、角色分级治理能力，同时要保证"运维通道不因数据库列滞后而被锁死"。
预期收益：三级角色体系（`user`/`admin`/`superadmin`）+ `admin_sid` 恒解析 superadmin 的引导启动（Bootstrap）身份机制，兼顾"迁移出错时仍可运维"的健壮性与"权限面对普通用户 404 不可发现"的隐蔽性防护，配合 `login_events` 表提供的登录审计能力，预期为平台的长期稳定运营与安全响应提供基础设施保障。

**综合收益**：上述 8 项建设动机共同指向同一个战略目标——把原本分散于多个异构渠道（Notion、GitHub wiki、口耳相传、教务系统、导师个人主页、会议官网、个人网盘、纯文字写作）的、面向新大学生"从入学到深造"全周期的信息与工具需求，收敛到单一域名（`feiyue.selab.top`）下的统一账号体系与统一交互范式中，以社区内容沉淀（笔记）为核心驱动力，以垂直工具（资料库/会议追踪/导师情报/学分自查）为场景化增值，以 AI 能力（DeepSeek）为体验粘合剂，最终降低新大学生获取"飞跃"所需信息与工具的综合成本。
