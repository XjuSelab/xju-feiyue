## 2 编程语言与语言特性运用

### 2.1 Python 3.11（后端）

`backend/pyproject.toml` 声明 `requires-python = ">=3.11"`。代码库对 3.11 特性的运用具体表现为：

**(1) async/await 贯穿整条请求路径。** 从路由处理函数到服务层到 ORM 会话，无同步阻塞调用穿插。例如 `backend/app/deps.py::get_db()` 是一个 `async def` 生成器依赖：

```python
async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionLocal() as session:
        yield session
```

`get_current_user`/`get_optional_user`/`require_admin`/`require_superadmin`（同文件）均为 `async def`，通过 `Depends()` 层层组合；`backend/app/services/ai_compose.py::compose()`/`stream_chunks()` 内部 `await client.chat.completions.create(...)` 调 DeepSeek（OpenAI 兼容 SDK），`stream_chunks()` 是 `AsyncIterator[str]` 异步生成器，逐 chunk `yield`。CPU 密集型的会议爬虫改为 `asyncio.to_thread(crawl_sync, ...)`（`backend/app/main.py:112`）切到线程池，不阻塞事件循环。

**(2) 类型注解全覆盖，PEP 604 联合语法取代 `Optional[X]`。** 全仓一致使用 `str | None`、`User | None` 而非 `typing.Optional`；函数签名普遍标注返回类型，如：

```python
async def get_optional_user(request: Request, db: AsyncSession = Depends(get_db)) -> User | None: ...
async def health() -> dict[str, str]: ...
```

`from __future__ import annotations` 在需要前向引用的模块中显式声明（`backend/app/services/ai_compose.py`、`services/materials.py`、`db/models.py`、`db/schools_engine.py`、`routes/drafts.py` 等），使类型注解延迟求值，允许类内引用尚未定义的同名类型或减少运行时开销。

**(3) SQLAlchemy 2.x 声明式类型化 ORM。** `backend/app/db/models.py` 全部列用 `Mapped[T]` + `mapped_column(...)` 声明而非旧式 `Column(...)`，例如：

```python
sid: Mapped[str] = mapped_column(String(11), primary_key=True)
preferred_name: Mapped[str | None] = mapped_column(String(120), nullable=True)
notes: Mapped[list[Note]] = relationship(back_populates="author", lazy="raise")
```

`StringList`（同文件）是一个 `TypeDecorator` 子类，`load_dialect_impl` 按方言在 Postgres 用 `ARRAY(Text)`、SQLite 用 `JSON()`，是"一套模型定义、两种物理列类型"的类型系统运用范例。

**(4) Pydantic v2 的两种基类与"薄 schema"约定。** `backend/app/schemas/_base.py` 定义 `CamelModel`/`SnakeModel`：

```python
model_config = ConfigDict(alias_generator=to_camel, populate_by_name=True, from_attributes=True)
```

`UtcDateTime = Annotated[datetime, PlainSerializer(_utc_iso_z, return_type=str)]` 是 `Annotated` + `PlainSerializer` 派生类型的运用，保证所有时间字段序列化为带尾 `Z` 的 ISO-8601 字符串。字段级约束用声明式 `Field(ge=0)`/`Field(ge=1, le=50)`（`schemas/note.py::NoteOut.likes`/`ListNotesQuery.limit`），但**全仓 `grep -rn "field_validator\|model_validator" backend/app/schemas/` 结果为零**——即业务校验（标题非空、发布前必填、summary 清洗）一律下沉到 routes/services 层的显式 `if`/`HTTPException`（如 `routes/notes.py::update_note` 的 `if not body.title.strip(): raise HTTPException(422, "标题不能为空")`），schema 层只承担"边界形状 + wire 命名转换"两个职责。这是一条可核实的、贯穿全仓的约定，而非某个文件的偶然写法。

**(5) 显式吞异常并注释理由。** 后台循环、降级路径普遍使用 `except Exception:  # noqa: BLE001` + 紧邻注释说明为何必须静默（`main.py::_author_sync_loop`/`_conf_crawl_loop`、`services/greeting.py::compose_greetings`/`_fetch_weather`、`db/schools_engine.py::boot`）。`noqa: BLE001` 是对 ruff bugbear 规则"禁止裸 except Exception"的显式豁免声明，配合注释使"刻意降级"与"疏忽裸捕获"在代码审查时可区分。

### 2.2 TypeScript 5.6 strict（前端）

`frontend/tsconfig.app.json` 开启的"strict 三件套"（文件内注释原文如此）：

```json
"strict": true,
"noUncheckedIndexedAccess": true,
"exactOptionalPropertyTypes": true,
"noImplicitOverride": true
```

**(1) `noUncheckedIndexedAccess` 强制每次数组/对象索引访问显式处理"可能越界"。** `frontend/src/features/materials/lib/tree.ts` 中同时可见两种应对方式：已验证越界安全时用非空断言 `flat[dragIndex]!`（`tree.ts:192`，紧邻 `dragIndex < 0` 已提前 `return null`）；未经验证时用 `?? null` 兜底：`const prevItem = items[insertIndex - 1] ?? null`（`tree.ts:212`）。这正是该编译开关意图达成的效果——迫使开发者在每处索引访问旁边显式写下"这里为什么安全"或"这里可能为空"的判断，而非依赖隐式 `undefined` 静默传播。

**(2) `exactOptionalPropertyTypes` 区分"未提供该字段"与"显式提供 `null`/`undefined`"。** `frontend/src/api/schemas/material.ts::ResourceUpdateInSchema` 用 `.nullish()`（等价 `.nullable().optional()`）而非 `.optional()`：

```typescript
export const ResourceUpdateInSchema = z.object({
  title: z.string().nullish(),
  description: z.string().nullish(),
  tag: ResourceTagSchema.nullish(),
})
```

因为 PATCH 语义要求区分"不改该字段"（不传）与"清空该字段"（显式传 `null`），`exactOptionalPropertyTypes` 在类型层面禁止把这两种意图混为一谈。

**(3) `verbatimModuleSyntax` + `isolatedModules` 强制类型导入与值导入分离。** 代码中一致使用 `import type { ZodType } from 'zod'`（`api/client.ts:1`）、`import type { UseQueryResult }`（`features/materials/hooks/useMaterials.ts:6`）等显式 `import type`，避免因 TS 类型擦除导致的运行时误导入。

**(4) Zod v3 运行时校验层与递归 schema。** `frontend/src/api/schemas/material.ts` 用 `z.lazy()` 打破文件树的前向引用环：

```typescript
export const MaterialFileSchema: z.ZodType<MaterialFile> = z.lazy(() =>
  z.object({ /* ... */ children: z.array(MaterialFileSchema) }),
)
```

`NoContentSchema = z.null()` 统一表达 204 响应；每个 `api/endpoints/*.ts` 函数末尾均以 `schema.parse()` 收尾（详见第 5 节数据层铁律）。

**(5) React 18 并发特性与 StrictMode。** `frontend/src/main.tsx` 用 `<StrictMode>` 包裹根组件：

```tsx
createRoot(rootEl).render(
  <StrictMode>
    <App />
  </StrictMode>,
)
```

`router.tsx` 全路由 `lazy()` + `Suspense`（自定义 `PageBoundary` 组件统一包裹 fallback），配合 `createBrowserRouter` 实现路由级代码分割；`App.tsx` 中 `QueryClient` 的 `staleTime: 30_000`/`retry: 1`/`refetchOnWindowFocus: false` 默认项体现了对 TanStack Query v5 并发数据获取语义的显式配置而非依赖库默认值。StrictMode 的双调用语义要求所有 hook（尤其 `useEffect`）幂等——`useMaterials.ts` 注释里提到的"sonner 对 strict-mode + HMR 敏感"是该约束在真实代码中留下的注释痕迹。

**(6) 三份互相独立的 tsconfig 工程引用。** `tsconfig.json` 的 `references` 仅含 `tsconfig.app.json`（业务源码）与 `tsconfig.node.json`（Vite 配置本身的类型环境）；第三份 `tsconfig.mock.json` 存在但未被引用（详见第 7 节）。

`backend/pyproject.toml` 的 ruff 配置：

```toml
[tool.ruff]
line-length = 100
target-version = "py311"
[tool.ruff.lint]
select = ["E", "F", "I", "B", "UP", "RUF"]
ignore = ["E501"]
```

`select` 五组规则含义：`E`（pycodestyle 错误）、`F`（pyflakes，未使用导入/变量等）、`I`（isort，导入排序与分组）、`B`（flake8-bugbear，常见易错模式如裸 `except`、可变默认参数）、`UP`（pyupgrade，强制使用当前 Python 版本的现代语法，如 `X | None` 而非 `Optional[X]`——与 2.1 节观察到的联合类型写法吻合）、`RUF`（ruff 自有规则，含 `noqa` 注释有效性检查）。**值得注意的细节**：`line-length = 100` 声明了行宽期望，但同时 `ignore = ["E501"]`（行过长）使 ruff CLI 实际不会因超长行报错——即行宽 100 是团队风格约定（编辑器/评审层面的软约束），而非 lint 层强制门禁，这是配置文件本身就能核实的事实，不是本报告推测。

`mypy>=1.13` 被列为 `[dependency-groups] dev` 依赖，但仓库内无 `[tool.mypy]` 配置块，`Makefile`/`.github/workflows/*.yml`/任何 `*.sh` 中均未出现对 `mypy` 的调用——类型检查工具已声明但未接入任何自动化门禁（详见第 7 节）。

## 4 设计模式与架构模式的实际运用

### 4.1 依赖注入（Dependency Injection）

`backend/app/deps.py` 是 DI 组合根：`get_db`/`get_current_user`/`get_optional_user`/`require_admin`/`require_superadmin` 均通过 FastAPI 的 `Depends()` 声明式注入，且相互组合（`require_admin` 依赖 `get_current_user`，`get_current_user` 依赖 `get_db`）：

```python
async def require_admin(user: User = Depends(get_current_user)) -> User:
    if not is_admin(user):
        raise HTTPException(status_code=404, detail="Not Found")
    return user
```

该 DI 边界的可替换性在测试中被直接验证：`backend/app/tests/conftest.py::client` fixture 用 `app.dependency_overrides[get_db] = _override_get_db` 把生产会话工厂替换为内存 SQLite 会话工厂，是"面向接口而非实现耦合"这一 DI 核心收益的具体证据。

### 4.2 单例（Singleton）——"模块级私有变量 + getter 函数"惯用法

该代码库的单例实现**不是**类的 `__new__` 覆写，而是统一采用"模块级私有变量 + `init_*`/`get_*` 函数"的过程式惯用法，在三处独立实现中重复出现：

1. `backend/app/db/schools_engine.py`：`SchoolsEngineHolder` 类本身封装只读引擎的生命周期，但其**进程内唯一实例**由模块级 `_holder: SchoolsEngineHolder | None = None` + `init_holder(data_dir)`/`get_holder()` 管理；`db/conferences_engine.py` 是孪生实现。
2. `backend/app/services/ai_compose.py::get_client()`：

```python
_client: AsyncOpenAI | None = None

def get_client() -> AsyncOpenAI:
    global _client
    if _client is None:
        _client = AsyncOpenAI(api_key=..., base_url=..., timeout=...)
    return _client
```

惰性初始化 + 全局单一 DeepSeek 客户端实例，避免每次请求重建 HTTP 连接池。

3. `backend/app/services/greeting.py::_weather_cache`/`_GREETINGS_CACHE`：模块级字典充当进程内缓存单例（前者 20 分钟 TTL 的天气缓存，后者 3 小时 TTL 的按 sid 分区问候缓存）。

`SchoolsEngineHolder` 内部另有一个值得记录的模式：**轮询 mtime 热重载**——`get_engine()` 每次调用先 `_maybe_reload()` 比对文件 mtime，变化则在 `asyncio.Lock` 保护下换新引擎并 `dispose()` 旧引擎，`force_reload()` 提供手动逃生口（供 `/admin/schools/reload` 调用）。

### 4.3 工厂（Factory）与生命周期管理

`backend/app/main.py` 顶层的 `app = FastAPI(title="xju-feiyue API", version="1.0", lifespan=lifespan)` 是内联式应用工厂调用点；`lifespan` 用 `@asynccontextmanager` 装饰，是 FastAPI 官方推荐的资源生命周期工厂范式——进入阶段依次创建 uploads 目录、`boot()` 两个只读引擎持有者、按开关拉起两个后台协程（`_author_sync_loop`/`_conf_crawl_loop`）；`finally` 块逆序执行：先 `cancel()` + `await` 两个后台任务（`contextlib.suppress(asyncio.CancelledError)` 吞掉取消异常），再 `dispose()` 两个引擎——是"获取顺序与释放顺序对称"的经典资源管理写法。

`HardenedStaticFiles(StaticFiles)`（`main.py:42-70`）是对 Starlette `StaticFiles` 的子类化扩展（覆写 `get_response` 异步方法，在响应上 `setdefault` 一个安全响应头），属于装饰器模式在类继承层面的变体（而非函数装饰器）。

### 4.4 策略（Strategy）——`build_prompt` 的模式-提示词映射

`backend/app/services/ai_compose.py::build_prompt(mode: str, options: dict) -> str` 把前端 `/ai/compose` 请求体的 7 种模式（`polish`/`shorten`/`expand`/`tone`/`translate`/`custom`/`summarize`）各自映射为一段定制系统提示词：

```python
if mode == "polish": return "你是一个中文文本润色助手。..."
if mode == "shorten": return "把下面文本压缩到原长度的 60%-70%。..."
...
if mode == "summarize":
    title = str(opts.get("title", "")).strip()
    avoid_title = f"笔记标题已经是「{title}」。简介**不要复述、改写或拆解**标题里的字面信息，..." if title else ""
    return "为下面这篇笔记写一段一句话简介，**不超过 35 个中文字符**。" + avoid_title + "..."
raise ValueError(f"unknown mode: {mode}")
```

调用方 `compose()`/`stream_chunks()`/`summarize_or_fallback()` 对 7 种策略一视同仁地经统一入口调度（同一份 DeepSeek client、同一套异常映射），是"算法族按运行时参数选择变体"的策略模式语义实现，只是用 `if`/`elif` 顺序判断而非注册表或多态子类实现——一种轻量级但语义等价的策略模式落地。

### 4.5 适配器（Adapter）——`CamelModel`/`SnakeModel` 的双向 wire 格式转换

`backend/app/schemas/_base.py` 的 `CamelModel` 用 `alias_generator=to_camel` + `populate_by_name=True` 把内部蛇形字段名（`author_sid`、`avatar_thumb`）双向适配为对外 camelCase（`authorSid`、`avatarThumb`），同时仍接受蛇形输入；`SnakeModel` 保留原样、仅服务 `/schools/*` 域（因其数据源——外部 supervisor-claw 导出的 `schools.sqlite`——本身即 snake_case，前端 `features/schools/types.ts` 与之逐字段镜像）。两个基类共享同一份 `from_attributes=True` 配置，对上层调用者屏蔽了"ORM 列名 vs 两套不同网络契约命名规则"之间的适配细节——是典型的适配器模式（统一接口，内部两套转换策略）。

### 4.6 装饰器（Decorator）——路由声明机制

`@router.get/@router.post/@router.patch/@router.delete`（`backend/app/routes/*.py` 全部端点）与 `@app.get("/health")`（`main.py`）是 FastAPI 基于 Python 装饰器语法对"HTTP 动词 + 路径 + 处理函数"三元组的声明式绑定；`@asynccontextmanager`（`lifespan`）、`@event.listens_for(...)`（`db/base.py` 的 `PRAGMA foreign_keys=ON` 监听器、`db/schools_engine.py` 的 `PRAGMA query_only=ON` 监听器）同样是装饰器语法在不同横切关注点（生命周期管理、连接级配置注入）上的运用。

### 4.7 观察者/失效驱动更新（Observer-like Cache Invalidation）与乐观更新+回滚

前端 TanStack Query 的 `invalidateQueries` 机制本质是发布-订阅：写操作成功后失效某个 query key，订阅该 key 的所有渲染中组件被动重新拉取，写操作本身无需知道谁在订阅。`frontend/src/features/materials/hooks/useMaterials.ts` 把"资源详情 + 文件树 + 资源列表三个 key 一起失效"封装为可复用的 `useInvalidateTree()` hook：

```typescript
function useInvalidateTree() {
  const qc = useQueryClient()
  return useCallback((rid: string) => {
    void qc.invalidateQueries({ queryKey: keys.files(rid) })
    void qc.invalidateQueries({ queryKey: keys.resource(rid) })
    void qc.invalidateQueries({ queryKey: ['materials', 'resources'] })
  }, [qc])
}
```

全站唯一贯彻"onMutate 快照 + 乐观写 → onError 回滚 → onSettled 对账"完整三段式的是 `useReorder()`（同文件 244-288 行）：`onMutate` 中 `cancelQueries` 防止在途请求覆盖乐观态，`qc.setQueryData` 用纯函数 `applyReorder()` 即时改写两份缓存（`files(rid)` 与 `resource(rid).files`），并返回快照作为 `context`；`onError` 用快照回滚两份缓存并 toast；`onSettled` 再 `invalidate` 与服务端对账兜底。

其乐观态计算函数 `applyReorder()`（`frontend/src/features/materials/lib/tree.ts:330-385`）在文档注释中明确写道"必须镜像后端 `reorder_file`（`services/materials.py`）语义"：`inside` 落点追加到目标文件夹子节点末尾（对应后端 `insert_index = len(siblings)`）、`before`/`after` 用"去除拖拽项后的兄弟列表"定位插入位置（对应后端在 `siblings = [row for row in rows if row.parent_id == new_parent_id and row.id != drag.id]` 上取索引）、环路守卫 `isReorderCycle()` 沿新父 `parent_id` 链上溯查是否遇到拖拽节点自身（对应后端 `reorder_file` 内等价的 `while cursor is not None` 上溯循环）。这是一处**刻意的前后端双份业务逻辑镜像**，而非代码坏味道——拖拽的实时反馈要求不能等待一次网络往返，但要求人工保持两侧实现同步。

### 4.8 模板方法式降级链（Template-Method-like Fallback Chain）

`services/greeting.py::compose_greetings()` 与 `services/ai_compose.py::summarize_or_fallback()`/`stream_chunks()` 共享同一降级骨架，在多处以相同结构重复出现（非单一抽象基类，而是过程式的"约定俗成"）：

1. 先查进程内缓存是否命中且未过期（`compose_greetings` 的 `_GREETINGS_CACHE`；`ai_compose.compose` 无缓存但 `deepseek_dry_run` 判断在同等位置）；
2. `dry-run`/无 API key（开发与测试环境）→ 本地确定性兜底（`_distinct_local`/echo 输入）；
3. 否则真实调用上游 DeepSeek，捕获 `RateLimitError`/`APIStatusError`/`(APIConnectionError, APIError)` 映射为 502/504；
4. 任何异常/结果不达标（`is_valid_greeting()` 校验）→ 兜底本地模板/首段启发式摘要（`services/notes.py::summary_from`），**永不 503**（`ai/compose(/stream)` 端点上游失败仍以 502/504 上抛是唯一例外，其余 AI 调用点——greetings、发布/编辑摘要——一律静默降级）。

这是架构基线 §7"AI 永不硬失败用户体验"设计原则在实现层的具体落地方式。

## 5 代码组织与分层约定

### 5.1 前端数据层铁律

```
Component → TanStack Query hook (api/index.ts barrel 导出)
          → api/endpoints/<resource>.ts（Zod schema.parse() 强制运行时校验）
          → api/client.ts::request<T>()（唯一 fetch 封装）
```

以 `frontend/src/features/materials/hooks/useMaterials.ts` 为例：`useResources`/`useReorder` 等 hook 只调用 `@/api/endpoints/materials` 导出的函数（`listResources`/`reorder` 等），从不直接 `fetch`；`api/endpoints/materials.ts` 内每个函数以 `schema.parse(raw)` 收尾（继承自 `api/client.ts::request<T>()` 统一实现，见第 2.2 节引用代码）。约 15 处合法例外（SSE 流式响应、文件预览/下载、`GitHubStars` 组件裸调 GitHub API）绕过该铁律直接 `fetch`，但均在架构基线 §6 中被明确记录为"白名单例外"而非未受控的散点直连——如 `features/materials/components/preview/PreviewPane.tsx` 直接 `fetch(resolveAssetUrl(url))` 取文件预览二进制流。

`api/endpoints/*.ts` 十个文件各自重复实现一份 `authHeaders()` 读取 `localStorage['labnotes.auth.token']`（经 `grep -rn "function authHeaders" api/endpoints/*.ts` 核实，`auth.ts`/`admin.ts`/`drafts.ts`/`materials.ts`/`schools.ts`/`interactions.ts`/`conferences.ts`/`ai.ts`/`uploads.ts`/`notes.ts` 十处独立函数定义），是一处已知但未收敛的重复（架构基线 §3.4 已指出"重复 ~8 次未集中"，本报告核实为 10 处）。

### 5.2 后端 routes/services/db 三层

以 `materials` 域为例，三层职责边界清晰可辨：

- **routes 层**（`backend/app/routes/materials.py`）：仅负责 HTTP 契约——`Depends(get_current_user)` 鉴权注入、调用 `services.materials.ensure_owner()` 做权限判定、把服务层结果/异常映射为响应模型与状态码；不含业务规则本身。
- **services 层**（`backend/app/services/materials.py`）：纯业务函数，`build_file_tree()`（flat SELECT → 递归 dict 装配）、`reorder_file()`（含环路守卫的排序算法）、`soft_delete_subtree()`（Python BFS 软删 + 物理 unlink）、`unique_upload_name()`（冲突自动改名探测）——全部是可独立单测的异步函数，仅在需要向路由层传递错误语义时才 `raise HTTPException`，不直接依赖 `Request`/`Response` 对象。
- **db 层**（`backend/app/db/models.py`）：只定义 ORM 列与关系（`Mapped[...]`/`relationship(..., lazy="raise", passive_deletes=True)`），不含任何业务规则；`MaterialFile.parent_id` 自引用树的"同名唯一性"约束**特意不建 DB `UniqueConstraint`**（因 SQLite 视 NULL `parent_id` 两两不等，DB 约束挡不住根级重名），而是在 services 层 `assert_name_free()`/`_name_taken()` 用显式 `SELECT ... WHERE parent_id IS NULL`（当 `parent_id is None`）与 `WHERE parent_id == parent_id`（否则）两条路径手写判等，是"数据完整性约束该放在哪一层"的一次具体权衡记录。

`lazy="raise"` + `passive_deletes=True` + 从不使用 `cascade="all, delete-orphan"` 是全仓一致的 ORM 关系配置策略（`User.notes`、`Note.likes`/`comments`、`MaterialResource.files`、`MaterialFile.parent`/`children` 等全部如此），其代价是任何需要遍历关系的场景（如资料树）必须由服务层"单条 flat SELECT + Python 分组"手工装配（`materials.py::build_file_tree`/`resources_with_trees`），换来的收益是彻底杜绝 async 环境下的隐式懒加载触发 `MissingGreenlet` 崩溃——`resources_with_trees()` 文档注释明确写道"用于列表视图，否则每张卡片都读 files=[] 显示'0 个文件'"，是该约束在真实功能缺陷上的具体印证。

### 5.3 前后端刻意镜像的双份业务逻辑清单

| 业务规则 | 前端实现 | 后端实现 | 镜像原因 |
|---|---|---|---|
| 文件树拖拽排序 | `features/materials/lib/tree.ts::applyReorder`/`isReorderCycle` | `services/materials.py::reorder_file` | 拖拽需要即时视觉反馈，不能等待网络往返 |
| 问候文本有效性校验 | `features/home/lib/greeting.ts::isValidGreeting` | `services/greeting.py::is_valid_greeting` | 首屏需要同步（无网络往返）展示上次缓存问候，且校验逻辑必须与后端生成时使用的标准一致 |
| 称呼派生 | `features/home/lib/greeting.ts` 的 `familiarName` | `services/greeting.py::familiar_name` | 同上，首屏本地兜底文案需要与后端个性化问候使用同一称呼规则 |
| 文本差异高亮 | `features/editor/ai/diffEngine.ts::computeDiff` | `services/diff.py::compute_diff_segments` | 均基于 `diff-match-patch` 算法（前端 npm 包 `diff-match-patch@1.0.5`，后端同名 Python 包），后端算出 diff 段随响应下发，前端需要用同一算法重新计算/校验渲染 |

四项均属"计算逻辑需要脱离网络往返独立运行"而产生的刻意重复，且均在源码注释中显式声明"必须与对侧保持一致"，构成需要人工维护同步的技术债务（而非疏漏）。

### 5.4 schools/conferences 域的 ORM 逃逸

`schools.sqlite`/`conferences.sqlite` 的表结构**不在** `backend/app/db/models.py`/`Base.metadata`/`backend/alembic/` 的管辖范围内——`services/schools_query.py`/`services/conferences_crawler.py`/`services/conferences_query.py` 全部使用 SQLAlchemy Core 的裸 `text()` SQL 或 raw `sqlite3` 连接直接操作，权威 schema 契约在外部 supervisor-claw 仓库的 `EXPORT_SCHEMA_v1.md`（不在本仓库）。这是架构基线 §4.2"三引擎拆分"在代码组织上的直接体现：ORM 层只对主库（`labnotes.db`）负责，两个只读参考库完全绕过 ORM。

## 6 构建与依赖管理

### 6.1 后端：uv + pyproject.toml

`backend/pyproject.toml` 用标准 `[project]`/`[dependency-groups]`/`[tool.uv]` 三段式声明依赖：

```toml
[tool.uv]
package = false
```

`package = false` 表示该项目不是一个待发布的可安装包，仅作内部应用运行——这与其单体 FastAPI 应用的定位吻合。`[dependency-groups] dev` 隔离开发期工具（`pytest`/`pytest-asyncio`/`pytest-mock`/`ruff`/`mypy`）与运行时依赖（`fastapi`/`sqlalchemy[asyncio]`/`asyncpg`/`alembic`/`pydantic`/`python-jose`/`bcrypt`/`openai`/`diff-match-patch`/`httpx`/`aiosqlite`/`pyyaml`/`python-multipart`/`pillow`/`pypinyin`/`ddgs`）。

`deploy.sh`（架构基线 §3.8）执行顺序是 `uv sync`（同步依赖）→ `alembic upgrade head`（迁移升级）→ `systemctl restart feiyue-backend.service`（重启进程）→ 健康检查——迁移先于重启完成，避免新版本代码遇到旧版本 schema。

**可核实的声明与实际用途落差**：`pyproject.toml` 的 `description = "xju-feiyue API — FastAPI + Postgres + DeepSeek"` 与显式依赖 `asyncpg>=0.30` 暗示生产数据库为 Postgres，但架构基线 §8（文档 vs 代码差异分析）已核实生产/开发目前均实际运行 `aiosqlite`（SQLite），`asyncpg` 处于"声明但未被 `settings.database_url` 实际指向"的备用状态——这是配置文件自身携带的"意图 vs 现状"落差，本报告据实记录，不重复展开架构基线已完成的论证。

### 6.2 前端：pnpm + package.json + Vite

`frontend/package.json` 的 `pnpm.onlyBuiltDependencies: ["esbuild"]` 是对 pnpm 默认阻断第三方包 `postinstall` 生命周期脚本这一安全策略的白名单开孔，仅放行 `esbuild`（Vite 依赖链上需要执行原生二进制下载脚本的包）。构建脚本管线：

```json
"build": "tsc -b && vite build",
"typecheck": "tsc -b --noEmit"
```

`tsc -b`（工程引用模式）先于 `vite build` 执行，即类型检查是构建流程的前置门禁而非事后检查——类型错误会直接阻断产物生成。`vite-bundle-visualizer` 作为 `bundle-viz` 脚本的依赖，用于产物体积分析（按需运行，非默认构建路径的一部分）。

### 6.3 Alembic 迁移体系

`backend/alembic/env.py` 是异步迁移环境，核心是强制用应用自身的 `settings.database_url` 覆盖 `alembic.ini` 静态配置：

```python
config.set_main_option("sqlalchemy.url", settings.database_url)
...
async def run_migrations_online() -> None:
    connectable = async_engine_from_config(..., prefix="sqlalchemy.", poolclass=pool.NullPool)
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()
```

保证迁移永远对齐 pydantic-settings 实际解析出的数据库地址，不会因 `alembic.ini` 与运行时环境变量不一致而误迁移。`backend/alembic/versions/` 严格线性单链 `0001_init.py → 0009_material_notice.py`（无分支合并），`0006_materials.py` 的模块 docstring 明确注明"autogenerate 后手改命名 + down_revision 手钉"——说明团队接受人工订正 autogenerate 生成产物（重命名迁移文件、手动锁定 `down_revision` 链路），而非盲目信任自动生成脚本的默认输出。

### 6.4 Git 提交门禁：Husky + commitlint + lint-staged

`frontend/.husky/pre-commit`：

```sh
#!/usr/bin/env sh
pnpm -C frontend exec lint-staged
```

`frontend/.husky/commit-msg`：

```sh
#!/usr/bin/env sh
pnpm -C frontend exec commitlint --edit "$1"
```

脚本注释解释了 `pnpm -C frontend`（而非先 `cd`）的原因：`.husky` 位于 `frontend/` 子目录下，而 `.git` 在仓库根一级，用 `-C` 选项能保证无论从哪个工作目录触发 hook 都能正确解析到 `frontend/package.json`。`package.json::lint-staged` 配置按文件类型分流：

```json
"lint-staged": {
  "*.{ts,tsx,js,jsx}": ["eslint --fix", "prettier --write"],
  "*.{json,md,css,html}": ["prettier --write"]
}
```

`commitlint.config.js` 继承 `@commitlint/config-conventional`（约定式提交规范）并做了两处放宽：关闭 `subject-case`/`scope-case` 大小写检查、`header-max-length` 从默认值放宽到 100（注释说明是因为团队使用形如 `frontend/round-1` 的带斜杠 scope，超出默认长度上限）。

**该提交门禁体系仅在 `frontend/` 子目录生效**——`.husky` 目录挂载于 `frontend/`，`backend/` 目录下的 Python 改动不经过任何 git hook 校验；`ruff`/`mypy`/`pytest` 均需开发者手动执行，本报告核实仓库内不存在针对 `backend/` 的 pre-commit 钩子或 CI 工作流（`.github/workflows/` 下仅 `deploy-pages.yml` 一份，且只负责把 `site/index.html` changelog 发布到 GitHub Pages，与后端代码质量无关）。

### 6.5 数据同步与部署编排的构建隔离

根 `Makefile` 提供 `sync-*`/`schools-*`/`conf-*`/`data-*` 系列目标，统一通过 `uv run --project scripts/sync python scripts/sync/...` 调用，其虚拟环境（`scripts/sync/.venv`）与 `backend/` 的 uv 环境物理隔离，避免同步脚本的依赖污染后端运行环境（`Makefile` 顶部注释原文如此）。

## 7 代码质量工具链与约束

### 7.1 静态检查矩阵

| 维度 | 后端 | 前端 |
|---|---|---|

### 7.2 pytest 测试基础设施与 AI 零联网策略

`backend/app/tests/conftest.py` 用 `StaticPool` 强制内存 SQLite 单连接共享（保证 `db_session` fixture 与经 `app.dependency_overrides[get_db]` 替换的 HTTP `client` fixture 看到同一份数据），是纯异步集成测试（`AsyncClient` + `ASGITransport(app=app)`，不起真实网络端口）的标准夹具写法。测试环境变量在任何应用导入前强制注入：

```python
os.environ.setdefault("DATABASE_URL", "sqlite+aiosqlite:///:memory:")
os.environ.setdefault("JWT_SECRET", "test-secret-do-not-use-in-prod")
os.environ.setdefault("DEEPSEEK_DRY_RUN", "1")
os.environ.setdefault("AUTHOR_SYNC_ENABLED", "0")
```

`DEEPSEEK_DRY_RUN=1` 是全局 kill-switch，使所有涉及 DeepSeek 的路径（`ai_compose.compose`/`stream_chunks`/`greeting.compose_greetings`）走 echo/本地模板分支，测试套件零联网、零 API 费用；`backend/app/tests/` 覆盖 `test_admin.py`/`test_auth.py`/`test_auth_service.py`/`test_comments.py`/`test_conferences.py`/`test_doc_upload.py`/`test_drafts.py`/`test_greeting.py`/`test_image_upload.py`/`test_likes.py`/`test_materials.py`/`test_notes.py`/`test_note_update.py`/`test_schools.py`/`test_smoke.py`/`test_transcript_stash.py`/`test_author_sync.py` 共 17 个业务测试文件 + `test_smoke.py`，逐个覆盖了架构基线 §3.1 列出的绝大多数路由域。

### 7.3 已核实的工具链断点

以下五项均为可在仓库内直接复核的事实性缺口（配置文件本身即可验证），而非本报告的主观推测：

1. **`mypy` 声明但未挂钩**：`backend/pyproject.toml` `[dependency-groups] dev` 含 `mypy>=1.13`，但全仓无 `[tool.mypy]` 配置段，`Makefile`、`.github/workflows/deploy-pages.yml`、任何 `.sh` 脚本均未出现 `mypy` 调用字样——类型检查目前只能由 Pydantic v2 运行时校验与人工代码评审承担，缺少编译期类型门禁。这与前端"`tsc -b` 是构建前置步骤"形成对比，是后端相对前端在类型安全工具化程度上的一处落差。

2. **`frontend/tsconfig.mock.json` 悬空未接入构建图**：该文件头部注释原文写明"Round 4 uses this tsconfig for mock fixtures...Keep this file checked in but DO NOT reference it from tsconfig.json until mock files exist, otherwise tsc -b reports TS18003"；核实当前 `frontend/tsconfig.json::references` 数组仅含 `tsconfig.app.json`/`tsconfig.node.json`，不含 `tsconfig.mock.json`。而 `frontend/src/api/mock/` 目录（架构基线 §3.4 提及的 mock handler 集合，`auth`/`notes`/`ai`/`notes-images`/`schools`/`conferences` 等 handler）确实存在且在 DEV 环境下被动态 import 使用——即该目录当前处于"运行时在用，但不受 `tsc -b` 静态类型检查覆盖"的状态。

3. **ruff `line-length=100` 与 `ignore=["E501"]` 共存**：行宽约定名义存在但 lint 层实际不报错（3.1 节已述），是声明性风格约束而非强制门禁的一个可验证实例。

4. **Playwright 已配置但未打通**：`frontend/playwright.config.ts` 与 `package.json::"e2e": "playwright test"` 均存在，但架构基线 §2 已核实 dev VPS 缺少 Playwright 所需 OS 级依赖，端到端测试目前处于"配置就绪、执行受阻"状态，不构成实际生效的质量门禁。

5. **提交门禁只覆盖 `frontend/` 子目录**：`.husky/` 挂载于 `frontend/`，`backend/` 的 Python 变更不经过任何本地 git hook 校验；仓库内唯一的 GitHub Actions 工作流 `.github/workflows/deploy-pages.yml` 只负责把 `site/index.html` changelog 发布到 GitHub Pages（架构基线 §3.8），与后端代码质量无关——即后端代码质量目前完全依赖开发者自律执行 `ruff`/`pytest`，没有自动化兜底。

### 7.4 小结

xju-feiyue 的代码质量工具链呈现"前端强、后端弱"的不对称格局：前端有 `tsc -b` 编译期强制门禁、ESLint+Prettier 双重风格约束、Husky 提交前钩子三层叠加；后端仅有 ruff（且行宽规则实际未强制）与一套覆盖面广但需手动触发的 pytest 套件，`mypy` 停留在依赖声明层面。这一格局与架构基线 §7 揭示的"后端是单体、前端是精细化分层 SPA"的整体架构特征相互印证——工程投入的精细化程度在前后端两侧并不对称，是团队在有限资源下的实际取舍结果，而非规范文档层面的疏漏。
