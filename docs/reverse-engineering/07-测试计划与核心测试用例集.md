## 1 引言

### 1.1 目的

本测试计划服务于两个目标：

1. **验收侧**：为 xju-feiyue（内部代号 Aurash）全栈平台已实现的 60+ 后端路由、6 大前端 feature 纵切给出一套可执行、可追溯的核心测试用例集，明确"黑盒功能正确性"与"白盒分支/边界覆盖"两条验证线。
2. **治理侧**：如实标注现有测试资产的覆盖边界与已知盲区（第 6 节），供后续测试补强与风险评审（如安全评审、上线前检查）参照，避免因"测试目录存在即等于测试充分"的误判。

### 1.2 范围

覆盖对象为：

- **后端单元/集成测试**：`backend/app/tests/*.py`，通过 `httpx.AsyncClient` + `ASGITransport` 直接驱动 FastAPI 应用（`backend/app/main.py` 中的 `app` 实例），不经真实网络与真实 nginx。
- **前端单元测试**：`frontend/src/**/*.test.{ts,tsx}`，Vitest + jsdom + Testing Library，覆盖纯函数（`lib/tree.ts`、`diffEngine.ts`、`parseTranscript.ts`、`rules.ts`、`classify.ts` 等）与少量组件（`CategoryBadge.test.tsx`、`Markdown.test.tsx`）。
- **前端端到端测试**：`frontend/playwright/e2e/editor-ime.spec.ts`，仅覆盖 CodeMirror IME 合成回归，且**当前开发 VPS 缺 `libnss3`/`libatk-bridge` 等 Playwright 所需 OS 依赖，无法在本机执行**（见 `frontend/README.md:126`、`docs/design-decisions.md:112` 的一致记载）。
- **不覆盖**：性能测试、压力/并发测试、渗透测试（后两者作为盲区在第 6 节讨论其必要性）；浏览器扩展 `extension/content.js` 与内容管线 `scripts/*` 的离线一次性脚本无自动化测试，超出本计划范围。

## 2 测试策略与层次

xju-feiyue 采用三层测试金字塔，层间边界与职责如下：

```
                         ┌───────────────────────────────┐
                         │   E2E（浏览器真环境驱动）        │
                         │   Playwright · 1 个 spec        │
                         │   editor-ime.spec.ts            │
                         │   状态：配置齐全，dev VPS 因缺   │
                         │   OS 依赖无法本机执行            │
                         └───────────────┬─────────────────┘
                                         │
                 ┌───────────────────────┴────────────────────────┐
                 │        集成测试（ASGI 应用内 + 临时/内存库）      │
                 │  backend: httpx.AsyncClient + ASGITransport      │
                 │           + sqlite+aiosqlite:///:memory:         │
                 │  frontend: 无（前端无组件级集成测试，              │
                 │            design-decisions.md 记载为主动放弃）    │
                 └───────────────────────┬────────────────────────┘
                                         │
     ┌───────────────────────────────────┴───────────────────────────────────┐
     │                     单元测试（纯函数 / 服务层）                          │
     │  backend: pytest (asyncio_mode=auto) —— services/*.py 直调, 无 HTTP      │
     │  frontend: Vitest + jsdom + Testing Library —— lib/*.ts, schemas/*.ts    │
     └───────────────────────────────────────────────────────────────────────┘
```

### 2.1 单元测试层

- **后端**：`pytest` + `pytest-asyncio`，`backend/pyproject.toml` 设 `asyncio_mode = "auto"`，因此绝大多数测试函数无需 `@pytest.mark.asyncio` 装饰器即可被识别为协程测试（`test_admin.py` 仍显式打了该装饰器，属历史遗留写法，在 `auto` 模式下是冗余但无害的双保险）。纯服务层单元测试的代表是 `test_auth_service.py`（不经 HTTP，直调 `app.services.auth.hash_password/verify_password/create_access_token/decode_token`）与 `test_author_sync.py`（直调 `app.services.author_sync.scan/repair`）。
- **前端**：Vitest，`environment: 'jsdom'`，`include: ['src/**/*.test.{ts,tsx}']` 显式排除 Playwright 的 `playwright/**/*.spec.ts`（`vite.config.ts` 注释明确说明原因：两套 `test()/describe()` 语义不同，混扫会互相污染）。测试对象以 `features/*/lib/*.ts` 纯函数为主（`tree.ts`、`diffEngine.ts`、`parseTranscript.ts`、`rules.ts`、`classify.ts`），辅以 `api/schemas/*.test.ts` 对 Zod 边界（如 `NoContentSchema`、递归文件树 `z.lazy()`）的校验。

### 2.2 集成测试层

- **后端**：不使用 `TestClient`（同步）而是 `httpx.AsyncClient(transport=ASGITransport(app=app))`（`conftest.py:65-67`），使测试与生产同为一套 `async def` 调用链，避免同步桥接掩盖协程时序问题（如 SQLAlchemy `lazy='raise'` 的 `MissingGreenlet`）。数据库替换为 `sqlite+aiosqlite:///:memory:` + `StaticPool`（`conftest.py:34-39`），并通过 FastAPI 依赖覆盖 `app.dependency_overrides[get_db]` 注入同一 `session_factory`，使 HTTP 请求与测试内直接的 `db_session` 断言共享同一份内存数据库连接。**该层完整替代了"对 Postgres 做集成测试"的需求**——`conftest.py` 顶部注释明确写明"integration tests against real Postgres are out of scope for now"。
- **前端**：无组件级集成测试（如对 `WritePage.tsx`/`NoteCard`/`Header` 挂载 + 用户交互模拟）。`docs/design-decisions.md:112` 记载了这一取舍的理由："组件级集成测试→工作量大，价值远低于 schema/util 测试"，属主动的测试策略决策而非遗漏，但也构成第 6 节所述的覆盖盲区之一。

### 2.3 端到端测试层

- Playwright 配置齐备（`frontend/playwright.config.ts`：`testDir: './playwright/e2e'`、`baseURL: 'http://localhost:5173'`、`webServer` 自动拉起 `pnpm dev`），但**当前唯一的 spec 是 `editor-ime.spec.ts`**，通过 CDP `Input.imeSetComposition`/`Input.insertText` 驱动真实 Chromium 模拟拼音输入法合成过程，回归验证 `patches/@codemirror__view.patch` 的桌面 EditContext 补丁在真实浏览器渲染管线下确实生效（而不仅是 `ime-patch.test.ts` 静态断言补丁文本存在）。该测试无需后端（`/write` 页面用 `mode:'authed'` + `token:null` 跳过 `/auth/me` 校验），仅 `chromium` 一个 project（CDP IME 能力是 Chromium 专有的）。
- **现状**：`frontend/README.md:126` 与 `docs/design-decisions.md:112` 一致记载：`pnpm e2e` 需要 `libnss3`/`libatk-bridge` 等 OS 级依赖，**当前开发 VPS 不具备**，因此该测试套件在日常开发环境中不可执行，需在"配好环境的机器"（如 CI runner 或本地开发机）上单独运行。这意味着 E2E 层目前是**已声明但未持续验证**的状态。

### 2.4 AI/外部依赖的测试隔离

平台涉及 DeepSeek（AI 润色/摘要/会议抽取）、open-meteo（问候天气）、ccfddl/DuckDuckGo（会议爬虫）等外部网络依赖。测试策略通过**单一全局开关**规避真实联网：

- `conftest.py:15` 在应用导入前设置 `os.environ.setdefault("DEEPSEEK_DRY_RUN", "1")`；`app/settings.py` 的 `deepseek_dry_run` 字段据此在整个测试进程生命周期内恒为真。
- `services/ai_compose.py` 的 `compose()`、`stream_chunks()`、`summarize_or_fallback()` 均在函数入口判断 `settings.deepseek_dry_run`，为真时直接走本地 echo/假 chunk/first-paragraph 兜底分支，**不实例化/调用 `AsyncOpenAI` 客户端**，从而使任何涉及草稿发布（`test_drafts.py`）、AI 问候（`test_greeting.py`）的测试都在零联网条件下确定性通过。
- `test_greeting.py` 额外用 `monkeypatch.setattr(greeting_svc, "_fetch_weather", ...)` 把天气获取替死为返回 `None` 的桩函数，双重保险切断对 open-meteo 的依赖。
- **代价**：`services/ai_compose.compose()`（真实 DeepSeek 单次润色）、`stream_chunks()`（SSE 流式）与 `services/conferences_crawler.py` 的 ccfddl/DDG 抓取路径，在 `deepseek_dry_run=1` 下走的是**兜底分支而非真实网络分支**，其"上游 502/504 上抛""SSE `{error}` 事件"等生产行为**没有任何测试直接验证**（详见第 6 节）。

## 3 测试环境与工具

### 3.1 后端夹具体系（`backend/app/tests/conftest.py`）

| Fixture | 作用域 | 说明 |
|---|---|---|
| `db_engine` | function | 新建 `create_async_engine("sqlite+aiosqlite:///:memory:", connect_args={"check_same_thread": False}, poolclass=StaticPool)`；`StaticPool` 使同一内存库在多个连接间共享，规避内存 SQLite "每连接一份数据库"的默认隔离问题。`async with engine.begin(): conn.run_sync(Base.metadata.create_all)` 用 ORM 元数据一次性建表——**因此测试库结构永远与 `models.py` 当前定义同步，不经过 `alembic upgrade head`**（该点与第 6 节"只读库 schema 无迁移校验"风险相关：主库测试跳过了 Alembic 链路本身，迁移脚本 0001-0009 的正确性不在本套测试的验证范围内）。 |
| `session_factory` | function | `async_sessionmaker(bind=db_engine, expire_on_commit=False)`。 |
| `db_session` | function | 供测试直接注入数据、断言落库结果的会话（"上帝视角"）。 |
| `client` | function | 通过 `app.dependency_overrides[get_db]` 把 FastAPI 的 `get_db` 依赖替换为返回同一 `session_factory` 的会话，再用 `AsyncClient(transport=ASGITransport(app=app))` 发起请求；`finally` 块中 `app.dependency_overrides.clear()` 保证测试间不残留覆盖。 |
| `demo_user` | function | 学号 `20211010001`、昵称 `zilun`、密码 `123456`（bcrypt 哈希）。 |
| `authed_token` / `auth_headers` | function | 先 `POST /auth/login` 换取真实 JWT，再包成 `Authorization: Bearer` 头——鉴权类测试统一走"真登录得真 token"而非直接伪造 `create_access_token`，确保覆盖登录路由本身。 |
| `seeded_notes` | function | 5 篇跨 3 分类笔记 + 4 个用户 + 差异化点赞/评论数，用于 `/notes` 列表排序（hot/latest/liked）、过滤（cat/q/tags）、游标分页的组合测试；docstring 中列出的时间/点赞/评论布局是断言排序正确性的唯一真值表。 |

module 级隔离：每个测试函数独立获得一份全新内存库（`db_engine` 是 function 作用域），**测试之间不共享状态**，这是 179 条用例可并行设想（虽然当前未显式配置 `pytest-xdist`）而不互相污染的基础。

### 3.3 真实数据依赖与 `skipif` 降级

`test_schools.py`、`test_conferences.py` 两个模块依赖仓库外的真实导出文件（`backend/data/schools/schools.sqlite`、`backend/data/conferences/conferences.sqlite`），这两个文件因体积大（schools ~20MB）被 `.gitignore`，仅通过 Hugging Face 私有 Dataset 镜像同步。两模块均在文件级用：

```python
pytestmark = pytest.mark.skipif(not _has_data(), reason="... not present")
```

**这意味着**：在未执行 `scripts/sync/` 拉取外部数据的干净 checkout 或 CI 环境中，这两个文件涉及的全部用例（`test_schools.py` 18 条、`test_conferences.py` 5 条，合计 23 条）会被**静默跳过**而非失败，`pytest` 汇总结果里表现为 `skipped` 而非 `passed`——第 6 节将展开讨论这一"假绿"风险。

### 3.4 前端测试基座

- `frontend/src/test/setup.ts` 仅一行 `import '@testing-library/jest-dom/vitest'`，为 jsdom 环境引入 `toBeInTheDocument()` 等 DOM 断言扩展。
- `ime-patch.test.ts` 的桩替换方式特殊：它不 mock 任何东西，而是从 `cwd` 向上遍历最多 6 层目录，直接 `readFileSync` 已安装的 `node_modules/@codemirror/view/dist/index.js` 源码文本，用字符串包含关系断言本地 patch（`patches/@codemirror__view.patch`）确实已应用到编译产物——这是一种"读已安装依赖的编译产物做回归锁"的非常规但务实的测试手段，专门防止 `pnpm install` 因 patch 失效而静默丢失修复。

## 6 缺陷风险与测试盲区

以下风险按"现状确认程度"排序：前三项已通过直接阅读源码确认为**当前事实**（非推测），后续为衍生的测试治理盲区。

### 6.1 `/ai/compose`（及其流式变体）无鉴权、无限流

`backend/app/routes/ai.py:16-18`：

```python
@router.post("/compose", response_model=AIComposeOut)
async def compose(body: AIComposeIn) -> AIComposeOut:
    return await ai_compose.compose(body)
```

与 `POST /ai/compose/stream`（同文件第 38 行起）均**没有 `Depends(get_current_user)` 或 `Depends(get_optional_user)`**，是全站唯一两个允许匿名调用且会触达真实付费 DeepSeek API 的端点（对照：`GET /ai/greetings` 有 `Depends(get_current_user)`，见第 21 行）。全仓搜索确认**不存在任何限流中间件或库**（`grep -rn "slowapi|rate_limit|RateLimit|Limiter"` 仅命中 `openai` SDK 自身的 `RateLimitError` 异常类型，未见业务侧限流实现）。

- **测试盲区**：`backend/app/tests/` 中**没有任何文件测试 `/ai/compose` 或 `/ai/compose/stream`**（`grep -rn "/ai/compose" tests/*.py` 仅命中 `test_smoke.py` 对 `/openapi.json` 路径存在性的断言，不涉及行为）。`services/ai_compose.py` 的 `compose()`/`stream_chunks()` 的真实调用分支（非 dry-run）、上游 `APIConnectionError`/`RateLimitError`/`APIStatusError` 时"仍以 502/504 上抛"的行为（基线 7 节所述"仅 `/ai/compose(/stream)` 处上游失败仍以 502/504 上抛"）**完全未经测试验证**。
- **风险**：任何未登录的第三方均可反复调用 `/ai/compose` 消耗 DeepSeek 配额产生费用，且无测试基线可用于验证"加鉴权/加限流"后不破坏现有契约——这是一处代码事实明确、测试与生产行为双重空白的高优先级缺口，与基线文档第 9 节"仍待解决的开放问题"中"无速率限制：`/ai/compose` 无鉴权却调付费 DeepSeek，是否仍是可接受缺口"完全一致。

### 6.2 多 worker 部署下进程内缓存与单写者机制的竞争风险

`services/greeting.py` 的 `_GREETINGS_CACHE`（`test_greeting.py` 中以 `greeting_svc._GREETINGS_CACHE.clear()` 直接操作的模块级字典）是**进程内**、按 `sid` 缓存 3 小时的问候语；`main.py` 的 `lifespan` 拉起的 `_conf_crawl_loop` 后台任务与 `services/conferences_crawler.py::crawl_sync()` 依赖"拷临时文件→`sqlite3` 直写→`os.replace()` 原子换→重写 manifest"的单写者假设。

- **测试盲区**：仓库内**未发现** systemd 单元文件或任何 `uvicorn --workers N` 配置能确认生产部署是单进程还是多进程（`deploy.sh`/`Makefile` 仅见 `systemctl restart feiyue-backend.service`，未见 worker 数配置）；`backend/app/tests/` 全部测试均在单一测试进程内运行 `client` fixture，**没有、也难以用现有 httpx-ASGI 集成测试框架**去模拟"多个独立 uvicorn worker 进程各自持有一份 `_GREETINGS_CACHE`"或"两个 worker 同时触发 `_conf_crawl_loop` 各自 `os.replace()` 互相覆盖"的场景。
- **风险**：若生产环境以 `--workers > 1` 部署，同一用户的三条问候语会因命中不同 worker 而在刷新页面时不一致（缓存未跨进程共享）；`conf_crawl_interval_hours` 循环若在多个 worker 中各自独立触发，会造成 ccfddl/DDG 抓取频率成倍放大且存在"后写覆盖先写"的竞争窗口。此风险在基线文档第 9 节已列为"仍待解决的开放问题"，本文书确认其在测试层面**完全没有对应用例**，属于结构性盲区而非细节疏漏。

### 6.3 只读参考库（schools/conferences）schema 无迁移校验，且测试对其"缺失"静默降级

`backend/alembic/` 的迁移链（0001-0009）仅管辖主库 `labnotes.db`；`schools.sqlite`/`conferences.sqlite` 的表结构**不在 `Base.metadata`/Alembic 管辖范围内**，唯一的契约来源是外部 supervisor-claw 仓库的 `EXPORT_SCHEMA_v1.md`（不在本仓库）与 `services/schools_query.py`/`conferences_crawler.py` 中手写的 raw SQL 字符串。

- **测试盲区（复合两层）**：
  1. **无 schema 校验测试**——不存在任何测试断言"当前 `schools_query.py` 里的列名/表名与 `schools.sqlite` 实际结构相符"；一旦外部 claw 导出格式变更（增删列、改列名），唯一的暴露方式是运行时 `sqlite3.OperationalError`，而非测试期的显式失败。
  2. **数据缺失时测试整体跳过而非失败**——`test_schools.py`/`test_conferences.py` 用 `pytest.mark.skipif(not _has_data(), ...)` 把 23 条用例（18+5）在文件不存在时标记为 `skipped`。对不熟悉本仓库 `data/` 目录约定（需 `scripts/sync/` 拉取 HF Dataset）的贡献者或 CI 环境，`pytest` 汇总的 `skipped` 计数不会让流水线变红，容易被误读为"该模块测试通过"而非"该模块测试根本没跑"。
- **建议**：在 CI/发布检查清单中显式区分"schools/conferences 测试被 skip"与"schools/conferences 测试通过"两种状态（如统计 skip 数并设阈值告警），并为 raw SQL 契约补充一份基于当前 `schools.sqlite`/`conferences.sqlite` 实际列的"schema 快照"测试，在外部导出格式漂移时第一时间在测试期报错而非等到路由层抛 500。

### 6.4 其余测试治理观察

- **Draft.summary 非空分支缺失显式测试**：`routes/drafts.py:126` 的 `if not summary:` 分支——用户已手写摘要时应完全跳过 AI 调用、原样落库——目前所有 `test_drafts.py` 发布用例创建草稿时均未显式传入 `summary`，该分支的"不覆盖"语义无测试锁定（见 TC-23）。
- **`diff.py`/`conferences_crawler.py` 纯函数缺少专属单测文件**：`services/diff.py::compute_diff_segments`、`services/conferences_crawler.py::_derive_state/_next_check` 均是无 I/O 依赖的纯函数，理论上是"性价比最高"的单测对象，但仓库中不存在 `test_diff.py`/`test_conferences_crawler.py` 之类文件——现有覆盖全部依赖端到端场景（`/ai/compose` 无测试；`conferences_crawler` 仅经由已落库种子数据的只读断言间接覆盖，从未直接调用爬取/派生逻辑本身）。
- **cursor 分页"游标失配静默重置"行为无测试锁定**：`services/notes.py:162-166` 在 `query.cursor` 未命中任何行时 `start` 保持 `0`，是一处未文档化、未测试化的隐式契约（TC-37）。
- **前后端"故意重复实现"的算法缺少跨语言一致性断言**：基线文档 7 节列举的 `lib/tree.ts`↔`services/materials.reorder_file`、`diffEngine.ts`↔`services/diff.py`、`isValidGreeting`/`familiarName`↔`greeting.py` 三组"双语言镜像"逻辑，各自在两侧都有独立测试（如 `tree.test.ts` 与 `test_materials.py` 均覆盖 reorder 语义），但**没有任何测试以同一份输入夹具跨语言比较两侧输出是否严格一致**——若某一侧后续演进时另一侧遗漏同步，现有测试矩阵无法探测到"双实现分叉"本身，只能分别验证"各自的内部一致性"。
- **E2E 覆盖单薄且当前不可本机执行**：见 2.3/4.3 节，`editor-ime.spec.ts` 是唯一 Playwright 场景，且 dev VPS 缺 OS 依赖无法执行；admin/materials/schools/conferences/credits 五个 feature 纵切均无浏览器级端到端用例，关键跨系统链路（如"浏览器扩展→transcript-stash→credits 客户端解析"完整链路、"materials 拖拽重排的真实 DOM 交互"）目前只能靠单元测试对各段分别验证，无法验证链路整体在真实浏览器中的可用性。
