# 一、引言

## 1.1 开发目的

本可行性研究报告面向**新疆大学飞跃手册（xju-feiyue，内部代号 Aurash，站名"费阅"）**平台的立项与建设，目的在于：在系统正式投入开发之前，从技术、经济、操作与社会因素四个维度论证本项目的可行性，明确所建议系统相对于现有非正式经验共享方式的必要性与改进之处，为后续需求分析、概要设计、详细设计与实现提供决策依据。

- 指导教师买合木提·买买提（高级实验师）及课程评阅教师；
- 项目团队全体成员（5 人小组，`[成员姓名/学号，待填]`），作为后续各阶段开发与文档编制的共同基线；
- 平台后续接手的维护人员与运营人员。


## 1.2 现状及意义

**现状。** 高校学生在升学（保研、考研、出国）、科研入门、课程复习、竞赛参与、教务事务等环节高度依赖学长学姐的一手经验，即所谓"飞跃"经验。目前新疆大学范围内此类经验的沉淀与传递主要依托 QQ / 微信群聊、学长口耳相传、个人公众号与零散文档，普遍存在以下问题：经验随聊天记录沉底而流失、缺乏结构化归档与检索、教务学分自查需人工比对培养方案、导师与会议信息不透明且分散。国内已有语雀、Notion、各类"考研/保研经验贴"公众号等通用工具，但它们并非面向单一高校的结构化知识库，既无法与本校教务系统（jwxt）打通做学分自查，也不承载本校导师/院校与 CCF 会议截稿等垂直数据。

**意义。** 飞跃手册以**Markdown 长文笔记**为核心，辅以共享课程资料库、CCF 会议截稿追踪、导师/院校信息库、教务成绩单一键导入的学分自查、以及 DeepSeek 驱动的 AI 写作辅助，将散落于社交群组的隐性经验固化为可检索、可延续、社群共建的显性资产。截至 2026 年 7 月初，平台已积累 **108 名注册用户、85 篇笔记、8 类共享资料（55 个文件）、41 次点赞、79 次登录事件（去重活跃用户 25 名）**，验证了该定位的真实需求。其意义在于：降低后来者重复踩坑的时间成本，缩小信息不对称，并为学院沉淀一份可持续维护的社群知识基座。

# 二、系统可行性分析

## 2 可行性研究的前提

对所建议开发的飞跃手册平台进行可行性研究的前提如下：项目须在一台低成本云服务器上，以社群自建、单人/小组可长期维护的形态，向本校学生提供结构化经验沉淀、共享资料、教务学分自查、导师/会议情报与 AI 写作辅助等功能，且在数据隐私（尤其教务成绩单）与内容安全（防存储型 XSS）方面满足基本要求。以下从要求、目标、条件假定与限制、研究方法与评价尺度分述。

### 2.1 要求

对所建议系统的基本要求：

**（1）功能要求。**

- 笔记系统：7 大分类（科研 / 课程 / 推荐 / 竞赛 / Kaggle / 工具 / 生活）的 Markdown 长文，支持草稿自动保存、发布、点赞与选段锚点评论；发布走"草稿 → 发布"唯一路径（无独立创建端点），发布时自动生成摘要。
- AI 辅助：基于 DeepSeek 的选段润色（生成前后差异对照，整块采纳/拒绝）、流式摘要生成、个性化欢迎语；AI 不可用时须自动降级，不阻断用户操作。
- 共享资料库：课程资料文件树，支持 PDF / Word / Excel / 图片在线预览、拖拽重排、带真实进度的上传；读公开、写限本人。
- 教务学分自查：浏览器扩展/用户脚本抓取本人成绩单 PDF，经中转端点回传，前端 `pdf.js` 客户端解析并按通识选修各模块规则校验达标情况；成绩单 PDF 全程不落服务器。
- 导师/院校信息库与 CCF 会议截稿追踪：导师检索（拼音模糊匹配）、C9 分组浏览；会议截稿约 230 场，约 72 小时自动爬取更新。
- 管理后台：三级角色（普通用户 / 管理员 / 超级管理员）的用户管理、登录审计与站务统计。

**（2）性能要求。** 面向校级、数千量级用户与笔记规模；笔记列表全量载入后在应用层排序、按标签过滤与游标分页（`services/notes.py::list_notes`，明确适用于数千量级）；导师库 4000+ 行须下推 SQLite 查询（FTS5 + LIKE + 拼音索引）；AI 摘要以 SSE 流式返回，首字节须尽快到达（先发注释行破 nginx 缓冲）。

**（3）输出。** 主要输出为：结构化笔记页与列表（JSON，camelCase 线格式，游标分页）、AI 生成的润色差异与摘要（一次性 JSON 或 SSE 流）、学分自查报告（前端本地生成，不外发）、会议截稿列表、导师检索结果、管理后台统计与登录审计。各类输出的接口契约见配套《软件需求规格说明书》与《概要设计说明书》。

**（4）输入。** 主要输入包括：用户注册资料与登录凭据、笔记草稿正文、资料文件（受扩展白名单与大小限制约束）、教务成绩单 PDF（经中转端点、按学号、5 分钟 TTL 内存暂存）、以及外部数据源（ccfddl / DeepSeek / open-meteo / DuckDuckGo）拉取的数据。数据来源、类型与频度详见需求说明书。

**（5）处理流程与数据流程。** 系统整体数据流："浏览器（React SPA）→ nginx 反代 → FastAPI 单体（routes → services → db）→ 主库 + 两只读引擎"；旁路有"浏览器扩展 → transcript-stash → /credits 自查"与"离线内容管线 → 主库/前端 mock"。基本数据流程图示与叙述见后续《系统需求分析》相关章节。

**（6）安全与保密要求。**

- 鉴权：`Authorization: Bearer <JWT>`（HS256，7 天有效），口令 bcrypt 存储；三级角色，`require_admin` 对非管理员返回 404 使后台不可发现。
- 上传内容防存储型 XSS：应用层拒绝 `.svg/.html/.htm/.xml` 等（`DENY_EXTS`）+ 魔术字节交叉校验 + 静态挂载对每个 `/uploads` 响应强制 `X-Content-Type-Options: nosniff`。
- 教务数据隐私：成绩单 PDF 不在服务器持久化，仅按学号在内存暂存 5 分钟、取即删；扩展从不读取口令与 HttpOnly Cookie。

**（7）同本系统相连接的其他系统。** 教务系统 jwxt、DeepSeek、ccfddl、open-meteo、DuckDuckGo、Notion / xju-course-wiki、supervisor-claw、Hugging Face Dataset、GitHub Pages（见 1.3）。

**（8）完成期限。** 本项目于 2026 年 7 月 4 日启动、7 月 9 日完成（见封面），在小学期规定周期内交付。

### 2.2 目标

所建议系统的主要开发目标：

- **管理信息服务的改进：** 将散落于社交群组的隐性经验固化为可检索、可延续的结构化知识库，显著提升经验获取效率与信息覆盖面。
- **处理速度的提高：** 教务学分自查由人工逐条比对培养方案，转为成绩单一键导入 + 客户端规则自动校验，秒级出报告。
- **人员利用率的改进：** 平台自助化运行，社群自维护，日常运营几乎无需专职人力；AI 辅助降低写作与摘要的人工投入。
- **控制精度的提高：** 会议截稿以约 72 小时自动爬取替代人工跟踪，降低漏截稿风险；导师检索以拼音模糊匹配提升命中率。
- **人力与设备费用的减少：** 采用单台云服务器 + 轻量数据库 + 免费灾备 + 全开源工具链，近零边际成本承载校级规模。

### 2.3 条件、假定和限制

- **运行寿命最小值：** 定位为学院可持续维护的社群基座，预期运行寿命不少于 `[N 年，待定]`。
- **方案选择比较的时间：** 立项阶段方案比较周期约 `[待定]`。
- **经费、投资来源与限制：** `[经费来源与预算上限，待填]`；就已实现形态而言，软件依赖全部开源、许可费用为零，灾备使用免费的 Hugging Face 私有 Dataset，主要现金支出为云服务器与域名续费及 DeepSeek 按量调用（金额见第 6 章，待填）。
- **法律与政策限制：** 处理学生个人教务数据须遵循数据最小化与隐私保护（成绩单不落盘、按学号短时暂存）；抓取教务系统须以学生本人会话、经本人授权进行；开源须符合 MIT License 义务。
- **硬件、软件、运行与开发环境的条件与限制：**
  - 运行环境：华为云单台 VPS + nginx + systemd（`feiyue-backend.service`，*待核实*）+ SQLite（*生产库类型待核实*）+ cron 定时同步。
  - 软件环境：Python 3.11 + FastAPI + SQLAlchemy 2.x（async）+ Alembic + Pydantic v2；Node + React 18.3 + Vite 5 + TypeScript（strict）；uv / pnpm 构建；age 加密。
  - 约束：单进程内含问候缓存与会议爬虫单写者，横向多 worker 扩展需另行处理缓存一致性与写者竞争；导师/会议两只读库 schema 不在 Alembic 管理内，须与外部导出保持同步。
- **可利用的信息和资源：** Notion 与 xju-course-wiki 既有内容、ccfddl 开放数据、supervisor-claw 导师导出、DeepSeek API、开源生态与免费 Hugging Face Dataset。
- **系统投入使用的最晚时间：** 不晚于 2026 年 7 月 9 日（随小学期课程设计交付）。

### 2.4 进行可行性研究的方法

本可行性研究采用**调查分析 + 技术验证 + 方案对比**相结合的方法：通过调研现有非正式经验共享方式（QQ/微信群、公众号、通用笔记工具）确认真实痛点；以已实现的原型系统与真实运行数据（108 用户 / 85 笔记等）验证技术路线的落地性；对若干可选系统方案（通用 SaaS、纯静态站、当前自建方案）从功能契合度、成本、可控性、隐私合规等尺度加权比较，确定推荐方案；并对关键因素（用户规模、AI 调用量、服务器档位）做敏感性分析。

### 2.5 评价尺度

对系统进行评价时采用的主要尺度（按优先次序）：

1. **功能契合度**：是否满足面向本校的结构化经验沉淀、教务学分自查、导师/会议垂直数据等核心需求；
2. **数据隐私与内容安全**：教务数据是否最小化处理、上传是否防存储型 XSS；
3. **成本**：建设与运行的现金支出与人力投入是否可长期承担（近零边际成本优先）；
4. **可控性与可维护性**：是否自主可控、单人/小组可长期维护；
5. **开发时间与使用难易程度**：能否在课程期限内交付，用户上手是否简单。

## 3 对现有系统的分析

**关于"现有系统"的界定：** 飞跃手册立项之前，学校并无面向"飞跃"经验沉淀的正式信息系统。本章所分析的"现有系统"，指学生实际在用的**非正式经验共享现状**：QQ / 微信群聊、学长学姐口耳相传、个人公众号与零散文档，以及教务系统内的人工学分比对。分析其局限性，用以论证建设新系统的必要性。

### 3.1 处理流程和数据流程

现有非正式方式的处理流程大致为：提问者在群内发问 → 若恰有知情学长在线则零散答复 → 信息随聊天记录下沉；经验载体为群消息、私聊与个人文档，无结构化归档与检索。学分自查流程为：学生登录教务系统查看成绩单 → 人工对照培养方案逐条核算通识选修各模块学分 → 手工判断是否达标。上述流程无统一数据流，信息高度分散、不可沉淀。

### 3.2 工作负荷

现有方式的"工作负荷"主要落在个人身上：知情学长重复回答同类问题、提问者反复检索与打捞历史消息、学分自查逐条人工核算。无集中系统承载，负荷不可度量、不可复用。

### 3.3 费用开支

现有非正式方式无专门系统投入，直接现金开支近似为零；但存在显著的**隐性时间成本**（重复问答、信息打捞、人工核算），难以货币化。`[如需以人民币量化隐性成本，其口径待定/待填。]`

### 3.4 人员

现有方式无专职运行与维护人员，依赖热心学长学姐的自发投入，稳定性与覆盖面无保障。

### 3.5 设备

现有方式不使用专门设备，仅依托个人终端与第三方社交/文档平台（QQ、微信、公众号等）。

### 3.6 局限性

现有方式的主要局限：

- 经验随聊天记录流失，**无结构化沉淀与检索**；
- 信息**高度分散**（群聊、私聊、公众号、个人文档并存），获取成本高、覆盖不全；
- 学分自查**纯人工**，易错且耗时；
- 导师与会议信息**不透明、不集中、更新滞后**；
- 资料共享**无统一场所**，版本与权限混乱；
- **无治理能力**（无账号体系、无审计、无质量管理）。

对上述非正式方式的"改进性维护"（如更勤地在群里置顶、整理文档）**无法从根本上解决结构化、可检索、可延续与自动化的问题**，故有必要建设专门系统。

## 4 所建议的系统

### 4.1 对所建议系统的说明

所建议系统即飞跃手册平台，采用**前后端分离 + 分层单体后端 + 三存储域**的整体架构：

- 前端：React 18 单页应用（feature 纵切；TanStack Query 管理服务端状态、Zustand 管理本地状态、Zod 校验数据边界；唯一 fetch 封装 `api/client.ts`）。
- 后端：FastAPI 单体，严格分层 `routes（薄控制器）→ services（业务逻辑）→ db（ORM + 只读引擎）`，依赖注入贯穿；应用工厂在 lifespan 内启动"作者同步"与"会议爬取"两个后台协程。
- 存储：可写主库 `labnotes.db`（用户 / 笔记 / 草稿 / 点赞 / 评论 / 登录事件 / 资料，由 Alembic 管理）+ 两个只读、可热重载的 SQLite（导师 schools、会议 conferences，由外部导出/爬虫再生）。
- 旁路子系统：浏览器扩展（教务成绩单跨源中转）、离线内容管线（Notion / 课程 wiki → Markdown → 入库）、基于 Hugging Face 私有 Dataset 的状态同步与灾备。

对第 2 章各项要求的满足方式：功能要求由上述各 feature 子系统实现；性能要求由游标分页、SQL 下推与 SSE 流式满足；安全要求由 JWT/bcrypt 鉴权、三级角色、上传两层防 XSS 与成绩单不落盘满足；AI 可靠性由"永不硬失败"的多层降级满足。所依据的基本方法与理论根据为：RESTful 分层架构、面向对象领域建模、单向数据流前端范式与最小信任的安全设计。

### 4.2 处理流程和数据流程

所建议系统的核心端到端数据流如下：

1. **登录鉴权：** 前端登录表单 → `POST /auth/login` → 校验口令（bcrypt）→ 签发 JWT → 记录登录事件 → 前端持久化令牌并预取问候。
2. **笔记发布 + AI 润色：** 编辑器草稿自动保存 → 选段 `POST /ai/compose` 返回差异段 → 前端呈现并整块采纳/拒绝；摘要经 `POST /ai/compose/stream`（SSE）流式生成 → `POST /notes/drafts/{id}/publish` 校验并（必要时）AI 兜底摘要，新建笔记、删草稿、单次提交。
3. **教务成绩单导入：** 扩展在教务页以本人会话抓取 PDF → `no-cors` 回传 `transcript-stash`（内存、按学号、5 分钟 TTL）→ `/credits` 轮询取件即删 → 前端 `pdf.js` 本地解析出学分报告（PDF 不落服务器）。
4. **资料上传与预览：** 建资源 → 上传（白名单 + 魔术字节 + 大小限制，自动改名）→ 文件树重建 → 前端按扩展懒加载 PDF/Word/Excel/图片预览器。
5. **会议自动爬取：** 后台每约 72 小时以 ccfddl 为主源、DDG + DeepSeek 兜底抽取 → 写临时库 → 原子替换 → 热重载只读引擎。
6. **内容离线管线：** Notion / 课程 wiki 拉取 → 转 Markdown → 加 frontmatter → 入库/生成前端 mock。

### 4.3 改进之处

按 2.2 目标逐项对照现有非正式方式的改进：

- **管理信息服务：** 由群聊零散信息 → 结构化、可检索、社群共建的知识库（7 分类、标签、锚点评论）。
- **处理速度：** 学分自查由人工逐条核算 → 一键导入 + 客户端规则自动校验，秒级出报告。
- **控制精度：** 会议截稿由人工跟踪 → 约 72 小时自动爬取；导师检索支持拼音模糊匹配。
- **人员利用率：** 自助化运行、社群自维护，几乎无需专职人力；AI 辅助降低写作与摘要投入。
- **人力与设备费用：** 单台云服务器 + 轻量库 + 免费灾备 + 全开源链路，近零边际成本承载校级规模。

### 4.4 影响

#### 4.4.1 对设备的影响

新增一台华为云 VPS 作为唯一运行节点即可满足当前规模；无特殊硬件要求。现有个人终端（浏览器）无需改造，仅教务导入需安装浏览器扩展或用户脚本。

#### 4.4.2 对软件的影响

采用全开源软件栈（FastAPI / SQLAlchemy / React / Vite / nginx / SQLite 等），无需改造既有商业软件；教务导入依赖用户浏览器支持的扩展/用户脚本机制。DeepSeek 以 OpenAI 兼容协议接入，便于替换。

#### 4.4.3 对用户单位机构的影响

平台由学生社群自建自维护，对学院机构无强制流程变更要求；建议明确 1 名负责人 + 若干内容维护者的轻量运营角色（`[人员安排，待填]`）。

#### 4.4.4 对系统运行过程的影响

- 用户操作规程：注册/登录后即可浏览、发布笔记、上传资料、自查学分；教务导入需按向导安装脚本并在教务页触发。
- 运行中心操作：运维经 `deploy.sh` 手动部署（`git pull --ff-only` → 迁移 → 重启 → 健康检查 → 前端构建）。
- 源数据处理：外部数据由后台协程与离线管线拉取入库；成绩单 PDF 仅内存短时暂存、取即删。
- 数据保存与恢复：主库与上传文件、加密密钥经 `scripts/sync/` 定时（cron 每 30 分钟）快照至 Hugging Face 私有 Dataset，支持异地恢复。
- 系统失效后果与恢复：只读库缺失时对应接口降级返回 503 而不影响全局；AI 上游失败自动降级；数据库可从最近 HF 快照恢复。

#### 4.4.5 对开发的影响

- 用户需进行的工作：教务导入需用户自行安装浏览器扩展/用户脚本并授权本人会话。
- 数据库所需数据资源：初始内容经离线管线从 Notion / 课程 wiki 导入；导师/会议数据由外部导出与爬虫再生。
- 开发与测试所需计算资源：一台开发机 + 一台 VPS；测试以 pytest（内存 SQLite，AI dry-run 零联网）与 Vitest 为主。
- 保密与安全问题：口令 bcrypt、令牌 JWT、上传防 XSS、教务数据不落盘；`.env` 密钥经 age 加密后备份。

#### 4.4.6 对地点和设施的影响

无建筑物改造或专门环境设施要求；仅需一台可访问公网的云服务器与域名。

#### 4.4.7 对经费开支的影响

主要现金开支为云服务器与域名续费、DeepSeek 按量调用；软件许可与灾备存储为零。具体金额见第 6 章（`[待填]`）。

### 4.5 局限性

所建议系统当前尚存的局限：

- 笔记列表与导师检索的部分逻辑为全量载入后应用层处理，适用于数千量级，超大规模需改造为数据库端分页/索引；
- 单进程假设（问候缓存、会议爬虫单写者）下，横向多 worker 扩展需另行解决缓存一致性与写者竞争；
- 导师/会议两只读库 schema 不在 Alembic 管理内，须人工与外部导出保持同步；
- `/ai/compose` 端点当前无鉴权且无应用层限流，存在被滥用调用付费 AI 的敞口；
- 部分生产事实（生产库类型、systemd 单元名、prod nginx 实际路由、`feiyue-importer.user.js` 托管位置）需以生产环境为准核实。

上述问题部分源于课程周期内的取舍（规模够用即可、优先交付核心功能），部分为待后续完善性维护解决的既知项。

### 4.6 技术条件方面的可行性

- **功能目标能否达到：** 能。所建议系统的全部核心功能均已在生产代码中实现并上线运行（108 用户 / 85 笔记等真实数据可证）。
- **现有技术能否实现：** 能。所用技术（FastAPI、SQLAlchemy async、React 18、Vite、pdf.js、DeepSeek 兼容 API、SQLite、nginx 等）均为成熟主流技术，社区与文档完备。
- **对开发人员数量与质量的要求及能否满足：** 需具备 Python 后端、React 前端、基础运维与数据库能力的小组；本项目由 5 人小组于 2026 年 7 月 4 日至 7 月 9 日（小学期）集中完成，要求可满足。
- **能否在规定期限内完成：** 能。本项目已在小学期规定期限内、于 2026 年 7 月 9 日完成。

## 5 可选择的其他系统方案

### 5.1 可选择的系统方案 1：直接采用通用第三方平台（Notion / 语雀 / 微信公众号）

参照第 4 章提纲，该方案以现成 SaaS 承载经验沉淀与资料共享，几乎无需开发。**未被选中的理由：** 无法与本校教务系统打通做学分自查，无法承载导师/院校与 CCF 会议截稿等垂直数据；数据与账号不自主可控，隐私（尤其教务数据）合规不可保证；难以定制 AI 辅助与社群治理；长期受制于第三方策略与收费。

### 5.2 可选择的系统方案 2：纯静态站点（Markdown + 静态生成器，如 VitePress/Hugo）

以静态站承载笔记，托管于 GitHub Pages 等。**未被选中的理由：** 无账号体系、点赞/评论、上传与后台治理，无法支持教务导入、AI 流式辅助与动态数据（会议/导师）；内容更新须走 Git 流程，社群参与门槛高。

（其余方案如"自建但采用重量级微服务/独立数据库集群"因规模不匹配、运维成本过高，亦不适用，故不展开。）

**结论：** 相较上述方案，当前自建的"轻量分层单体 + 三存储域 + 免费灾备"方案在功能契合度、隐私可控、成本与可维护性上综合最优，故予采纳。

## 7 社会因素方面的可行性

### 7.1 法律方面的可行性

- **知识产权与许可：** 项目以 MIT License 开源（© 2026 winbeau），依赖均为许可宽松的开源组件，无侵犯专利/版权之虞；引用第三方数据源（ccfddl 等）应遵循其使用条款。
- **个人信息与隐私：** 处理学生教务成绩单遵循数据最小化——PDF 不在服务器持久化、按学号内存暂存 5 分钟且取即删；扩展以学生本人会话运行、从不读取口令与 HttpOnly Cookie。用户资料（口令 bcrypt、令牌 JWT）按最小信任设计存储。

- **行政管理与工作制度：** 平台为学生社群自助工具，不改变学院既有行政流程，采用即可、不采用亦不影响教务，落地阻力小。
- **用户素质：** 目标用户为在校大学生，具备基本的浏览器与 Markdown 使用能力；教务导入提供安装向导（实时探测脚本管理器、全程动画引导），手动上传 PDF 作为兜底，使用门槛低。真实运行数据（108 用户 / 25 名活跃）表明用户可顺利使用。
