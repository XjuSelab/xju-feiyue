## 1 引言

### 1.1 编写目的

本说明书面向 xju-feiyue 全栈平台的维护者、二次开发者与运维人员，自代码层反推并固化系统的总体体系结构、模块划分、模块间接口与耦合关系、数据结构逻辑设计、出错处理策略与安全保密设计。它介于《需求规格说明书》与《详细设计说明书》之间：向上落实需求项（FR-xx），向下约束各模块的详细实现契约，是理解"系统如何被结构性组织"的权威工程文档。

### 1.2 背景与范围

xju-feiyue 是面向新疆大学学生/校友的社区式知识沉淀平台，已在生产域名 `feiyue.selab.top` 上线（基线记录约 108 用户 / 85 篇笔记）。其功能纵切包括：7 分类长文笔记（草稿→发布、点赞、锚点引用评论）、共享课程资料库、CCF 会议截稿追踪、导师/院校招生情报库、教务成绩单一键导入的学分自查、以及 DeepSeek 驱动的个性化问候与 AI 笔记润色。

本说明书覆盖以下代码资产的概要设计：
- 后端 FastAPI 单体：`backend/app/`（`main.py`、`deps.py`、`settings.py`、`routes/*`、`services/*`、`schemas/*`、`db/*`）；
- 前端 React 18 SPA：`frontend/src/`（`router.tsx`、`api/*`、`stores/*`、`components/*`、`features/*`）；
- 浏览器扩展与用户脚本：`extension/`、`frontend/src/features/credits/*`；
- 离线内容管线与状态同步：`scripts/*`、`scripts/sync/*`。

本说明书**不**覆盖详细的函数级实现（属《详细设计说明书》）、外部 supervisor-claw 导出仓库的内部结构（仅约定其只读契约），以及生产 nginx 逐行配置（仓库内无权威副本，须以 huawei2 主机实配为准）。

| 术语 | 含义 |

## 2 总体设计

### 2.1 体系结构

xju-feiyue 采用"**前端组件化 SPA + 后端分层单体**"的两段式体系结构，前后端以 HTTP/JSON 契约解耦（Bearer JWT 鉴权、camelCase 线格式、游标分页为主）。二者各自有清晰的内部分层。

#### 2.1.1 前端分层（组件化 + 单向数据流，近 Flux/MVVM 特征）

前端在逻辑上分为四层，遵循一条不可绕过的"数据层铁律"：

```
  ┌───────────────────────────────────────────────────────────────┐
  │  ① 展示层  features/*（垂直切片：editor/credits/materials/     │
  │            schools/conferences/comments/home/admin/settings…）  │
  │            components/ + hooks/ + lib/ + types，只消费 hook     │
  └───────────────────────────┬───────────────────────────────────┘
                              │  调用 TanStack hook（视图声明式订阅）
  ┌───────────────────────────▼───────────────────────────────────┐
  │  ③ 状态层  TanStack Query v5（所有服务端状态；写后 invalidate │
  │            + 乐观更新） · Zustand v5+persist（仅 authStore /    │
  │            draftStore 两个纯客户端 store）                      │
  └───────────────────────────┬───────────────────────────────────┘
                              │  hook 内部委派给端点函数
  ┌───────────────────────────▼───────────────────────────────────┐
  │  ② 数据访问层  api/index.ts（barrel + 所有 hook）              │
  │      → api/endpoints/<resource>.ts（每资源一文件，必 Zod       │
  │        .parse()） → api/schemas/<resource>.ts（Zod 边界校验）   │
  │      → api/client.ts request<T>()（唯一 fetch 封装）           │
  └───────────────────────────┬───────────────────────────────────┘
                              │  fetch（Bearer；useMock 判据分支）
                              ▼   真后端 / 内存 mock
```

架构风格判定：**组件化 + 单向数据流**。视图（展示层）只以声明方式订阅 TanStack Query 缓存与 Zustand store，用户交互触发的写操作经 hook → endpoint → `client.ts` 下行，写成功后由 TanStack `invalidate` 反向驱动视图刷新，形成"Action → 数据层变更 → 视图重渲染"的单向环路，具备 Flux/MVVM 的核心特征（视图不直接持有可变服务端状态，ViewModel 角色由 hook + Query 缓存承担）。关键约束由代码固化：`api/client.ts` 是**唯一** `request<T>()` fetcher，且无论真假响应一律 `schema.parse()`；SSE 流、文件预览/下载、GitHubStars 等约 15 处为**登记在案的合法直 fetch 例外**（基线第 6 节）。

#### 2.1.2 后端分层（分层单体 + DI）

后端是单一 FastAPI 进程，内部三层自上而下、单向依赖：

```
  ┌───────────────────────────────────────────────────────────────┐
  │  路由层  backend/app/routes/*（11 router，60 装饰器）          │
  │    auth/notes/uploads/drafts/interactions/ai/admin/            │
  │    schools/conferences/materials/transcript                    │
  │    —— 薄控制器：解析请求、DI 注入依赖、调用服务、组装 schema   │
  └──────────────┬───────────────────────────────┬────────────────┘
      Depends(...)│ get_db / get_current_user /   │
                 │ require_admin / client_ip …   │  DI 容器：deps.py
  ┌──────────────▼───────────────────────────────┴────────────────┐
  │  服务层  backend/app/services/*                                │
  │    ai_compose / greeting / diff / auth / comments /            │
  │    conferences_crawler / conferences_query / materials /       │
  │    notes / schools_query / schools_pinyin / schools_overlay /  │
  │    uploads_common / author_sync                                │
  │    —— 业务缝：编排持久化 + 外部服务（DeepSeek/open-meteo/爬虫）│
  └──────────────┬───────────────────────────────┬────────────────┘
                 │ AsyncSession（主库读写）        │ get_engine()（只读）
  ┌──────────────▼───────────────────────────────▼────────────────┐
  │  数据层  backend/app/db/*                                       │
  │    base.py(Base+PRAGMA foreign_keys=ON) · models.py(ORM) ·     │
  │    session.py(主库 async engine) · schools_engine.py /         │
  │    conferences_engine.py(只读单例 Holder + mtime 热重载)       │
  └───────────────────────────────────────────────────────────────┘
```

架构风格判定：**分层单体 + 依赖注入**。三层为严格单向依赖（路由 → 服务 → 数据），横切关注点（数据库会话、当前用户、客户端 IP、角色门）全部经 `deps.py` 以 `Depends(...)` 注入，路由函数不直接构造会话或读全局单例，符合控制反转（IoC）。`main.py` 充当**应用工厂**：`app = FastAPI(..., lifespan=lifespan)`，在 `lifespan` 内建 uploads 目录、`init_schools_holder` / `init_conferences_holder` 并 `boot()`、按开关拉起 `_author_sync_loop` 与 `_conf_crawl_loop` 两个后台协程、teardown 时 `cancel` 任务并 `dispose` 引擎；随后按固定顺序 `include_router`（auth→notes→uploads→drafts→interactions→ai→admin→schools→conferences→materials→transcript），并以 `HardenedStaticFiles` 挂载 `/uploads`。

### 2.2 运行环境与部署拓扑

| 维度 | 内容 |
|---|---|
| 主机 | 华为云 VPS（基线代号 huawei2） |

| 数据库 | 主库现用 SQLite（aiosqlite；asyncpg/Postgres 为备用未启用，生产口径以 huawei2 `.env.local` 的 `DATABASE_URL` 为准）；两只读参考库为独立 SQLite 文件 |
| 证书 | Let's Encrypt / certbot |
| 备份/同步 | cron `*/30` 触发 `scripts/sync/` 推送到 Hugging Face 私有 Dataset：`state/`（主库 `VACUUM INTO` 快照 + uploads 确定性 tar + `.env` age 加密）、`schools/`、`conferences/` 三命名空间，各以 `delete_patterns` 限定前缀互不清除 |

## 3 模块接口与耦合关系

### 3.1 前后端契约（接口约定）

| 契约项 | 约定 | 例外/说明 |
|---|---|---|
| 线格式 | 全站 camelCase（`CamelModel`，`populate_by_name` 兼收 snake_case，`from_attributes` 直接从 ORM 序列化） | **唯一例外** `/schools/*` 用 snake_case（`SnakeModel`），因外部 claw 导出与前端 `schools/types.ts` 本就 snake_case |
| 时间 | 恒 ISO-8601 且带尾 `Z`（`UtcDateTime`） | 无 |
| 鉴权 | `Authorization: Bearer <JWT>`（HS256，7 天，`{sub:sid,iat,exp}`）；各 `endpoints/*.ts` 各自 `authHeaders()` 读 `localStorage['labnotes.auth.token']`（**重复约 8 次未集中**，属已知技术债） | `/ai/compose(/stream)`、`/ai/greetings`、`/notes/transcript-stash`(POST) 为未鉴权端点 |
| 分页 | 主流游标分页（`nextCursor`；`/notes` 线性 id 扫描、评论真 keyset `(created_at,id)` 元组比较 over-fetch+1） | **第二套约定**：`/schools/list` 用 offset 分页 `page/page_size`，返回 `PaginatedAdvisors{items,total,page,page_size}` |
| baseURL | 默认 `''`（同源直穿，prod nginx 按顶层前缀反代） | `VITE_API_BASE` 仅跨源 dev/staging 需要；**非** 旧文档所称 `/api` |
| 204 | 前端用 `NoContentSchema = z.null()` | 错误体统一 `ApiError({detail})`（`client.ts`/`upload.ts` 一致，`AbortSignal` 语义统一） |

**mock↔真后端切换判据**：`useMock = import.meta.env.DEV && !VITE_API_BASE`，`client.ts` 与 `api/index.ts` 以同一判据动态 import mock。切真后端只需在 `.env.local` 写 `VITE_API_BASE=...`，**无需改源或注释导入**（旧文档"注释一行 import"已是 no-op）。mock 覆盖不全：仅 auth、notes(list/hot/latest/liked/get)、ai、notes-images、schools、conferences 有 handler；drafts / interactions / materials / admin / notes-files **无 mock，纯 mock 下抛 501** —— 本地开发大部分功能已需真后端。

### 3.2 后端 routes ↔ services ↔ db 调用关系与耦合度

- **路由层 → 服务层**：单向调用，路由为薄控制器，仅做请求解析、DI 注入、schema 组装，业务逻辑一律下沉服务层。耦合度低（接口式依赖）。典型：`routes/drafts.py` 的 `/publish` 委派 `services/ai_compose.summarize_or_fallback` 与 `services/notes` 计算 `read_minutes`。
- **服务层 → 数据层**：服务经注入的 `AsyncSession`（主库）或 `get_engine()`（只读）读写。耦合度中等：服务层显式感知"`lazy='raise'` 禁隐式懒加载"这一持久化约束——`services/materials.py` 必须以单条 flat SELECT + Python 分组（`build_file_tree`）装配文件树，不能走 ORM `.children/.parent` 遍历（否则触 `MissingGreenlet`）。这是数据层设计策略对服务层实现方式的**有意约束**，而非无意耦合。
- **横切依赖收敛于 `deps.py`**：所有路由对"会话/当前用户/IP/角色门"的依赖统一经 DI，避免路由散落构造。`routes/admin.py` re-export `require_admin`/`require_superadmin` 供 `schools.py`/`conferences.py` 的 `/admin/*` 子路由复用，是受控的模块间共享而非循环依赖。
- **耦合度评述**：整体为"高内聚、低耦合"的分层单体——路由内聚于一个资源域，服务内聚于一段业务，数据层内聚于一种存储。跨层依赖方向严格单向；唯一需注意的是"前端刻意重复后端业务逻辑"造成的**跨栈逻辑耦合**：`frontend/lib/tree.ts` 拖拽投影镜像 Python `reorder_file`、`isValidGreeting`/`familiarName` 镜像后端、diff 算法两端一致——这些是为响应式体验有意付出的双份维护成本，任一侧修改须同步另一侧。

### 3.3 只读引擎热重载接口

`schools_engine.py` 与 `conferences_engine.py` 为孪生单例 Holder，对服务层暴露统一接口：

- `init_holder(data_dir)`：`main.py` lifespan 内构造并 `boot()`；文件缺失不崩启动，仅令后续查询抛 `*DataMissing`。
- `get_engine()`：每次调用轮询数据文件 mtime，若变化则在 `asyncio.Lock` 保护下换新只读 `AsyncEngine`（`mode=ro&immutable=1&uri=true` + `PRAGMA query_only=ON` 纵深防御），并 `dispose` 旧引擎；同时缓存 sibling `manifest.json`。
- `force_reload()`：逃生口，由 `/admin/{schools,conferences}/reload` 与会议爬虫收尾后主动调用，跳过 mtime 轮询立即换新。
- 缺失降级：文件不存在时服务层捕 `*DataMissing`，路由层译成 **HTTP 503**，不影响 app 启动与其他域。

写者契约（爬虫/外部导出）恒为"临时文件 → `os.replace()` 原子换 → 重写 manifest"，保证读者绝不见半成品。schools/conferences 的表结构对 Alembic 与 `Base.metadata` 完全不可见，权威 schema 契约在外部 supervisor-claw 仓库的 `EXPORT_SCHEMA_v1.md`，本仓库只能从 `services/schools_query.py`、`services/conferences_crawler.py` 的 raw SQL 反推。

## 4 数据结构与逻辑设计

### 4.1 主库实体关系（E-R 逻辑）

主库唯一 schema 源为 `backend/app/db/models.py`。核心枢纽是 `users.sid`（`String(11)`，11 位学号，**自然主键**，非自增 id），所有归属型外键均以 `ondelete='CASCADE'` 指向它。

```
                users (sid PK, name, nickname, preferred_name, avatar,
                       avatar_thumb, bio, wechat, phone, email,
                       password_hash, role[user/admin/superadmin], created_at)
   ┌──────────┬─────────────┬──────────────┬──────────────┬────────────┐
   ▼          ▼             ▼              ▼              ▼            ▼
 notes     drafts        likes        comments      login_events  material_
 author_sid owner_sid  user_sid+     author_sid     user_sid       resources
 (id PK)   (id PK)     note_id 复合PK (id PK)        (id 自增PK)     owner_sid(id PK)
   │                       │             │                            │
   │ id                    │ note_id     │ note_id                    │ resource_id
   ▼                       ▼             ▼                            ▼
 notes.id 被 likes / comments 引用（均 CASCADE）              material_files
 删 Note → DB 级联删 Likes/Comments                          (id PK, parent_id 自引用
                                                              CASCADE 树, is_folder, ext,
 material_notice: 单例(id='default', content, visible,        mime, size_bytes, url,
                  updated_by_sid 无FK纯审计, updated_at)       storage_path, sort_order, deleted)
```

实体与关系要点：
- **User (1) — (N) Note**：`notes.author_sid → users.sid CASCADE`；`Note.author` 关系为 `lazy='joined'`（列表可即时带作者），`User.notes` 为 `lazy='raise'`（禁隐式懒加载）。
- **User (1) — (N) Draft**：`drafts.owner_sid`；`drafts.id` 独立，**不 FK 关联最终 Note**——发布时新建 Note 并删 Draft，草稿与笔记逻辑解耦。
- **Note (1) — (N) Like**：`likes` 复合主键 `(note_id, user_sid)`，另有一条功能冗余但无害的 `UniqueConstraint uq_likes_note_user`；`passive_deletes=True` 依赖 DB 级 CASCADE（避免 SQLAlchemy 先 NULL 掉作为复合主键一部分的 `note_id` 而违约）。
- **Note (1) — (N) Comment**：`comments` 含可选锚点三元组 `anchor_text/anchor_offset_start/anchor_offset_end`（选段引用，MVP 仅渲染 `anchor_text`，偏移预留未来高亮恢复）。
- **User (1) — (N) LoginEvent**：`login_events.id` 是全库**唯一的自增整型主键**（其余皆 String uuid/自然键），IP 取自反代 XFF/X-Real-IP。
- **MaterialResource (1) — (N) MaterialFile**：`material_files.parent_id` 自引用（根为 NULL，`remote_side=[id]`），`is_folder` 区分纯 DB 行的文件夹与携 `url/storage_path/ext/mime/size_bytes` 的文件；`(resource_id, parent_id)` 同名唯一性**在服务层强制而非 DB 约束**（SQLite 视 NULL parent_id 两两不等，DB 约束挡不住根级重名）。
- **MaterialNotice**：单例（固定 PK `SINGLETON_ID='default'`），"删除"是软隐藏 `visible=False`；`updated_by_sid` 为纯审计无 FK。
- **删除策略**：全库统一"DB 级 `ondelete='CASCADE'` + ORM `passive_deletes=True` + 关系 `lazy='raise'`"，**从不用** `cascade='all, delete-orphan'`。`base.py` 对每个 SQLite 连接发 `PRAGMA foreign_keys=ON`，否则 CASCADE 在 SQLite 静默失效。
- **类型策略**：7 分类 `CATEGORY_VALUES` 以普通字符串列存储（非原生枚举，保 SQLite 测试廉价）；`StringList` TypeDecorator 在 PG 用 `ARRAY(Text)`、SQLite 用 `JSON`，屏蔽方言差异。

### 4.2 三存储域拆分理由

| 引擎 | 文件 | 读写 | 内容 | 拆分理由 |
|---|---|---|---|---|
| 主库 | `labnotes.db` | 读写（ORM+Alembic） | 用户/笔记/草稿/点赞/评论/登录/资料 | 事务性 UGC，须随 HF `state/` 命名空间镜像备份 |
| schools | `data/schools/schools.sqlite` | 只读（`mode=ro&immutable=1`+`query_only`） | 4000+ 导师/院校行；schema **不在** models.py/Alembic | 外部 supervisor-claw 离线导出，体量大（~20MB，git push 超时故 gitignore），可再生 |
| conferences | `data/conferences/conferences.sqlite` | 只读引擎读 / 爬虫 `sqlite3` 直写 | ~230 CCF 会议 + crawl_state | ccfddl/DeepSeek 爬虫再生，同样 gitignore + HF 同步 |

拆分的本质是**按"信任级 × 生命周期 × 再生性"隔离**：主库是必须强一致备份的权威 UGC；两只读库是可再生的外部/爬取参考数据，物理隔离使它们不进入 Alembic 迁移链、不受主库事务牵连，并可独立热重载与独立同步（`scripts/sync/schools.py`、`conferences.py` 各自镜像各自命名空间）。只读三重防御（`mode=ro` + `immutable=1` + `PRAGMA query_only=ON`）加"原子换文件 + mtime 热重载"，确保数据刷新对在线查询零停机、零脏读。

## 5 出错处理设计

### 5.1 HTTP 状态约定

| 场景 | 状态码 | 设计意图 |
|---|---|---|
| 未携带/失效 Bearer 且端点需鉴权 | 401 | `deps.get_current_user`：解签失败或 `db.get(User,sid)` 空 |
| 非管理员访问 `/admin` 面 | **404** | `require_admin` 有意返 404 而非 403，使整个管理面**不可发现**（见 6.1） |
| 普通 admin 访问超管专属操作 | 403 | `require_superadmin`：他已知面存在故给 403；非 admin 仍 404 |
| 只读库文件缺失/未同步 | **503** | schools/conferences Holder 抛 `*DataMissing`，路由译 503，不崩全局 |
| 上传超限/格式非白名单/魔术字节不符 | 4xx | `uploads_common` 分块流式在超限前中止、扩展白名单+`DENY_EXTS`、`sniff_magic()` 交叉校验 |
| 成绩单非法（非 `%PDF-` 或 >10MB） | 4xx | `transcript.py stash_transcript` 校验后拒收 |
| 成绩单取走但无本人条目 | 204 | GET stash pop 空返 204（前端 `NoContentSchema`） |
| `/ai/compose(/stream)` 上游 DeepSeek 失败 | 502/504 | 仅此处允许硬失败上抛（其余 AI 端点降级不失败） |

前端 `client.ts`/`upload.ts` 将错误体统一为 `ApiError({detail})`，并统一 `AbortSignal` 取消语义；每个响应无论真假一律 Zod `.parse()`，schema 不符即在数据访问层抛出，阻止污染视图层。

### 5.2 AI 降级（永不硬失败用户体验）

- **问候** `services/greeting.py`：DeepSeek 不可用/异常时静默降级 7 时段本地模板，`/ai/greetings` **永不 503**；`is_valid_greeting()` 校验去重，最多 `min(6,2n)` 次调用；天气只传定性描述不传数值。
- **发布/编辑摘要** `summarize_or_fallback()`：DeepSeek 失败时降级首段启发式，永不抛错。
- **流式摘要** `/ai/compose/stream`：先吐 `: ready` 注释行破 nginx 缓冲（`X-Accel-Buffering:no`），上游异常进 `{error}` SSE 事件而**不破坏 HTTP 200 状态**；前端 `composeStream()` 有 3 层兜底（mock 合成 / 非 OK 回退一次性 / 5s 无首字节回退一次性重放）。
- **全局 kill-switch** `deepseek_dry_run`：令测试零联网（echo/假 chunk）。

### 5.3 只读库 503 与前端乐观更新回滚

- **只读库 503**：文件未同步/缺失时降级 503，前端对应查询进入 error 态并提示，不影响其他域正常渲染。
- **乐观更新回滚**：写操作（点赞 `useToggleLike`、资料拖拽 `useReorder` 等）在 TanStack Query `onMutate` 阶段先乐观改缓存（`applyReorder` 镜像后端 `reorder_file` 语义），`onError` 回滚到快照，`onSettled` 统一 `invalidate` 拉取权威态。这保证在网络失败或后端 4xx/5xx 时视图自动纠偏，实现"乐观-回滚-最终一致"的错误闭环。

## 6 安全与保密设计

### 6.1 三级角色与 404-不可发现

- **角色模型**：`user`（默认）/ `admin` / `superadmin`，源在 `users.role` 列，但配置的 bootstrap 超管 `settings.admin_sid` 在 `services/auth.effective_role()` **运行时恒解析为 superadmin，无视 role 列**——迁移出错或列滞后也锁不死运维（0008 的回填仅为仪表盘显示一致）。
- **门控分级**：`require_admin` 对非管理员返回 **404**（非 401/403），使整个 `/admin` 面**不可被探测发现**；`require_superadmin` 对普通 admin 升 403（他已知面存在）、对非 admin 仍 404。前端在 `AdminPage` 内也 mirror "404-for-non-admin"，且 `/admin` 无导航入口仅 URL 可达，形成前后端一致的隐蔽面。

### 6.2 上传两层防存储型 XSS

- **防线①（应用层）** `services/uploads_common.py DENY_EXTS`：上传时直接拒收 `.svg/.html/.htm/.xml`；配合扩展白名单、`sniff_magic()` 魔术字节交叉校验（PDF/OOXML-via-zipfile/OLE2/PNG/GIF/JPEG/WEBP）、服务端定 MIME（不信客户端）、`safe_display_name()` 转义 markdown 元字符。
- **防线②（传输层）** `HardenedStaticFiles`：对每个 `/uploads` 响应 `setdefault("X-Content-Type-Options","nosniff")`，令浏览器绝不 MIME-sniff 将上传载荷当可执行页面。二者叠加使改名的可执行文件无法被当作 HTML/script 提供。
- **运维半（第三层）**：更强的 doc 类 `Content-Disposition: attachment`（强制下载而非内联）**刻意推到 prod nginx `location /uploads/`**（免破头像/PDF 内联预览，属"二选一"），在 `main.py` 与 `uploads_common.py` docstring 声明，须由 huawei2 runbook 落实，**仓库内不可验证**。

### 6.3 成绩单中转的隐私模型

`/notes/transcript-stash` 是有意的**未鉴权、内存、短时**中转，隐私设计要点：
- **为何未鉴权**：jwxt 教务页无 feiyue JWT，且 WebVPN/CAS 登录重定向后的 COOP 切断 `window.opener`、postMessage 不可靠，故绕道未鉴权中转；挂在既有 `/notes` nginx location 内免改反代。
- **数据最小化与限时**：内存 `dict[sid]=(bytes, now+300s)`，5 分钟 TTL、单条 ≤10MB、`%PDF-` 魔术字节校验；GET（需 Bearer）**取即删本人 sid 条目**，无本人条目返 204。
- **PDF 全程不落服务器**：成绩单解析（`features/credits/lib/parseTranscript.ts` + `rules.ts`）在客户端由 pdf.js 完成，学分报告纯本地生成，服务器仅做 ≤5 分钟的比特中转。
- **sid 来源不读 cookie**：扩展 `content.js` 从返回文件名 `<sid>_<ts>.pdf` 解析 sid（因 `webvpn_username` 常 HttpOnly），`/guest/i` 视为未登录；扩展从不见用户密码（用学生自身 jwxt session）。

### 6.4 其他安全措施小结

- 密码 bcrypt 哈希；JWT HS256 7 天；`client_ip` 只信自家 nginx 的 XFF 最左/X-Real-IP（防伪造）。
- CORS `allow_credentials=False` + `cors_origin_list` 白名单。
- 资料库为**有意的共享知识库**：读全公开/optional-auth 返所有人非删资源，仅改动需 `owner_sid==user.sid`。
- 无速率限制：`/ai/compose` 未鉴权却调付费 DeepSeek，是基线记录的已知开放安全缺口，建议后续补速率/配额门控。
- 同步/部署防脚：三命名空间共一 HF Dataset 以 `delete_patterns` 互不清除；deploy `--pull-data` 绝不拉 `state/` 以免回滚生产 DB。
