034||一、引言
035||1.1 开发目的
036||新疆大学飞跃手册（xju-feiyue）拟建设为面向新疆大学在校生与毕业校友的知识沉淀平台。项目开发目的在于把学长学姐在升学、科研、课程、竞赛与教务各环节积累的一手“飞跃”经验，从分散、易失的社交群聊固化为结构化、可检索、社群共建的显性资产，并为教务学分自查、导师院校信息与会议截稿追踪等高频需求提供一站式支撑。本可行性研究报告围绕该课题展开，目的是在系统建设前，从技术、经济、操作与社会因素四个维度论证项目是否具备实施条件，明确所建议系统相对现有非正式经验共享方式的必要性与改进之处，为后续需求分析、概要设计、详细设计、实现与测试各阶段提供立项依据。
037||本报告的预期读者包括：
038||①指导教师买合木提·买买提（高级实验师）及课程评阅教师；
039||②项目小组全体成员：组长赵文彪；组员丁水悦、王艺婷、石建华、陶语涵（组员按姓氏笔画排序），作为后续各阶段开发与文档编制的共同基线；
040||③平台后续的维护与运营人员。
041||在研究思路上，本项目拟采用“社群共建内容＋前后端分离的 Web 平台＋客户端隐私优先的教务成绩单解析＋离线内容与情报采集管线”的技术路线，先由社群贡献与离线管线整理基础内容，再以轻量分层的服务端支撑检索、鉴权与治理。研究范围涵盖笔记发布、资料共享、CCF（China Computer Federation，中国计算机学会）会议截稿追踪、高校与导师信息、教务学分自查以及 AI（Artificial Intelligence，人工智能）写作辅助六大功能域；技术要求的底线是系统能够在单台低成本云主机上长期稳定运行，教务隐私数据不落盘，上传内容能够防范存储型 XSS（Cross-Site Scripting，跨站脚本攻击）。本报告写作时点为项目立项与总体方案论证阶段，文中对系统结构、接口、数据模型与部署方式的描述均为拟采用方案，后续将在需求分析、设计、实现与测试阶段逐步细化和验证。
042||1.2 现状及意义
043||高校学生在保研、考研、出国、科研入门、课程复习、竞赛参与与教务事务等环节，高度依赖学长学姐的一手经验，即所谓“飞跃”经验。就新疆大学而言，此类经验的沉淀与传递目前主要依托 QQ 与微信群聊、学长口耳相传、个人公众号及零散文档，普遍存在结构化程度低、检索困难、易随聊天记录沉底而流失等问题。
044||从更大范围看，国内已有语雀[1]、Notion[2] 等通用知识库工具，以及各类“考研、保研经验贴”公众号；国外则有以 GitHub 为载体的课程笔记与 awesome 清单[3]、高校自建 wiki 及 Reddit 等社区讨论。两者的共同局限在于：均非面向单一高校的结构化知识库，既不与本校教务系统（jwxt）打通以支持学分自查，也不承载本校导师、院校与 CCF 会议截稿等垂直数据。
045||综合上述现状，本项目应解决的主要问题可归纳为：
046||（1）经验载体分散、易失：一手经验散落于群聊与个人文档，缺乏结构化归档与全文检索，随聊天记录沉底而流失；
047||（2）教务学分自查依赖人工：学生需登录教务系统逐条比对培养方案、核算通识选修各模块学分，易错且耗时；
048||（3）升学与科研信息不对称：导师与院校信息不透明、不集中，会议截稿信息更新滞后；
049||（4）资料共享缺乏统一场所：课程资料版本与权限混乱，难以延续与治理。
050||针对上述问题，将现有几类经验共享方式与飞跃手册在关键能力维度上加以对比，如表1-1所示（○表示满足、△表示部分满足、×表示不满足、—表示不涉及）。可见通用工具虽在结构化归档上有所改进，但均无法与本校教务打通，亦不承载导师、院校与会议等垂直数据，这正是建设本平台的必要性所在。

071||1.3 背景
072||本节从系统标识、线上地址、参与各方、运行环境、外部系统关系与知识产权六个方面交代平台背景。
073||①软件系统名称：新疆大学飞跃手册（xju-feiyue）。
074||②线上地址：主域名 https://feiyue.selab.top，部署于华为云 VPS（Virtual Private Server，虚拟专用服务器）。
075||③任务提出者、开发者与用户：任务由项目小组在《工程项目开发综合实践》课程框架下自选课题提出，指导教师为买合木提·买买提（高级实验师）；开发者为项目小组，成员包括组长赵文彪，组员丁水悦、王艺婷、石建华、陶语涵。其中，组长赵文彪负责总体方案、需求梳理、系统设计、部署运维与文档统稿，其他组员参与资料整理、功能验证、测试记录与文档校对。用户为新疆大学在校学生与毕业校友，匿名访客亦可浏览公开内容。
076||④运行环境（计算中心与网络）：平台拟部署于一台华为云 VPS（Virtual Private Server，虚拟专用服务器）的 Linux 云主机，由 nginx（高性能 Web 服务器与反向代理）对外提供前端静态资源并反向代理后端 API（Application Programming Interface，应用程序编程接口）；后端拟基于 FastAPI（Python 异步 Web 框架），以 systemd（Linux 系统与服务管理器）服务单元 feiyue-backend.service 常驻运行。数据持久化拟采用 SQLite（嵌入式关系数据库，保留向 PostgreSQL 迁移的路径），并以私有 Hugging Face Dataset 作为数据库快照、上传文件与加密密钥的异地灾备。服务器的操作系统版本、vCPU、内存与带宽等具体规格详见后续《系统需求分析》运行环境一节。
077||⑤与其他系统或机构的往来关系：本平台计划与若干外部系统发生数据往来，其整体关系如图1-1所示。其中，教务系统 jwxt 经浏览器扩展以学生本人会话只读抓取成绩单 PDF（Portable Document Format，便携式文档格式），并遵循数据最小化原则——成绩单不落盘、按学号在内存暂存 5 分钟 TTL（Time To Live，生存时间）取即删、扩展不读取口令与 HttpOnly Cookie；抓取严格限于用户本人、经本人授权、仅取本人成绩单，不代管教务事务、不绕过教务系统鉴权，符合《中华人民共和国个人信息保护法》关于个人信息处理最小必要的要求[5]，如推广前教务处或网络安全管理部门对第三方自动化访问另有规定，将据其要求进一步确认。DeepSeek 提供 AI 润色、摘要与个性化欢迎语，为可降级的外部服务；ccfddl 为 CCF 会议数据主源；DuckDuckGo（经 ddgs 兜底检索）与 open-meteo（欢迎语天气，仅取定性描述）为可降级的辅助数据源；Notion 与 xju-course-wiki 为离线内容管线的素材来源；supervisor-claw 提供导师与院校数据的外部离线导出；私有 Hugging Face Dataset 承担状态同步与灾备；GitHub Pages 仅用于发布 site/ 更新日志。

081||二、系统可行性分析
082||2 可行性研究的前提
083||本可行性研究报告依据《计算机软件文档编制规范》（GB/T 8567，以下简称 GB8567）[7]中可行性研究报告的结构编制。对所建议的飞跃手册平台开展可行性研究的前提如下：项目须在一台低成本云服务器上，以社群自建、单人或小组即可长期维护的形态，面向本校学生提供结构化经验沉淀、共享资料、教务学分自查、导师与会议情报及 AI 写作辅助等功能，并在数据隐私（尤其教务成绩单）与内容安全（防存储型 XSS）两方面满足底线要求。下文从要求、目标、条件假定与限制、研究方法与评价尺度五方面分述。
084||2.1 要求
085||对所建议系统的基本要求如下。
086||（1）功能要求。系统须提供六大功能域：一是笔记系统，围绕科研、课程、推荐、竞赛、Kaggle、工具、生活七大分类的 Markdown 长文，支持草稿自动保存、发布、点赞与选段锚点评论，发布遵循“草稿到发布”的唯一路径并在发布时自动生成摘要；二是 AI 写作辅助，基于 DeepSeek 提供选段润色（生成前后差异对照、可整块采纳或拒绝）、流式摘要与个性化欢迎语，且在 AI 不可用时自动降级、不阻断用户操作；三是共享资料库，以文件树组织课程资料，支持 PDF、Word、Excel 与图片在线预览、拖拽重排及带真实进度的上传，遵循读公开、写限本人；四是教务学分自查，由浏览器扩展或用户脚本在本校教务系统（jwxt）以本人会话抓取成绩单，经中转端点回传后在前端本地解析并按通识选修各模块规则校验达标情况；五是导师与院校信息库及 CCF 会议截稿追踪，支持导师拼音模糊检索、院校分组浏览与约 230 场会议截稿的定期自动更新；六是管理后台与班级空间，面向三级角色提供用户管理、登录审计与站务统计，并支持班级点名、小组与组内文件、甘特任务等班级协作能力。
087||（2）性能要求。系统面向校级、数千量级的用户与笔记规模。笔记列表在应用层排序、按标签过滤并以游标分页，设计容量为数千量级；导师库经数据库端的全文与模糊索引下推查询；AI 摘要以 SSE（Server-Sent Events，服务器推送事件）流式返回，须尽快送达首字节。
088||（3）输出。主要输出包括：结构化的笔记页与列表（经统一 API 以 JSON（JavaScript Object Notation）线格式、游标分页返回）、AI 生成的润色差异与摘要（一次性返回或经 SSE 流式返回）、学分自查报告（前端本地生成、不外发）、会议截稿列表、导师检索结果，以及管理后台的统计与登录审计。各类输出的接口契约详见配套《软件需求规格说明书》与《概要设计说明书》。
089||（4）输入。主要输入包括：用户注册资料与登录凭据、笔记草稿正文、受扩展名白名单与大小限制约束的资料文件、教务成绩单 PDF（经中转端点、按学号在内存暂存，5 分钟 TTL 内取即删），以及从 ccfddl、DeepSeek、open-meteo、DuckDuckGo 等外部数据源拉取的数据。
090||（5）处理流程与数据流程。系统整体数据流为“浏览器（React SPA（Single Page Application，单页应用））经 nginx 反向代理，进入 FastAPI 单体（routes 到 services 到 db 的单向分层），再访问一个可写主库与两个只读引擎”；旁路另有“浏览器扩展经中转端点供学分自查”与“离线内容管线入库”两条。详细的数据流图与叙述见 4.2 及后续《系统需求分析》。
091||（6）安全与保密要求。①鉴权采用 Authorization 头携带 JWT（JSON Web Token，JSON 网络令牌），签名算法为 HS256（HMAC-SHA256，基于 SHA-256 的哈希消息认证码签名算法），令牌 7 天有效，口令以 bcrypt 存储；三级角色权限中管理接口对非管理员返回 404，使后台不可被探知。②上传内容防存储型 XSS：应用层拒绝 .svg、.html、.htm、.xml 等可承载脚本的扩展名，辅以魔术字节交叉校验，并对每个 /uploads 响应强制置入 nosniff 响应头。③教务数据隐私：成绩单不在服务器持久化，仅按学号在内存暂存 5 分钟、取即删，扩展从不读取口令与 HttpOnly Cookie。
092||（7）同本系统相连接的其他系统。包括教务系统、DeepSeek、ccfddl、open-meteo、DuckDuckGo、Notion 与 xju-course-wiki、supervisor-claw、Hugging Face Dataset 与 GitHub Pages（详见 1.3 背景）。
093||（8）完成期限。课程要求的最迟完成期限为 2026 年 7 月 9 日（见封面）。项目计划在该期限前完成核心功能开发、部署联调、测试验证与课程文档整理，并形成可演示版本。
094||2.2 目标
095||建成后，所建议系统相对现有非正式方式带来的可衡量改进，按管理服务、速度、人员利用、精度与费用五个维度列举如下。
096||（1）管理信息服务的改进：将散落于社交群组的隐性经验固化为可检索、可延续的结构化知识库，显著提升经验获取效率与信息覆盖面。
097||（2）处理速度的提高：教务学分自查由人工逐条比对培养方案，转为成绩单一键导入与客户端规则自动校验，秒级产出报告。
098||（3）人员利用率的改进：平台自助化运行、社群自维护，日常运营几乎无需专职人力，AI 辅助进一步降低写作与摘要的人工投入。
099||（4）控制精度的提高：会议截稿以约 72 小时一轮的自动爬取替代人工跟踪，降低漏截稿风险；导师检索以拼音模糊匹配提升命中率。
100||（5）人力与设备费用的减少：采用单台云服务器、轻量数据库、免费灾备与全开源工具链承载校级规模，边际成本很低（具体金额与量化见第 6 章）。
101||2.3 条件、假定和限制
102||所建议系统的条件、假定与限制如下。
103||（1）运行寿命最小值：本平台定位为学院可持续维护的社群知识基座，预期运行寿命不少于 3 年，以覆盖在校生一个完整培养周期并支撑至少一届学长的经验交接。
104||（2）方案选择与比较的时间：方案比选在开发启动前的立项预研阶段完成，构成本可行性论证的组成部分。
105||（3）经费、投资来源与限制：项目由团队自筹、无专项经费，年度现金预算上限与第 6 章支出估算一致；按拟建形态估算，软件依赖全部开源、许可费用为零，灾备使用免费的 Hugging Face 私有 Dataset，主要现金支出为云服务器 VPS 与 DeepSeek 按量调用（金额见第 6 章）。
106||（4）法律与政策限制：处理学生个人教务数据须遵循《中华人民共和国个人信息保护法》[5]的数据最小化原则（成绩单不落盘、按学号短时暂存），抓取教务系统须以学生本人会话、经本人授权进行，开源须履行 MIT License 义务。
107||（5）运行环境：拟采用华为云单台 VPS，经 nginx 提供前端静态资源与 API 反向代理，后端以 systemd 服务 feiyue-backend.service 常驻，持久化默认使用 SQLite（保留 Postgres 迁移路径），并以 cron 定时同步至异地灾备。
108||（6）软件环境：后端为 Python 3.11、FastAPI、SQLAlchemy 2.x（异步）、Alembic 与 Pydantic v2；前端为 React 18.3、Vite 5、TypeScript（strict 模式），以 uv 与 pnpm 构建、以 age 加密密钥。
109||（7）主要约束：单进程内含问候缓存与会议爬虫的单写者，横向多 worker 扩展需另行处理缓存一致性与写者竞争；导师与会议两个只读库的 schema 不在 Alembic 管理内，须与外部导出保持同步。
110||（8）可利用的信息与资源：Notion 与 xju-course-wiki 既有内容、ccfddl 开放数据、supervisor-claw 导师导出、DeepSeek API、开源生态与免费的 Hugging Face Dataset。
111||（9）系统投入使用的最晚时间：不晚于课程期限 2026 年 7 月 9 日，届时应完成可演示系统、主要测试记录与课程文档。
112||2.4 进行可行性研究的方法
113||本可行性研究采用调查分析、方案比较、原型验证与风险评估相结合的方法。首先，通过调研现有非正式经验共享方式（QQ 与微信群、公众号、通用笔记工具）确认真实痛点；其次，围绕教务成绩单不落盘、文件上传安全、会议数据更新、AI 服务降级等关键问题进行技术方案论证和必要的原型验证；再次，对若干可选系统方案（通用 SaaS、纯静态站、自建轻量平台）按 2.5 的评价尺度加权比较，确定推荐方案；最后，对用户规模、AI 调用量与服务器档位等关键因素做敏感性分析。

124||3 对现有系统的分析
125||关于“现有系统”的界定：飞跃手册立项之前，学校并无面向“飞跃”经验沉淀的正式信息系统。本章所分析的“现有系统”，指学生实际在用的非正式经验共享现状——QQ 与微信群聊、学长学姐口耳相传、个人公众号与零散文档，以及教务系统内的人工学分比对。下文从处理流程与数据流程、工作负荷、费用开支、人员、设备与局限性六方面分析其状况，用以论证建设新系统的必要性。
126||3.1 处理流程和数据流程
127||现有非正式方式的处理流程与数据流程大致分为两条相互割裂的主线，如图2-1所示。其一为经验共享：提问者在群内发问，若恰有知情学长在线则给出零散答复，信息随后随聊天记录下沉，经验载体为群消息、私聊与个人文档，无结构化归档与检索。其二为人工学分自查：学生登录教务系统查看成绩单，人工对照培养方案逐条核算通识选修各模块学分，再手工判断是否达标。两条流程彼此独立、无统一数据流，信息高度分散、难以沉淀与复用。
128||【软工要求待补图】 以两条并列泳道绘制。泳道一（经验共享）：学生提问→在群聊/私聊等待→偶遇在线学长→零散答复→信息随聊天记录下沉，标注“无归档、无检索”。泳道二（人工学分自查）：学生登录教务系统→查看成绩单→逐条对照培养方案→手工核算各模块学分→手工判定是否达标。两条泳道之间无共享数据存储，用虚线框标注“信息分散、不可沉淀”。图中用字为五号宋体。

130||3.2 工作负荷
131||现有方式的工作负荷主要落在个人身上：知情学长重复回答同类问题、提问者反复检索并打捞历史消息、学分自查逐条人工核算，且无集中系统承载，负荷不可度量、不可复用。为便于后续分析，此处给出量级估算（为估算假设，非实测统计）：以一个约 30 人规模的班级、每届数百名新生计，估算每名学生每学期约花费 2 至 4 小时在群聊中检索与打捞历史经验，另花费 1 至 2 小时人工核算学分。
132||3.3 费用开支
133||现有非正式方式无专门系统投入，直接现金开支近似为零，但存在显著的隐性时间成本（重复问答、信息打捞、人工核算）。沿用 3.2 的量级估算并按每小时 20 元的时间价值折算（估算口径、非实际支付），单名学生每学期的隐性时间成本约为 60 至 120 元；此口径可供第 6 章收益折算参考。
134||3.4 人员
135||现有方式无专职运行与维护人员，完全依赖热心学长学姐的自发投入，人员随届更替而流动，稳定性与覆盖面均无保障，也无从建立持续的内容治理责任。
136||3.5 设备
137||现有方式不使用专门设备，仅依托学生个人终端与第三方社交及文档平台（QQ、微信、公众号等），无自主可控的存储与服务节点。
138||3.6 局限性
139||综合上述分析，现有非正式方式存在六方面主要局限。
140||（1）经验随聊天记录流失，无结构化沉淀与检索；
141||（2）信息高度分散（群聊、私聊、公众号、个人文档并存），获取成本高、覆盖不全；
142||（3）学分自查纯人工，既易出错又耗时；
143||（4）导师与会议信息不透明、不集中、更新滞后；
144||（5）资料共享无统一场所，版本与权限混乱；
145||（6）无治理能力，缺账号体系、审计与内容质量管理。
146||对上述非正式方式的改进性维护（如更勤地在群内置顶、集中整理文档）只能缓解表象，无法从根本上解决结构化、可检索、可延续与自动化的问题，故有必要建设专门系统。
147||4 所建议的系统
148||针对第 3 章所列现状局限，本章提出并说明所建议的系统——飞跃手册平台，依次给出系统总体说明、处理与数据流程、相对现状的改进之处、实施影响、实施约束与技术条件方面的可行性。
149||4.1 对所建议系统的说明
150||所建议系统即飞跃手册平台，采用前后端分离、分层单体后端与三存储域的整体架构，其静态结构如图2-2所示。
151||（1）前端：React 18 单页应用，按功能域纵向切分为笔记编辑与浏览、共享资料、导师与院校、CCF 会议、教务学分自查、AI 写作辅助、班级空间（班级点名、小组、组内文件与甘特图任务）、评论、管理后台、个人设置等 feature 模块；以 TanStack Query 管理服务端状态、Zustand 管理本地状态、Zod 校验数据边界，并将网络请求收敛于唯一的 fetch 封装。
152||（2）后端：FastAPI 单体，严格分层为 routes（薄控制器）、services（业务逻辑）、db（ORM（Object-Relational Mapping，对象关系映射）与只读引擎），依赖注入贯穿全程；应用工厂在生命周期钩子内启动“作者同步”与“会议爬取”两个后台协程。
153||（3）存储：一个可写主库（承载用户、笔记、草稿、点赞、评论、登录事件、资料，以及班级、小组、组内文件与甘特任务等，由 Alembic 管理），加两个只读、可热重载的 SQLite 引擎（导师与会议数据，由外部导出与爬虫再生）。
154||（4）旁路子系统：教务成绩单跨源中转的浏览器扩展、由 Notion 与课程 wiki 向 Markdown 转换并入库的离线内容管线，以及基于 Hugging Face 私有 Dataset 的状态同步与灾备。
155||对第 2 章各项要求的满足方式：功能要求拟由上述各 feature 子系统连同班级空间与管理后台共同实现；性能要求由游标分页、数据库端查询下推与 SSE 流式满足；安全要求由 JWT 与 bcrypt 鉴权、三级角色、上传两层防 XSS 与成绩单不落盘满足；AI 可靠性由“永不硬失败”的多层降级满足。整体遵循 RESTful（Representational State Transfer，表述性状态转移）分层架构、面向对象领域建模、单向数据流的前端范式与最小信任的安全设计。
156||【软工要求待补图】 自上而下分层绘制：顶层为前端 React 单页应用，框内列出笔记、资料、导师院校、会议、学分、AI 辅助、班级空间、评论、管理后台、设置等 feature 模块；其下为 nginx 反向代理层；中层为 FastAPI 分层单体，内部以箭头标出 routes→services→db 的单向依赖；底层为三存储域——可写主库与导师、会议两个只读引擎；右侧以旁路框标注三个旁路子系统：浏览器扩展、离线内容管线、Hugging Face 灾备。用箭头表示自上而下的调用方向。图中用字为五号宋体。

158||4.2 处理流程和数据流程
159||所建议系统的核心端到端数据流如图2-3所示，其中三条主线计划如下。
160||（1）登录鉴权：前端登录表单提交至登录端点，服务端以 bcrypt 校验口令、签发 JWT，并记录登录事件；前端持久化令牌并预取个性化问候。
161||（2）笔记发布与 AI 润色：编辑器草稿自动保存，选段经润色端点返回差异段供整块采纳或拒绝，摘要经流式端点以 SSE 生成；发布走草稿到发布的唯一路径，校验并（必要时）由 AI 兜底生成摘要，随后新建笔记、删除草稿、单次提交完成。

163||其余数据流一句带过：资料上传与预览（白名单与魔术字节校验、自动改名、文件树重建、按扩展名懒加载预览器）、会议自动爬取（每约 72 小时以 ccfddl 为主源抽取、写临时库、原子替换、热重载只读引擎）、内容离线管线（Notion 与课程 wiki 拉取转 Markdown 入库），以及班级空间协作（班级点名、小组申请与审批、组内文件与甘特任务），详见后续《系统需求分析》。

177||4.4 影响
178||4.4.1 对设备的影响：新增一台华为云 VPS 作为唯一运行节点即可满足预期初期规模，无特殊硬件要求；用户现有个人终端（浏览器）无需改造，仅教务导入需安装浏览器扩展或用户脚本。
179||4.4.2 对软件的影响：采用全开源软件栈（FastAPI、SQLAlchemy、React、Vite、nginx、SQLite 等），无需改造既有商业软件；DeepSeek 以 OpenAI 兼容协议接入，便于按需替换。
180||4.4.3 对用户单位机构的影响：平台由学生社群自建自维护，对学院机构无强制流程变更要求；建议由项目小组承担轻量运营与内容维护，当前成员为组长赵文彪，组员丁水悦、王艺婷、石建华、陶语涵。
181||4.4.4 对系统运行过程的影响：系统上线后，用户注册或登录即可浏览、发布笔记、上传资料与自查学分，教务导入需按向导安装脚本并在教务页触发；运维经部署脚本手动发布（拉取代码、执行迁移、重启服务、健康检查、构建前端）；外部数据由后台协程与离线管线拉取入库，成绩单仅内存短时暂存、取即删；主库、上传文件与加密密钥经 cron 每 30 分钟快照至 Hugging Face 私有 Dataset，支持异地恢复；只读库缺失时对应接口降级返回 503 而不影响全局，AI 上游失败自动降级。
182||4.4.5 对开发的影响：初始内容经离线管线从 Notion 与课程 wiki 导入，导师与会议数据由外部导出与爬虫再生；开发与测试以一台开发机加一台 VPS 承载，后端以 pytest（内存 SQLite、AI dry-run 零联网）、前端以 Vitest 为主；保密方面口令 bcrypt、令牌 JWT、上传防 XSS、教务数据不落盘，密钥经 age 加密后备份。
183||4.4.6 对地点和设施的影响：无建筑物改造或专门环境设施要求，仅需一台可访问公网的云服务器与域名。
184||4.4.7 对经费开支的影响：主要现金开支为云服务器与 DeepSeek 按量调用，软件许可、域名子域与灾备存储为零，具体金额见第 6 章。
185||4.5 局限性与实施风险
186||所建议系统在实施阶段需要关注以下约束与风险。
187||（1）笔记列表与导师检索的部分逻辑为全量载入后在应用层处理，适用于数千量级，超大规模需改造为数据库端分页与索引；
188||（2）单进程假设（问候缓存、会议爬虫单写者）下，横向多 worker 扩展需另行解决缓存一致性与写者竞争；
189||（3）导师与会议两个只读库的 schema 不在 Alembic 管理内，须人工与外部导出保持同步；
190||（4）AI 写作辅助的两个端点——选段润色（/ai/compose）与流式摘要（/ai/compose/stream）——若未接入登录鉴权与应用层限流，可能被匿名滥用并消耗付费 AI 额度；实施时应统一加入登录鉴权与调用频次限制；
191||（5）前端组件级集成测试与端到端（E2E，End-to-End，端到端）测试尚不完备，部分页面的打包体积仍有优化空间。
192||上述问题主要源于课程项目规模、时间和运行成本的约束。实施阶段应优先保障核心功能、隐私安全与可演示性，并将 AI 端点鉴权限流、测试覆盖和前端体积优化列为后续完善事项。

207||5.1 可选择的系统方案 1
208||方案 1：以通用 SaaS 承载。即直接采用语雀、Notion 等成熟在线文档或知识库产品搭建经验沉淀与资料共享空间，几乎无需自行开发。逐尺度评估如下。
209||（1）功能契合度：不契合。此类产品面向通用团队协作，无法与本校教务系统打通做成绩单一键导入与学分自查，也不承载导师、院校与 CCF 会议截稿等垂直数据；AI 辅助与社群治理只能沿用其内置能力，难以按本校场景定制。
210||（2）成本：中等且持续。团队版按人数订阅计费，年费随用户增长而递增[8][9]，长期现金支出高于本方案的单机部署。
211||（3）可控性：弱。数据与账号托管于第三方，导出格式、接口策略与收费方案均受其单方面调整约束，自主可控性差。
212||（4）隐私合规：不可保证。学生教务成绩单等敏感数据若托管于第三方 SaaS，无法满足数据最小化与不落盘的合规要求。
213||综合上述，方案 1 在功能契合度与隐私合规两项关键尺度上均不达标，故不推荐。
214||5.2 可选择的系统方案 2
215||方案 2：以纯静态站承载。即将笔记以 Markdown 源文件维护、经静态站点生成器构建后托管于 GitHub Pages[10] 等免费平台。逐尺度评估如下。
216||（1）功能契合度：弱。静态站无账号体系、点赞与评论、文件上传与管理后台，无法支持教务成绩单导入、AI 流式辅助与会议、导师等动态数据。
217||（2）成本：极低（近乎为零），是其唯一显著优势。
218||（3）可控性与可维护性：中等。内容更新须走 Git 提交与构建流程，普通学生参与门槛偏高，不利于社群共建。
219||（4）隐私：不涉及后端数据处理，但也正因缺乏后端而无法承担教务数据的受控处理。
220||综合上述，方案 2 虽成本最低，却无法覆盖平台的核心动态能力，社群参与门槛偏高，故不推荐。
221||此外，另有自建但采用重量级微服务、独立数据库集群的方案，其功能可完全契合，但对校级、数千量级的规模而言运维复杂度与成本明显过高，与单人或小组可长期维护的定位不符，同样不予采纳。相较之下，本报告采纳的轻量分层单体、三存储域、免费灾备方案在功能契合度、隐私可控、成本与可维护性上取得最佳平衡，综合最优。

236||6.2 收益
237||本项目收益分为可货币化与不可货币化两类。
238||可货币化收益以自动化替代人工的时间节省为主，计算口径为：年时间节省价值＝各功能年使用工时节省之和×学生时间折算单价。其中，学分自查按每学年约 100 人次使用、人均替代人工核算 0.5 小时计，约 50 小时／年；结构化经验检索按建设初期 25 名活跃用户、人均每学年节省重复提问与信息打捞 3 小时计，约 75 小时／年；会议截稿自动追踪按科研向用户约 10 人、人均节省 2 小时／年计，约 20 小时／年；合计约 145 小时／年。按校内勤工助学时薪量级保守取 25 元／小时折算，年时间节省价值约 3600 元。此外，以本方案替代等量商业 SaaS 订阅，年可规避订阅费约 1000 至 2000 元。两项合计，年可货币化收益基准约 4800 元（随规模与折算单价在 3000 至 8000 元区间浮动）。
239||不可货币化收益包括：（1）经验传承与信息对称度提升，降低后来者重复踩坑成本，潜在提高升学、竞赛与申请的成功率；（2）为学院沉淀一份可持续维护的社群知识资产，改善新生引导体验；（3）团队在真实工程项目中获得从设计、实现到运维的全流程锻炼（教学收益）；（4）数据自主可控与隐私合规带来的风险规避。
240||6.3 收益／投资比
241||收益投资比反映单位投资所产生的收益水平。将其定义为系统生命期内累计可货币化收益与累计投资之比，计算式如（2-1）所示。
242||	R = B总 ÷ C总	（2-1）
243||式中：R 为收益投资比；B总 为运行寿命（取 3 年）内累计可货币化收益；C总 为同期累计投资，等于一次性投入与逐年运行支出之和。代入基准情景数据，B总 ＝ 3×4800 ＝ 14400 元，C总 ＝ 8000＋3×1200 ＝ 11600 元，得 R ≈ 1.24，大于 1，经济上可行；若仅以年度运行现金口径（年可货币化收益÷年运行支出）计，则约为 4.0，且尚未计入大量不可货币化收益。
244||6.4 投资回收周期
245||投资回收周期指累计收益首次超过累计投资所需的时间。以年净收益（年可货币化收益减年度运行支出）摊销一次性投入，计算式如（2-2）所示。
246||	T = C0 ÷ (B年 − C年)	（2-2）
247||式中：T 为投资回收周期；C0 为一次性投入（约 8000 元，以折算开发人力为主，属教学项目未实付现金）；B年 为年可货币化收益（基准约 4800 元）；C年 为年度运行支出（基准约 1200 元）。代入得 T ＝ 8000÷(4800－1200) ≈ 2.2 年，短于 3 年运行寿命，可在寿命内回收投资。
248||6.5 敏感性分析
249||对决策指标（收益投资比与回收周期）影响最灵敏的因素为用户规模、AI 调用量与服务器档位。以运行寿命 3 年、一次性投入 8000 元为基准，考察三因素在保守、基准、乐观区间内的变动对两项指标的影响，如表2-5所示。
250||表2-5　关键变量对收益投资比与回收周期的敏感性
251||情景 | 关键变量 | 年投资（元） | 年收益（元） | 收益投资比 | 回收期（年）
252||基准 | 初期规模，AI 受控 | 1200 | 4800 | 1.24 | 2.2
253||规模扩大 | 活跃翻倍，VPS 升档 | 2000 | 8000 | 1.71 | 1.3
254||AI 失控 | 无限流，费用激增 | 3000 | 4800 | 0.85 | 4.4
255||保守 | 增长停滞，时薪偏低 | 1200 | 3000 | 0.78 | 4.4
256||乐观 | 活跃与内容双增 | 1500 | 8000 | 1.92 | 1.2
257||分析表明：当用户规模扩大、活跃度翻倍时，虽需将 VPS 升一档使年投资升至约 2000 元，但时间节省收益同步升至约 8000 元，收益投资比提高至 1.71、回收周期缩短至 1.3 年，规模效应为正。最敏感的负向因素是 AI 调用量：若 /ai/compose 等端点因缺乏鉴权与限流而被滥用，DeepSeek 按量费显著抬升（年投资升至约 3000 元），收益投资比将跌至 0.85（小于 1）、回收周期延至 4.4 年并超出运行寿命；保守情景（增长停滞、折算时薪偏低）亦使两项指标劣化至同一量级。由此可见，控制 AI 调用成本是保障本项目经济可行性的关键，实施时应通过登录鉴权、调用频次限制、缓存和服务降级控制风险。
258||7 社会因素方面的可行性
259||本章从法律合规与使用条件两方面论证社会因素方面的可行性。
260||7.1 法律方面的可行性
261||（1）知识产权与许可。本项目以 MIT 许可证（MIT License）开源（Copyright © 2026 winbeau）[6]，所依赖组件均为许可宽松的开源软件，无侵犯专利或版权之虞；引用第三方数据源（如 ccfddl）遵循其使用条款，并履行 MIT 许可要求的版权与许可声明保留义务。
262||（2）个人信息与数据合规。平台处理学生教务成绩单，须遵循《中华人民共和国个人信息保护法》[5]、《中华人民共和国数据安全法》[13]与《中华人民共和国网络安全法》[14]关于个人信息处理最小必要、知情同意与数据安全的要求。对应措施为：成绩单 PDF 不在服务器持久化，仅按学号在内存暂存 5 分钟（TTL）且取即删；浏览器扩展以学生本人会话运行，从不读取登录口令与 HttpOnly Cookie；用户口令经 bcrypt 存储，令牌按最小信任签发。
263||（3）抓取合规边界。教务数据抓取严格限定四条边界：仅取用户本人成绩单、经本人授权触发、不绕过教务系统既有鉴权、不代管任何教务事务，不触及他人数据。为审慎起见，平台在面向全校推广放量前，将就浏览器自动化抓取是否符合教务系统与 webvpn 使用条款，向学校教务处与网络信息中心确认后再行推广。
264||7.2 使用方面的可行性
265||（1）行政与制度层面。平台为学生社群自助工具，不改变学院既有行政与教务流程，采用与否均不影响正常教务运转，落地阻力小。
266||（2）用户素质与易用性。目标用户为在校大学生，普遍具备基本的浏览器操作与 Markdown 使用能力；教务导入拟提供安装向导，并以手动上传本地 PDF 作为兜底，使用门槛较低。项目建设完成后，可通过注册数、活跃用户数、笔记数量、共享文件数量和用户反馈来检验实际可用性。
267||8 结论
268||综合技术、经济、操作与社会因素四方面分析，结论如下。
269||（1）技术可行。所建议系统所用技术均为成熟主流栈，各功能模块边界清楚，已有开源生态和学校现有数据源能够支撑实现。
270||（2）经济可行。年度现金支出基准约 1200 元、以折算人力为主的一次性投入约 8000 元，基准情景下收益投资比约 1.24（大于 1）、投资回收周期约 2.2 年（短于 3 年运行寿命），且尚有大量不可货币化收益未计入；敏感性分析表明，只要控制 AI 调用成本，经济性即可稳定成立。
271||（3）操作可行。平台采用自助化运行和社群自维护思路，目标用户具备基本浏览器操作能力，日常运营无需专职人员。
272||（4）社会因素可行。以 MIT 许可开源、依赖合规；教务数据经最小信任与数据最小化设计处理，法律与隐私风险可控。
273||据此判断，本项目在四个维度上均具备可行性，建议立项建设。后续建设与运营中应优先完成的事项包括：实现核心功能并补齐测试材料；为 /ai/compose 与 /ai/compose/stream 两个端点补充鉴权与应用层限流，封堵付费 AI 被滥用的敞口；按预留路径在扩容时评估迁移 Postgres；将生产 nginx 配置纳入版本管理；继续完善图表、测试材料与运维文档。

407||2.1.5 性能需求
408||
409||面向校级、数千量级的用户与笔记规模；信息流每页 6～8 条游标分页，评论按键集（keyset）分页，楼内回复折叠按需加载；
410||AI 摘要以流式返回，首字节应尽快到达（服务端先行输出注释行以规避反向代理缓冲）；
411||润色与摘要等已接入 AI 的链路不得阻断发布主流程；AI 审查为后续规划项，后续实现时应采用异步旁路，不计入发布/评论/举报请求响应时间；
412||评论图片九宫格为前端后续接入项，后续实现时应优先加载缩略图，点击再取原图；
413||AI 润色/摘要请求超时上限 30s，超时按失败降级处理。
414||
415||2.1.6 安全性
416||
417||鉴权与授权：所有写操作要求登录（Bearer JWT）；编辑/删除笔记限作者本人；评论删除采用"评论作者或笔记作者"双向授权；举报裁决为后续规划，后续实现时仅管理员可执行。
418||上传内容安全：评论图片复用平台统一上传管线——扩展名白名单（png/jpg/webp/gif）、文件头魔术字节交叉校验、大小上限 8MB、静态资源响应强制 X-Content-Type-Options: nosniff，防止存储型 XSS。
419||治理防滥用：踩计数不在普通前端公开，避免数字攻击性传播；举报幂等合并、AI 审查辅助判定和管理员最终裁决为后续治理规划，当前仅有 Report/Block 数据模型预留。
420||隐私：拉黑为单向隐藏，被拉黑方无感知；经验明细仅本人可见。
421||AI 降级安全：一切 AI 依赖（润色、摘要、审查）故障时均降级而不阻断用户主流程，与平台"AI 永不硬失败"的整体设计原则一致。

832||4．运行设计
833||
834||4.1 运行模块组合
835||
836||本模块与平台其他模块（共享资料库、导师院校库、CCF 会议追踪、管理后台基础设施）共同运行于同一 FastAPI 进程内。运行时的模块组合为：
837||
838||笔记读写链路：React SPA → nginx → FastAPI routes（notes/drafts/ai）→ services → SQLite 主库
839||互动通信链路：React SPA → nginx → FastAPI routes（interactions）→ services → SQLite 主库
840||合集链路：React SPA → nginx → FastAPI routes（collections）→ services → SQLite 主库
841||治理旁路（规划）：当前仅有 Report/Block 数据模型，尚无 /reports、/blocks 路由和 AI 审查任务
842||成长旁路：React SPA → auth → routes/auth.py → check_ins / xp_events 写入
843||
844||4.2 运行控制
845||
846||系统以 feiyue-backend.service 为 systemd 单元常驻运行
847||写操作全部要求登录态（Bearer JWT），前端通过 localStorage 持久化令牌
848||管理操作经服务端角色校验（非管理员返回 404 隐藏后台入口）
849||AI 润色和摘要链路具备降级兜底；AI 审查任务当前未实现，作为治理能力后续补齐
850||签到、表态、收藏以数据库唯一约束/复合主键保证幂等；举报幂等仍需在接口实现时补齐

1274||3.4.1 每日签到 checkin
1275||
1276||项目 | 内容
1277||函数名 | `daily_checkin`
1278||设计人 | 石建华
1279||参数 | `db`, `user_sid`, `today`（date）
1280||返回值 | 签到结果（含经验值、连续签到天数）
1281||功能描述 | 执行每日签到，幂等保护
1282||算法描述 | 1. 检查 (user_sid, today) 是否已存在 —— 若存在返回"今日已签到"（幂等），gained_exp=0；2. 首次签到 INSERT check_ins；3. 固定追加经验 +5；4. INSERT xp_events(source_type=daily_checkin, delta=5)；5. UPDATE users SET exp += 5，并按阈值更新 level；6. 查询连续签到天数并返回
1283||输入 | user_sid: 当前用户；today: 当前日期
1284||输出 | {checkedInDate, streak, gainedExp, exp, level, alreadyCheckedIn}
1285||测试要点 | 1. 同一天重复签到幂等不重复加经验；2. 首次签到写入 check_ins 与 xp_events；3. /auth/me/xp-events 可查询经验流水
