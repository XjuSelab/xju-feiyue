## 1．引言

本章对"飞跃"平台的**笔记系统与社区互动模块**进行需求分析，明确模块的目标、功能需求、数据流与用例模型，作为后续概要设计、详细设计、编码与测试的依据。预期读者为项目团队成员、指导教师及课程评阅人。

按增量模型，本模块的需求划分为两个增量：

- **增量一（已交付）**：笔记写作与发布的核心链路——Markdown 草稿编辑与自动保存、AI 选段润色、AI 摘要生成、发布、信息流浏览与检索、笔记详情、锚点评论、笔记点赞。
- **增量二（本轮新增）**：社区互动与治理的扩展——笔记/评论点踩、笔记收藏、两层楼中楼回复、评论图片（九宫格）、内容举报与"AI 审查 + 管理员裁决"双通道治理、用户拉黑、每日签到与用户等级。

本章以增量二为重点，同时完整给出增量一的需求条目，保证模块需求的整体性与可追溯性。

## 2．任务概述

### 2.1 目标

#### 2.1.1 开发意图

高校学生的升学、竞赛、课程经验多以群聊消息与口耳相传的形式流失，缺乏结构化的沉淀与检索手段。本模块作为"飞跃"平台的内容核心，意图以 Markdown 长文笔记承载经验沉淀，以 AI 辅助降低写作与提炼门槛，以评论、表态、收藏等互动机制促进内容流通，并以"举报 + AI 审查 + 人工裁决"的治理机制与"签到 + 经验等级"的激励机制，保障社区内容质量与用户活跃度的可持续。

#### 2.1.2 应用目标

1. 让注册用户能够低门槛地撰写、润色并发布结构化经验笔记，草稿自动保存，摘要可由 AI 自动生成；
2. 让所有访客（含未登录）能按分类、关键词、标签检索浏览笔记，并阅读评论；
3. 让注册用户能通过点赞/点踩、收藏、锚点评论、楼中楼回复（含图片）等方式与内容和作者互动；
4. 为社区提供内容治理能力：用户可举报违规内容并选择举报类型，系统先经 AI 审查分类、再由管理员裁决处置；用户可拉黑不愿看到的用户；
5. 以每日签到与互动经验驱动的等级体系，提升用户粘性并正向激励优质内容创作。

#### 2.1.3 产品描述

本模块由四个功能域组成（社区口语中的"帖子"即本文的"笔记"，全文统一称"笔记"）：

| 功能域 | 页面/入口 | 内容 |
|---|---|---|
| 写作域 | `/write` | 草稿编辑与自动保存、AI 选段润色（差异对照）、AI 摘要流式生成、发布 |
| 浏览互动域 | `/browse`、`/note/:id` | 信息流（过滤/排序/分页）、笔记详情、赞/踩、收藏、锚点评论、楼中楼回复（含图片九宫格）、评论赞/踩 |
| 治理域 | 内容卡片/详情页操作菜单、`/admin` 举报队列 | 举报（8 类型）、AI 审查分类、管理员裁决、拉黑 |
| 成长域 | 上线弹窗、个人中心 | 每日签到、经验结算、等级展示与明细 |

模块运行于平台统一的前后端架构之上：React 18 单页应用作为前端，FastAPI 单体作为后端，SQLite 作为主库；AI 能力经 OpenAI 兼容协议接入（详见 4.1、4.3 节）。

|---|---|---|---|

| FR-02 | AI 选段润色 | 划选正文片段（≥4 字）后可选择润色/缩写/扩写/改语气/翻译/自定义指令等模式；系统返回修改前后差异对照（增/删/不变三类分段），用户可整块采纳或拒绝 | 一 |
| FR-03 | AI 摘要流式生成 | 编辑器中可触发 AI 生成一句话简介，结果以流式方式逐字呈现；流式失败时自动回退为一次性生成 | 一 |
| FR-04 | 草稿发布 | 发布为笔记产生的**唯一路径**；发布前校验标题、正文、分类必填（缺一提示并中止）；摘要为空时由 AI 自动生成，AI 不可用时降级为取正文首段，**发布不因 AI 故障而阻塞** | 一 |
| FR-05 | 笔记编辑与删除 | 仅作者本人可编辑、删除自己的笔记；删除时其下点赞、评论等关联数据一并清除 | 一 |
| FR-06 | 信息流浏览与检索 | 按 7 大分类、关键词、标签过滤，按最新/热门/我赞过排序，游标分页加载；对任意访客开放 | 一 |
| FR-07 | 笔记详情阅读 | 展示正文（Markdown 渲染）、作者、阅读时长、互动计数；对任意访客开放 | 一 |
| FR-08 | 笔记表态（赞/踩） | 登录用户可对笔记点赞或点踩；同一用户对同一笔记赞与踩**互斥**（点踩自动取消已有的赞，反之亦然）；重复操作幂等；**踩计数不公开展示**（仅本人可见自己的踩状态，管理员可见计数用作治理信号） | 赞：一；踩：二 |
| FR-09 | 收藏笔记与我的收藏 | 登录用户可收藏/取消收藏笔记；个人中心提供"我的收藏"列表页；收藏行为向作者发放经验（见 FR-20） | 二 |
| FR-10 | 锚点评论 | 在详情页划选正文片段可发起引用评论，评论携带被引用文字；阅读评论时可高亮回跳到原文位置 | 一 |
| FR-11 | 楼中楼回复 | 评论采用**两层结构**：顶层评论 + 楼内回复；楼内回复可 @ 楼内其他参与者；楼内默认折叠展示前 3 条，可展开分页加载 | 二 |
| FR-12 | 评论图片 | 评论与回复可携带图片（含 GIF 动图），每条最多 **9 张**，以**九宫格**形式展示，点击查看大图；单图不超过 8MB，格式限 png/jpg/webp/gif | 二 |
| FR-13 | 评论表态（赞/踩） | 登录用户可对评论/回复点赞或点踩，规则同 FR-08（互斥、幂等、踩数不公开）；评论被赞向评论者发放经验（见 FR-20） | 二 |
| FR-14 | 评论删除 | 评论作者可删除自己的评论；笔记作者可删除自己笔记下的任意评论；删除顶层评论时其楼内回复一并删除 | 一（扩展至楼中楼：二） |
| FR-15 | 内容举报 | 登录用户可举报笔记或评论，须选择举报类型之一：垃圾广告 / 色情低俗 / 辱骂攻击 / 违法违规 / 涉政敏感 / 虚假信息 / 侵权抄袭 / 其他（可附最多 200 字补充说明）；同一用户对同一内容重复举报不重复计数（幂等更新）；举报提交后进入待审队列并触发 AI 审查 | 二 |
| FR-16 | AI 审查分类 | 系统在两个时机对内容做 AI 审查：①笔记发布、评论提交后**异步**自动分类；②内容被举报时触发复审。AI 输出 `{分类标签, 置信度, 理由}`，标签体系与举报类型对齐（另含"正常"）。置信度 ≥ 0.9 的违规判定 → 内容自动转为"处理中"隐藏态待人工复核；< 0.9 → 仅标记优先级入队。**AI 服务不可用时举报直接进入人工队列，举报提交与内容发布均不受阻断** | 二 |
| FR-17 | 管理员举报裁决 | 管理员在举报队列中按优先级/时间查看工单（含目标内容快照与 AI 审查结论），执行裁决：维持违规判定并删除/隐藏内容，或驳回举报并恢复展示；处置记录（处置人、时间、动作）留档可查 | 二 |
| FR-18 | 拉黑用户 | 登录用户可拉黑/取消拉黑其他用户，效果为**单向隐藏**：信息流中被拉黑者的笔记对我过滤不显示；评论区中被拉黑者的评论折叠为"已屏蔽该用户的评论"占位（可手动展开，避免楼中楼断层）；**被拉黑方无任何感知，仍可正常互动**；个人中心提供拉黑名单管理 | 二 |
| FR-19 | 每日签到 | 登录用户每日首次进入站点时弹窗提示签到，一键完成，展示所得经验与连续签到天数；当日已签到则不再弹出；签到以（用户，日期）唯一约束保证每日至多一次（幂等） | 二 |
| FR-20 | 经验与等级 | 经验来源：每日签到 +5、评论被点赞 +2/次、笔记被点赞 +5/次、笔记被收藏 +8/次；取消赞/收藏时**对称扣回**（防刷分）。等级阈值 0/50/150/300/600/1000 对应 Lv0 新生 → Lv1 助跑 → Lv2 起跳 → Lv3 腾空 → Lv4 飞跃 → Lv5 登顶；等级徽章展示于评论区昵称旁与个人主页，个人中心可查看经验明细（最近变动记录） | 二 |

#### 2.1.5 性能需求

1. 面向校级、数千量级的用户与笔记规模；信息流每页 6～8 条游标分页，评论按键集（keyset）分页，楼内回复折叠按需加载；
2. AI 摘要以流式返回，首字节应尽快到达（服务端先行输出注释行以规避反向代理缓冲）；
3. AI 审查为**异步**过程，不计入发布/评论/举报请求的响应时间；发布响应目标 < 2s，AI 分类结论最迟 5 分钟内落库；
4. 九宫格评论图片优先加载缩略，点击再取原图；
5. AI 润色/摘要请求超时上限 30s，超时按失败降级处理。

#### 2.1.6 安全性

1. **鉴权与授权**：所有写操作要求登录（Bearer JWT）；编辑/删除笔记限作者本人；评论删除采用"评论作者或笔记作者"双向授权；举报裁决仅管理员可执行。
2. **上传内容安全**：评论图片复用平台统一上传管线——扩展名白名单（png/jpg/webp/gif）、文件头魔术字节交叉校验、大小上限 8MB、静态资源响应强制 `X-Content-Type-Options: nosniff`，防止存储型 XSS。
3. **治理防滥用**：同一用户对同一内容的重复举报幂等合并；踩计数不公开，避免数字攻击性传播；AI 审查仅作辅助判定，**内容的最终处置权在管理员**，防止模型误判直接伤害用户内容。
4. **隐私**：拉黑为单向隐藏，被拉黑方无感知；经验明细仅本人可见。
5. **AI 降级安全**：一切 AI 依赖（润色、摘要、审查）故障时均降级而不阻断用户主流程，与平台"AI 永不硬失败"的整体设计原则一致。

## 4．支持信息

### 4.1 运行环境

- 服务器：华为云 VPS 单机（Linux），nginx 反向代理 + systemd 托管后端进程；
- 后端：Python 3.11 + FastAPI + SQLAlchemy 2.x（async）+ SQLite 主库（Alembic 管理迁移）；
- 前端：React 18 + TypeScript（strict）+ Vite 构建的单页应用，现代浏览器（Chrome/Edge/Firefox 近两年版本）；
- AI 服务：OpenAI 兼容协议接入；润色/摘要用 DeepSeek（deepseek-v4-flash）；审查分类**接口先行**——先以 DeepSeek 提示词方案占位跑通，微调完成后以同一协议切换至 Gemma4 12B 微调模型（部署于 GPU 主机，另行接入）。

### 4.2 支持软件

uv（Python 依赖管理）、pnpm（前端包管理）、Alembic（数据库迁移）、pytest / Vitest（测试）、TanStack Query（前端服务端状态）、Zod（前端数据边界校验）、diff-match-patch（差异计算，前后端同库）。

### 4.3 接口（外部，内部）

- **外部接口**：AI 服务（OpenAI 兼容 chat.completions，一次性与流式两种；审查分类要求以 JSON 返回 {label, confidence, reason}）；
- **内部接口**：前端一律经统一请求封装访问后端 REST API（camelCase JSON，Bearer JWT）；本模块涉及的资源域为 notes / drafts / interactions（扩展 reactions、favorites、reports、blocks、check-ins）/ ai；写操作成功后由前端缓存失效机制驱动界面刷新。

### 4.4 控制

写操作全部要求登录态；管理员操作经服务端角色校验（非管理员不可见举报队列）；AI 审查异步任务由后端进程内任务驱动，失败自动降级不重试阻塞；签到、表态、举报等均以数据库唯一约束/复合主键保证幂等，允许客户端安全重试。

### 4.5 需求注释

以下为需求分析阶段与团队确认的决策记录，作为本章各条需求的裁量依据：

1. 收藏做完整功能（按钮 + 我的收藏页 + 作者经验），而非仅计数；
2. 楼中楼采用两层形态（不做无限嵌套），楼内可 @；
3. 拉黑语义为"仅单向隐藏"，被拉黑方无感知且可正常互动；
4. AI 审核模型采取"接口先行 + DeepSeek 占位"策略，Gemma4 12B 微调模型就绪后经同一 OpenAI 兼容接口无缝切换；
5. 签到不设独立入口，以每日首次上线弹窗承载；
6. 评论图片九宫格展示、每条最多 9 张；
7. 踩计数不公开；赞踩互斥；点踩不产生经验变动；
8. 经验规则与等级阈值见 FR-20，取消行为对称扣回。

## 5．需求分析总结

本章以增量模型为框架，完成了笔记系统与社区互动模块的需求分析：明确了四类执行者与 16 个用例，给出 20 条功能需求（其中增量二新增 12 条）及性能、安全约束；以 1 张系统流程图、1 张顶层数据流图、2 张功能级数据流图和 1 张用例图刻画了模块的处理与数据流转结构；以 4 份用例叙述表细化了"草稿发布""AI 选段润色""锚点评论""举报—AI 审查—人工裁决"四条核心链路；数据字典覆盖 12 类数据流、6 类数据存储、16 类数据项与 9 个加工。

本阶段确立的两条设计原则将贯穿后续设计与实现：其一，**AI 永不硬失败**——润色、摘要、审查一切 AI 依赖故障时降级而不阻断用户主流程；其二，**治理以人为最终裁决**——AI 审查仅提供辅助分类与优先级，内容处置权保留在管理员。下一阶段（概要设计）将据此给出功能与程序的映射、数据库逻辑/物理结构与出错处理设计。
