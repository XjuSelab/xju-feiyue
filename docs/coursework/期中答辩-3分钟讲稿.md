老师好，我是赵文彪，本次负责"飞跃"平台的**笔记系统与社区互动模块**。

先说它要解决的问题：高校同学的升学、竞赛、课程经验，大多散落在群聊和口耳相传里，既难检索、也难沉淀。我这个模块就是把这些经验做成**结构化的长文笔记**，再用评论、表态、收藏、合集和签到这些社区机制，让内容能被持续地阅读、讨论和维护。

功能上分成五个域：**写作域**支持 Markdown 草稿自动保存、AI 选段润色、AI 一句话摘要和发布；**浏览互动域**是信息流检索、详情阅读、赞踩、收藏和锚点评论、楼中楼回复；**合集域**让作者把自己的笔记组织成专题；**成长域**是每日签到与经验等级；**治理域**是举报、AI 审查和拉黑。

我按**增量模型**推进：增量一先把写作、发布、浏览和 AI 辅助这条核心链路跑通并交付；增量二这一轮扩展了社区互动和笔记合集。这里要如实说明——治理域目前只落了 Report、Block 两张数据模型，举报接口、AI 审查任务和前端入口是**后续规划项**，我没有把它当成已交付功能。

技术上是前后端分离：前端是 React 18 单页应用，后端是 **FastAPI 单体**，严格按 routes → services → db 三层单向依赖，数据用 SQLite 加 SQLAlchemy 管理；AI 走 OpenAI 兼容协议接 DeepSeek。

设计上我最想强调三点：**第一，AI 永不硬失败**——润色和摘要在故障时降级而不阻断，比如摘要生成不了就取正文首段兜底，发布照常完成；**第二，发布是建笔记的唯一路径**，草稿到笔记在一次事务里原子完成，不会出现"草稿删了、笔记没建成"的中间态；**第三，幂等**——点赞、点踩、收藏、签到都靠数据库的复合主键和唯一约束保证可以安全重试，赞和踩还做了互斥。

目前后端功能大部分已实现并用 pytest 做了覆盖，全量测试 189 通过、23 跳过；需求、概要、详细、测试四章文档和 22 张配图也已经整合进 Word。下一步我会优先把"后端已有、前端还没接"的能力补上——收藏、合集侧栏、评论九宫格图片，再推进治理链路。

以上是我负责的模块，请老师批评指正。
