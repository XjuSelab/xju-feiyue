## 一、结论总览

当前代码库已经**完整或较完整实现**了“增量一”的主链路，即：

- 草稿编辑 + 自动保存
- AI 选段润色 + diff 差异展示
- AI 简介流式生成 + 三层降级
- 草稿发布（唯一发布路径）
- 笔记列表浏览 / 分类 / 搜索 / 标签 / 排序 / 游标分页
- 笔记详情阅读
- 点赞（幂等）
- 锚点评论
- 评论删除（评论作者 / 笔记作者双权限）
- 笔记编辑与删除

而“增量二”中大部分新功能**尚未进入实际代码实现**，包括：

- 收藏
- 点踩
- 楼中楼回复
- 评论图片
- 评论点赞 / 点踩
- 举报 + AI 审查 + 管理员裁决
- 拉黑
- 签到
- 经验 / 等级

因此，若文档要从第 4 章重新起笔，并且要求“对未完成功能进行代码补充”，那么后续开发重点应放在**增量二功能的裁剪与落地优先级**上，而不是重写已实现的增量一。

## 2.1 已实现或基本实现

### FR-01 草稿创建与自动保存
**结论：已实现（前端本地草稿 + 后端发布前中转草稿）**

**证据：**
- 前端草稿状态：`frontend/src/stores/draftStore.ts`
  - Zustand `persist` 到 `localStorage`
  - `ensureCurrent / createDraft / updateDraft / saveDraft / deleteDraft`
- 自动保存：`frontend/src/features/editor/hooks/useAutoSave.ts`
  - 800ms debounce
- 写作页接线：`frontend/src/features/editor/WritePage.tsx`
  - `useAutoSave(...)`
  - 新建草稿模式持续更新本地 draft store

**说明：**
- 当前“自动保存”的主要落点是前端本地持久化，不是实时同步后端。
- 真正发布时，前端会先 `createDraft` 到后端，再调用 `/publish`，因此“服务端草稿”更多用于发布链路而非编辑期持续同步。

### FR-04 草稿发布
**结论：已实现，且符合“唯一发布路径 + AI 失败降级”**

**证据：**
- 前端发布入口：`frontend/src/features/editor/WritePage.tsx`
  - 新建模式下先 `draftsApi.createDraft(...)`
  - 再 `draftsApi.publishDraft(server.id)`
- 后端唯一路径：`backend/app/routes/drafts.py`
  - `POST /notes/drafts/{draft_id}/publish`
- 校验逻辑：
  - 标题为空 -> 422
  - 正文为空 -> 422
  - 分类为空 -> 422
- 摘要为空时：
  - `ai_compose.summarize_or_fallback(...)`
  - 回退 `summary_from(content)` 首段截断
- Note 创建后删除 Draft：
  - 同一发布事务中完成（逻辑上是一条链）

**说明：**
- 这是第 7 章测试的核心条目之一：“唯一发布路径”
- 第 6 章顺序图可以直接围绕该链路画。

## 三、实现与第 4 章文档的一致性建议

## 3.1 可以直接写进后续章节的“已实现事实”
以下内容可以直接在第 5～7 章中作为真实实现来描述：

1. **发布为唯一建笔记路径**  
   后端没有 `POST /notes` 创建接口，只有 `POST /notes/drafts/{id}/publish`。

2. **AI 摘要不会阻塞发布**  
   摘要为空时调用 AI；AI 失败则退化为正文首段摘要。

3. **AI 简介存在三层降级**  
   stream 正常 -> one-shot 回放 -> mock 伪流式。

4. **点赞接口幂等**  
   复合主键 / 唯一约束保证重复点赞无副作用。

5. **锚点评论三字段真实存在**  
   `anchor_text / anchor_offset_start / anchor_offset_end`

6. **评论删除是双向授权**  
   评论作者 或 笔记作者可删。

7. **信息流具备真实的分类 / 搜索 / 标签 / 排序 / 游标分页**
