# 期中答辩速记 & 问答手册 —— 笔记系统与社区互动模块

> 上台前扫一遍。左边是"一眼记住"，后面是"被问到怎么答"。
> 三条底线：① 举报/拉黑/AI审查 = 只有数据模型，规划项；② 很多能力"后端已实现、前端待接入"；③ 经验目前只来自签到。

---

## 一、一句话速记

- **模块画像**：把学生的升学/竞赛/课程经验，做成可检索、可互动、可维护的**结构化长文笔记**；配套评论、表态、收藏、合集、签到成长。
- **我负责**：笔记系统（写作 + 展示）+ 社区互动（评论/表态/收藏/合集/签到/治理预留）。
- **一句技术**：React 18 前端 + FastAPI 单体后端（routes→services→db 三层）+ SQLite/SQLAlchemy + AI（OpenAI 兼容，接 DeepSeek）。

## 二、技术栈 & 规模速查

| 项 | 内容 |
|---|---|
| 前端 | React 18 + TypeScript(strict) + Vite；TanStack Query 缓存、Zod 边界校验、统一 `api/client.ts` 封装、Bearer JWT |
| 后端 | Python 3.11 + FastAPI 单体 + SQLAlchemy 2.x(async)；三层单向依赖 routes→services→db |
| 数据库 | SQLite 主库 + Alembic 迁移（`StringList` 已适配 Postgres 的 ARRAY/JSON，可平滑迁移） |
| AI | OpenAI 兼容 `/v1/chat/completions`；一次性(润色) + SSE 流式(摘要)；当前接 DeepSeek，测试 dry-run |
| 部署 | 华为云 VPS，nginx 反代 + systemd（`feiyue-backend.service`）；数据 cron 快照到 HF 私有 Dataset 灾备 |
| 规模 | 21 条功能需求(FR-01~21)、18 个用例、15 个数据实体；后端测试 189 通过 / 23 跳过 |

## 四、数据模型（15 实体）& 核心关系

**核心四表**：User、Note、Draft、Comment
**互动/成长扩展**：Like(赞)、NoteDislike(踩)、Favorite(收藏)、CommentReaction(评论表态)、CheckIn(签到)、XpEvent(经验流水)、Collection(合集)、CollectionEntry(合集条目)
**治理预留**：Report(举报)、Block(拉黑) —— 只有表，没有路由
**辅助**：LoginEvent(登录审计)

核心关系（记住基数就够）：
- User 1—N Note / Draft / Comment / Collection / CheckIn / XpEvent
- User N—M Note（经 Like / NoteDislike / Favorite 三张关联表）
- Note 1—N Comment；Comment 0..1—N Comment（自引用 `parent_id` = 楼中楼）
- Collection 1—N CollectionEntry；CollectionEntry N—1 Note，且 **note_id 唯一约束 = 单篇笔记只属于一个合集**

## 五、六个"为什么这么设计"（最容易被追问）

1. **为什么发布是唯一建笔记路径？** 没有独立 `POST /notes`，笔记只能从草稿 `POST /notes/drafts/{id}/publish` 发布。→ 保证"草稿→笔记"单一入口和**原子性**，摘要兜底、阅读时长计算都统一在这一步，杜绝中间态。
2. **为什么 AI 要降级？** 升学经验是刚需内容，不能因为第三方 AI 抖动就发不出去。→ **AI 永不硬失败**：摘要三层降级 SSE 流式→one-shot→首段截断兜底。
3. **为什么用复合主键/唯一约束做幂等？** 客户端会重试。→ 点赞`(note_id,user_sid)`、签到`(user_sid,date)` 天然幂等；赞踩互斥（点赞自动删点踩）。
4. **为什么合集单归属？** 避免一篇笔记在多个专题里语义混乱。→ `collection_entries.note_id` 唯一约束 + 加入时三重校验（归属/已发布/本人）。
5. **为什么锚点评论存偏移而不是存 DOM？** 原文可能被编辑。→ 存引用文字 + 起止偏移，渲染后的文本流上定位回跳；原文改了就不高亮，但引用文字仍展示。
6. **为什么不画 ER 图、用 UML 类图？** 课程本次要求"不要 ER，全部改类图"；且后端以 SQLAlchemy ORM 模型为**单一结构来源**，类图能直接对应代码里的 User/Note/Comment 等，字段和关联一目了然。

## 六、分层架构与依赖（一句话讲清）

```
浏览器(React SPA) → nginx(静态直出+API反代) → FastAPI[ routes → services → db(SQLAlchemy) ] → SQLite
                                                        └─ ai_compose ─→ 外部AI服务(DeepSeek)
```
- **单向依赖**：routes 调 services 和 db，services 调 db，绝不反向。
- routes 层：notes / drafts / interactions / collections / ai / auth（6 个路由模块）。
- services 层：notes（查询）/ comments（评论 keyset 分页）/ ai_compose（润色+摘要降级）。
- 互动、签到没有单独 service，逻辑直接在 `routes/interactions.py`、`routes/auth.py` 里——**别说成 `services/interactions.py`（不存在）**。

## 七、高频问答（背熟这些就稳）

**Q1 你具体负责什么？** 飞跃平台的笔记系统与社区互动模块，涵盖写作、展示、评论表态收藏、合集、签到成长，以及治理能力的数据模型预留。

**Q2 举报/拉黑/AI 审查做了吗？** 只做了 Report、Block 两张数据模型预留，没有 `/reports`、`/blocks` 路由，也没有前端入口和工单队列。文档里明确写成规划项，不当作已交付。

**Q3 AI 挂了发布会失败吗？** 不会。摘要是可降级步骤：AI 超时/失败就取正文首段截断兜底，发布主流程照常完成。这是"AI 永不硬失败"原则。

**Q4 楼中楼几层？前端做了吗？** 后端支持两层（顶层 + 楼内回复），三层嵌套被 `parent_id` 校验拒绝；前端目前平铺展示，树形 UI 和评论图片是后续接入。

**Q5 合集完成到什么程度？** 后端 CRUD、条目管理、上下文查询都实现了；详情页侧栏和个人中心合集 Tab 前端待接入。所以说法是"后端已实现，前端待接入"。

**Q6 赞踩怎么保证不重复、互斥？** 复合主键 `(note_id,user_sid)` 幂等；点赞时先删点踩记录、点踩时先删点赞，实现互斥；踩的数量不在普通前端公开，防数字攻击。

**Q7 分页怎么做的？** 游标（keyset）分页：按 `created_at`/`id` 倒序，多取一条判断有没有下一页；比 offset 分页在数据变动时更稳定，不会重复或漏项。

**Q8 为什么用 SQLite？扛得住吗？** 面向校级、数千量级用户与笔记，SQLite 足够且轻量易部署；用 SQLAlchemy + Alembic 管理，`StringList` 已按 Postgres/SQLite 双适配，规模上来可平滑迁 Postgres。

**Q9 安全怎么保证？** 写操作全部要 JWT 登录；编辑/删除限作者本人；评论删除是"评论作者或笔记作者"双向授权；图片走扩展名白名单 + 文件头魔术字节校验 + 8MB 上限 + `nosniff`，防存储型 XSS；管理接口服务端角色校验。

**Q10 经验/等级规则？** 目前经验只来自每日签到 +5，按阈值 0/50/150/300/600/1000 算 Lv0~Lv5；被赞、被收藏产生经验是后续扩展。

**Q11 前后端怎么通信？** REST + camelCase JSON + Bearer JWT，前端统一 `client.ts` 封装，TanStack Query 管服务端状态、Zod 做边界校验。

**Q12 测试做了什么？** pytest + httpx 异步客户端 + 内存 SQLite + AI dry-run 零联网；覆盖评论、赞踩互斥、楼中楼、合集三重校验、签到幂等、热门排序；全量 189 通过、23 跳过。

**Q13 你个人的难点/贡献？** 锚点评论的文本流定位与回跳、赞踩互斥的幂等设计、合集单归属的三重校验、AI 摘要的三层降级链路，以及把文档程序结构和真实代码逐项对齐。

**Q14 为什么图里没有 ER 图？** 本次要求不要 ER、全部改 UML 类图；且类图直接对应 SQLAlchemy 模型，比 ER 更贴合实现。（图 5-2 领域模型类图、图 6-1 程序结构类图）

**Q15 下一步计划？** 先补"后端已有、前端没接"的能力（收藏、合集侧栏、评论图片九宫格、评论赞踩），再推治理链路（举报接口、AI 审查异步任务、管理员工单、拉黑过滤）。

## 八、术语速查（防止被问懵）

- **游标/keyset 分页**：用上一页最后一条的 `(created_at,id)` 当锚点取下一页，替代 `LIMIT/OFFSET`，数据变动时不漏不重。
- **幂等**：同一操作重复执行结果一致（重复点赞不报错、当天重复签到不重复加经验）。
- **SSE 流式**：Server-Sent Events，服务端逐字推送，摘要边生成边显示；失败回退一次性生成。
- **锚点评论**：评论绑定正文某段（存引用文字+起止偏移），可从评论跳回原文并高亮。
- **楼中楼**：两层评论——顶层评论 + 楼内回复，靠 `parent_id` 自引用，禁止三层嵌套。
- **JWT**：登录后签发的令牌，写操作放在 `Authorization: Bearer` 头里做鉴权。
- **ORM / Alembic**：对象关系映射（Python 类↔数据库表）/ 数据库迁移工具。
- **ECB（图 6-1 的 stereotype）**：boundary(界面/接口)、control(路由/服务)、entity(数据模型)三类设计元素。
- **DFD 三符号**：矩形=外部实体、圆/圆角=加工、开口矩形=数据存储。
