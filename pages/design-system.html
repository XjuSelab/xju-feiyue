从一次失败的 Kaggle 复盘说起

Zilun · 2026-05-09 · Kaggle

上周 CIBMTR · Equity in Post-HCT Survival 比赛结束。我从 public LB 第 27 名，一路 shake 到 private 第 312 名 —— 这是一个非常昂贵的教训。这篇笔记不是炫耀，是把 我能复用到下一场比赛 的部分写下来。

一、问题出在哪里

事后看，我犯了一个最朴素的错误：把 public LB 当成了 validation set。我每天提交 5 次，盯着 LB 调超参，但 OOF 分数其实从第三天起就没有再涨过。这意味着我后面所有的「优化」都是噪声。

如果你的 OOF 不动而 LB 在涨，几乎可以肯定你在过拟合榜单。把它写在桌子上。

二、新的工作流程

从这场比赛开始，我把工作流改成了下面这样：

  1. 开赛 24h 内确定 CV 切分方式，并在 README 里画出 fold 分布图。
  2. 每个实验跑前先写一行 hypothesis:，跑完写一行 result:。
  3. 每天最多 2 次提交，且只提交 OOF 显著上涨的版本。
  4. 最后三天禁止改 CV 切分，禁止加新特征。

关键代码片段

下面是新的 CV 模板，我把它放在 utils/cv.py：

pythonfrom sklearn.model_selection import StratifiedGroupKFold
import numpy as np

def make_folds(df, n_splits=5, seed=42):
    """Stratify on event, group by patient_id."""
    sgkf = StratifiedGroupKFold(n_splits=n_splits, shuffle=True, random_state=seed)
    df = df.copy()
    df["fold"] = -1
    for i, (_, val_idx) in enumerate(sgkf.split(df, df.event, df.patient_id)):
        df.loc[val_idx, "fold"] = i
    return df

行内的小提示也是常用的：把 random_state 固定成你出生年月的后四位，跑 ablation 时永远别动它。

三、我会保留的东西

决策带来的提升是否保留
StratifiedGroupKFold+0.012 OOF是
Target encoding (smoothing=20)+0.004 OOF是
Pseudo labeling−0.003 OOF否
SWA on last 3 epochs+0.002 OOF试探

一段命令行

bash# 在 tmux 里同时跑 4 个 fold
for f in 0 1 2 3; do
  tmux new-window -n "fold-$f" \
    "uv run python train.py fold=$f seed=42"
done

四、下次开赛前的检查清单

  • CV 切分是否反映 test 分布？
    • 分组列是什么？
    • 分层列是什么？
  • 是否有第二个独立指标和 LB 交叉验证？
  • 提交配额是否被合理分配（前期探索 / 后期收敛）？
